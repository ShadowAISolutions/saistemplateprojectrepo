## Why This Plan Exists

Plans 9 and 9.1 both require GAS server involvement in the polling loop — either via `doGet` (plan 9) or `google.script.run` (plan 9.1). Both consume server execution quota on every poll cycle.

This plan explores a fundamentally different approach: **write-once, read-many via a public Google Drive file**. The GAS server writes a session indicator file to Drive only on sign-in (rare). The client browser polls that file directly via HTTP — no GAS server involvement in the read path. In theory, this achieves **zero server cost for polling**.

### The Core Idea

1. When a user signs in, the GAS server creates (or overwrites) a small public `.txt` file in Google Drive containing the current session's start timestamp
2. The client stores its own session start timestamp locally
3. Every N seconds, the client fetches the public Drive file URL and compares the timestamp
4. If the file's timestamp is newer than the client's → another device signed in → show eviction overlay
5. The Drive file acts as a "last sign-in" beacon — whoever signed in most recently owns the session

### Comparison to Plans 9 and 9.1

| Aspect | Plan 9 (`doGet`) | Plan 9.1 (`google.script.run`) | Plan 9.1.1 (Drive file) |
|--------|-----------------|-------------------------------|------------------------|
| Server cost per poll | 1 `doGet` + 3 CacheService ops | 1 `google.script.run` + 1 CacheService op | **0** (client reads Drive directly) |
| Server cost on sign-in | Existing (unchanged) | Existing (unchanged) | Existing + 1 DriveApp write |
| Daily quota impact (10 users, 10s) | ~89,280 executions (exceeds 20k) | ~29,760 executions | ~0 executions for polling |
| CORS compatibility | N/A (iframe) | N/A (internal RPC) | **Problematic** (see below) |
| CDN cache staleness | N/A (real-time) | N/A (real-time) | **Unknown delay** (see below) |
| Eviction detection latency | 10s (guaranteed) | 10s (guaranteed) | 10s + cache lag (variable) |

## Critical Technical Challenges

### Challenge 1: CORS — Google Drive Does Not Set `Access-Control-Allow-Origin`

**This is the primary obstacle.** Google Drive file download URLs (`drive.google.com/uc?id=...`, `drive.usercontent.google.com/download?id=...`) do **not** include `Access-Control-Allow-Origin` headers in their responses. A direct `fetch()` or `XMLHttpRequest` from the GitHub Pages domain to a Google Drive URL will be blocked by the browser's CORS policy.

**Sources:**
- [Google Issue Tracker #150193301](https://issuetracker.google.com/issues/150193301) — "Accessing files from Google Drive results in CORS error"
- [Chrome Community thread](https://support.google.com/chrome/thread/29767271) — CORS policy blocks Google Drive downloads
- [mnaoumov.NET workaround](https://mnaoumov.wordpress.com/2018/02/05/download-google-drive-file-via-ajax-ignoring-cors/) — documents the CORS limitation and proxy workarounds

**This means `fetch()` cannot be used.** Alternative read mechanisms are documented in the "Implementation Approaches" section below.

### Challenge 2: CDN Caching — Stale Content Window is Unknown

Google Drive serves public files through a CDN. When the file is overwritten on sign-in, the CDN may continue serving the old content for an unknown duration. Google does not document the cache TTL for publicly shared Drive files.

**Related data points from Google Cloud Storage (which powers much of Google's infrastructure):**
- Default public cache TTL: **1 hour** ([Google Cloud Storage caching docs](https://docs.cloud.google.com/storage/docs/caching))
- Google recommends `max-age` of **15-60 seconds** for frequently-updated content, but Drive files do not expose `Cache-Control` header configuration
- Cache invalidation takes ~10 seconds on Cloud CDN ([Google Cloud CDN invalidation docs](https://cloud.google.com/cdn/docs/invalidating-cached-content)), but Drive does not expose invalidation APIs

**Impact:** If Drive serves stale content for even 2-5 minutes, the eviction detection window could be **worse** than the existing heartbeat (5 min production). The system's usefulness depends entirely on how quickly Drive reflects the file update — and this is not guaranteed or configurable.

**Mitigation strategies:**
- **Cache-buster query parameter:** append `?t=Date.now()` to the Drive URL. May or may not bypass CDN caching depending on how Drive's CDN handles query strings
- **Google Sheets "Publish to Web" alternative:** published Google Sheets update in ~5-10 minutes ([Google Docs Community](https://support.google.com/docs/thread/4836432?hl=en)) — too slow
- **Version the filename:** instead of overwriting the same file, create a new file each sign-in and pass the file ID to the client. The CDN has no cached version of a new file. Cleanup old files afterward

### Challenge 3: DriveApp OAuth Scope

Using `DriveApp.createFile()` or `DriveApp.getFileById()` requires the `https://www.googleapis.com/auth/drive` or `https://www.googleapis.com/auth/drive.file` OAuth scope. The current GAS project may not request this scope.

**Impact:** Adding a Drive scope triggers a **re-authorization prompt** for all existing users. This is a one-time disruption but must be communicated.

**Mitigation:** Use the narrower `drive.file` scope (only accesses files the script creates), which is less alarming in the consent screen than full `drive` access.

## Implementation Approaches

Because `fetch()` is blocked by CORS, the client must use an alternative mechanism to read the Drive file. Three approaches are viable:

### Approach A: Hidden `<img>` Tag with `onerror` / Pixel Tracking Pattern

**How it works:**
1. GAS writes a 1x1 transparent PNG to Drive (not a `.txt`) with the session timestamp encoded in the filename or a metadata URL parameter
2. Client creates a hidden `<img>` tag pointing to the public Drive URL
3. The image loads (or fails) — success means the file exists with that timestamp
4. The client checks periodically by changing the `src` attribute

**Problem:** This only gives you a binary signal (image loads or doesn't). You can't read the timestamp content. You'd need a different image file per session, and the old device would need to know the new file's URL — which requires server communication anyway, defeating the purpose.

**Verdict:** Not viable for timestamp comparison.

### Approach B: Hidden `<iframe>` Loading the Drive File

**How it works:**
1. GAS writes a small HTML file to Drive (not a `.txt`), containing an inline `<script>` that posts the timestamp to the parent via `postMessage`
2. Client loads this HTML file in a hidden iframe
3. The iframe's script reads its own content (it's same-origin with itself) and posts the timestamp to the parent page
4. The parent receives the timestamp and compares it

**Drive URL for iframe loading:** `https://drive.google.com/file/d/FILE_ID/preview` or hosting the HTML directly

**Problem:** Google Drive wraps served HTML in a viewer/preview frame. It does not serve raw HTML for execution. The inline `<script>` would not run because:
- Drive's preview mode renders documents in a sandboxed viewer
- Drive's download mode triggers a file download, not page rendering
- Google explicitly prevents JavaScript execution in Drive-hosted files (Google deprecated "Google Drive hosting" for websites in 2016)

**Verdict:** Not viable — Drive does not serve HTML for execution.

### Approach C: Google Sheets "Publish to Web" as a Proxy (CORS-Compatible)

**How it works:**
1. GAS writes the session timestamp to a cell in a Google Sheet (not a Drive file)
2. The Sheet is published to the web as CSV via File → Share → Publish to the web
3. Client fetches the published CSV URL — published Sheets CSV at `https://docs.google.com/spreadsheets/d/SHEET_ID/export?format=csv` has historically supported CORS (via the `spreadsheets.google.com` base URL)
4. Client parses the CSV, extracts the timestamp, and compares

**CORS note:** The `spreadsheets.google.com` base URL supports CORS ([cors-enabled gsheet gist](https://gist.github.com/pbogden/d46d6dbfcd6f35a3ccda)), but the `docs.google.com/spreadsheets` URL does **not**. The CORS-compatible URL format is:
```
https://spreadsheets.google.com/feeds/cells/SHEET_ID/1/public/full?alt=json
```
However, this is the **Legacy Sheets API v3** format, which Google [deprecated in 2020](https://developers.google.com/sheets/api/guides/migration) and may remove at any time.

**Update delay:** Published Google Sheets take **5-10 minutes** to reflect changes ([Google Docs Community](https://support.google.com/docs/thread/4836432?hl=en), [Published Sheets not updating](https://support.google.com/docs/thread/243001304)). This is too slow for session enforcement.

**Verdict:** Not viable for real-time detection due to 5-10 minute update delay and deprecated CORS-friendly API.

### Approach D: GAS Web App as a Transparent Proxy (CORS-Enabled)

**How it works:**
1. GAS writes the session timestamp to a Drive file (or CacheService, or a Sheet cell) on sign-in
2. A GAS web app endpoint (`doGet`) serves the timestamp as a plain text response with proper CORS headers
3. Client fetches this endpoint directly via `fetch()` — GAS web apps deployed as "Anyone" automatically support CORS for GET requests via `doGet`
4. Client parses the response and compares timestamps

**Wait — this is just Plan 9 again.** The `doGet` call is the same overhead we're trying to avoid. The only difference is what the `doGet` reads from (Drive file vs CacheService). This approach provides no benefit over Plan 9 and adds Drive write overhead on sign-in.

**Verdict:** Defeats the purpose — reintroduces `doGet` overhead.

### Approach E: Client-Side `<script>` Tag Injection (JSONP-Style)

**How it works:**
1. GAS writes a small `.js` file to Drive containing: `window.__sessionTimestamp = "2026-03-15T01:00:00Z";`
2. Client injects a `<script>` tag pointing to the Drive URL
3. The browser loads and executes the script (no CORS restriction on `<script>` tags)
4. Client reads `window.__sessionTimestamp` and compares it to the local session start time

**Drive URL for script loading:** `https://drive.google.com/uc?id=FILE_ID&export=download`

**This might actually work.** `<script>` tags bypass CORS entirely — the browser fetches and executes the file regardless of `Access-Control-Allow-Origin` headers. The content must be valid JavaScript.

**Concerns:**
1. **CDN caching** — same unknown staleness window as any Drive file. Cache-buster (`?t=Date.now()`) may help
2. **Security** — loading JavaScript from a Google Drive URL is a potential XSS vector if the file content is compromised. The GAS server controls the file, but if the Drive account is compromised, an attacker could replace the `.js` file with malicious code that executes in the parent page's context. **This is a significant security risk for an authentication system**
3. **Content-Type** — Drive may serve the file with a `Content-Type` that causes the browser to refuse script execution (e.g., `text/plain` with strict MIME checking)
4. **Cleanup** — old `.js` files accumulate in Drive. Need periodic cleanup
5. **Global namespace pollution** — `window.__sessionTimestamp` pollutes the global namespace. Use a unique, non-guessable property name

**Verdict:** Technically viable but carries significant security risk (arbitrary JS execution from external source). See "Recommended Implementation" for how to mitigate.

### Approach F: `<link rel="preload">` / `<object>` / `<embed>` Tag Tricks

Various HTML tags can load cross-origin resources without CORS:
- `<link rel="preload" as="fetch">` — preloads the resource but doesn't expose the content to JavaScript
- `<object>` / `<embed>` — render the content in a sandboxed context, not accessible to the parent page

**Verdict:** None of these expose the file content to JavaScript for timestamp comparison. Not viable.

## Recommended Implementation (If Proceeding)

**Approach E (`<script>` tag injection)** is the only technically viable path for zero-server-cost polling via Drive. The implementation below includes mitigations for the security and caching concerns.

### Risk Assessment

| Risk | Severity | Mitigation | Residual Risk |
|------|----------|------------|---------------|
| CDN serves stale file content | **High** | Cache-buster query param; version-per-sign-in file names | **Medium** — Google's CDN behavior is undocumented; staleness window could be seconds or minutes |
| XSS via compromised Drive file | **High** | Validate file content format before acting on it; use a Content Security Policy nonce; constrain the script to set only one known variable | **Medium** — if the Drive account is compromised, the attacker can execute arbitrary JS in the user's browser |
| DriveApp scope adds OAuth re-auth prompt | **Low** | Use `drive.file` scope (narrower); communicate one-time disruption | **Negligible** |
| Old files accumulate in Drive | **Low** | Cleanup old files on sign-in; periodic background cleanup | **Negligible** |
| Browser strict MIME checking blocks execution | **Medium** | Set file MIME type on creation; test across browsers | **Low** — Drive typically serves downloads as `application/octet-stream` which most browsers will execute as script |
| Background tab throttles `setInterval` | **Low** | Identical to plans 9/9.1 — acceptable degradation to ~60s | **Low** |

**Overall recommendation:** Plan 9.1 (`google.script.run`) is the safer choice for a security-critical feature. This plan (9.1.1) trades server cost savings for caching unpredictability and a non-trivial XSS attack surface. **Use this plan only if GAS execution quota is a hard constraint that plan 9.1 cannot satisfy.**

## Implementation Phases

### Phase 1: GAS — Session Beacon File Management

**Risk:** Low
**Files:** `testauth1.gs`
**What:** Add functions to create/overwrite a public "session beacon" file in Google Drive on sign-in.

**New server-side functions:**

```javascript
/**
 * Creates or overwrites a session beacon file in Google Drive.
 * The file is a tiny .js file that sets a global variable with the session
 * start timestamp. Public read access allows client-side <script> tag
 * loading without CORS restrictions.
 *
 * @param {string} email — user email (used to locate their beacon file)
 * @param {number} sessionStartTimestamp — Date.now() value at session creation
 * @return {string} — the public URL for the beacon file
 */
function writeSessionBeacon(email, sessionStartTimestamp) {
  var cache = CacheService.getScriptCache();
  var beaconFileIdKey = 'session_beacon_' + email;
  var existingFileId = cache.get(beaconFileIdKey);

  // Content: a minimal JS file that sets a single global variable
  // Use a unique, namespaced variable to avoid global pollution
  var content = 'window.__gasSessionBeacon='
    + JSON.stringify({
        ts: sessionStartTimestamp,
        em: email.substring(0, 3) + '***'  // Truncated email for verification (not full)
      })
    + ';';

  var file;

  if (existingFileId) {
    // Try to overwrite existing file
    try {
      file = DriveApp.getFileById(existingFileId);
      // Overwrite content using Blob
      file.setContent(content);
    } catch (e) {
      // File was deleted or inaccessible — create a new one
      existingFileId = null;
    }
  }

  if (!existingFileId) {
    // Create new file
    file = DriveApp.createFile(
      'gas-session-beacon-' + email.replace(/[^a-zA-Z0-9]/g, '_') + '.js',
      content,
      'application/javascript'
    );

    // Make publicly readable
    file.setSharing(DriveApp.Access.ANYONE_WITH_LINK, DriveApp.Permission.VIEW);

    // Cache the file ID for future overwrites (24-hour TTL)
    // Also store in ScriptProperties for persistence beyond cache TTL
    cache.put(beaconFileIdKey, file.getId(), 86400);
    PropertiesService.getScriptProperties().setProperty(beaconFileIdKey, file.getId());
  }

  // Return the direct download URL (used by client <script> tag)
  return 'https://drive.google.com/uc?id=' + file.getId() + '&export=download';
}

/**
 * Retrieves the beacon file ID from persistent storage.
 * Called when CacheService TTL has expired but the file still exists in Drive.
 *
 * @param {string} email — user email
 * @return {string|null} — the file ID, or null if not found
 */
function getBeaconFileId(email) {
  var cache = CacheService.getScriptCache();
  var beaconFileIdKey = 'session_beacon_' + email;

  // Check cache first (fast)
  var fileId = cache.get(beaconFileIdKey);
  if (fileId) return fileId;

  // Fall back to ScriptProperties (persistent)
  fileId = PropertiesService.getScriptProperties().getProperty(beaconFileIdKey);
  if (fileId) {
    // Re-populate cache
    cache.put(beaconFileIdKey, fileId, 86400);
    return fileId;
  }

  return null;
}

/**
 * Cleans up old beacon files for a user.
 * Called during sign-out to remove the public file from Drive.
 *
 * @param {string} email — user email
 */
function cleanupSessionBeacon(email) {
  var beaconFileIdKey = 'session_beacon_' + email;
  var fileId = getBeaconFileId(email);

  if (fileId) {
    try {
      DriveApp.getFileById(fileId).setTrashed(true);
    } catch (e) {
      // File already deleted — ignore
    }
  }

  // Clear references
  CacheService.getScriptCache().remove(beaconFileIdKey);
  PropertiesService.getScriptProperties().deleteProperty(beaconFileIdKey);
}
```

**Design decisions:**

1. **File content is JavaScript, not plain text** — the file must be valid JavaScript for the `<script>` tag approach to work. A `.txt` file would require a CORS-compatible read mechanism (which doesn't exist for Drive).

2. **`window.__gasSessionBeacon` variable** — uses a namespaced, non-guessable global variable. Contains the timestamp and a truncated email (for verification that the beacon belongs to the expected user — not the full email, to limit information exposure).

3. **File ID stored in both CacheService and ScriptProperties** — CacheService has a 6-hour max TTL, so after 6 hours the file ID would be lost. ScriptProperties persist indefinitely. The cache is checked first (fast), then ScriptProperties as fallback (slower but reliable).

4. **Overwrite vs. create-new** — overwriting the same file is simpler than creating a new one each time. However, CDN caching may serve the old version of an overwritten file. If caching proves problematic, the alternative is to create a new file each sign-in (with a new file ID that has no cached version) and pass the new URL to the client. This trades CDN staleness for Drive file accumulation (cleaned up on sign-in or sign-out).

5. **MIME type `application/javascript`** — explicitly sets the content type so browsers don't refuse to execute it due to strict MIME checking.

6. **Cleanup on sign-out** — trashes the beacon file to prevent stale beacons from lingering. Drive trash is automatically emptied after 30 days.

### Phase 3: Client — `<script>` Tag Polling Loop

**Risk:** High (security-sensitive — loads external JavaScript)
**Files:** `testauth1.html`
**What:** Add a polling loop that injects a `<script>` tag to load the session beacon from Drive, then reads the timestamp and compares it.

#### 3a. HTML_CONFIG toggle

```javascript
CROSS_DEVICE_SESSION_CHECK: true
```

#### 3b. State variables

```javascript
var _sessionBeaconUrl = null;        // Set from gas-session-created response
var _sessionStartTimestamp = null;    // Set on sign-in (client's own session start time)
var _sessionCheckInterval = null;    // Timer ID
var _evictedByServer = false;
```

#### 3c. `startSessionBeaconCheck()` function

```javascript
function startSessionBeaconCheck() {
  if (!HTML_CONFIG.CROSS_DEVICE_SESSION_CHECK) return;
  if (!_sessionBeaconUrl) return;
  if (_sessionCheckInterval) return;  // Already running

  var interval = 10000;  // 10 seconds (server-provided value could override)

  _sessionCheckInterval = setInterval(function() {
    if (_tabSurrendered) return;

    // Remove previous beacon script if it exists
    var oldScript = document.getElementById('session-beacon-script');
    if (oldScript) oldScript.remove();

    // Clear previous beacon value
    delete window.__gasSessionBeacon;

    // Inject new <script> tag with cache-buster
    var script = document.createElement('script');
    script.id = 'session-beacon-script';
    script.src = _sessionBeaconUrl + '&t=' + Date.now();

    script.onload = function() {
      // Script loaded — check the beacon value
      var beacon = window.__gasSessionBeacon;
      if (!beacon || typeof beacon !== 'object') return;

      // Validate structure (basic tamper check)
      if (typeof beacon.ts !== 'number') return;

      // Compare timestamps
      if (beacon.ts !== _sessionStartTimestamp) {
        // Another device signed in — session was superseded
        handleCrossDeviceEviction();
      }

      // Clean up
      delete window.__gasSessionBeacon;
      script.remove();
    };

    script.onerror = function() {
      // File unavailable — could be network error or file deleted (sign-out)
      // Don't evict on error — fail open
      script.remove();
    };

    document.head.appendChild(script);
  }, interval);
}
```

**Security mitigations in the polling function:**

1. **`delete window.__gasSessionBeacon`** — clears the global variable before and after each check. Minimizes the window during which attacker-injected scripts could read or overwrite the value.

2. **Structure validation** — checks that the beacon is an object with a numeric `ts` field. This is a minimal tamper check, not a cryptographic guarantee. An attacker who controls the Drive file can craft valid-looking beacons.

3. **Cache-buster `&t=Date.now()`** — appends a unique query parameter to each request, attempting to bypass CDN caching. Effectiveness depends on how Drive's CDN handles query strings — **this is not guaranteed to work**.

4. **`onerror` fails open** — if the script fails to load (network error, file deleted, Drive outage), the user is NOT evicted. This prevents false evictions from infrastructure issues.

#### 3d. `stopSessionBeaconCheck()` function

```javascript
function stopSessionBeaconCheck() {
  if (_sessionCheckInterval) {
    clearInterval(_sessionCheckInterval);
    _sessionCheckInterval = null;
  }
  var oldScript = document.getElementById('session-beacon-script');
  if (oldScript) oldScript.remove();
  delete window.__gasSessionBeacon;
}
```

#### 3e. `handleCrossDeviceEviction()` function

```javascript
function handleCrossDeviceEviction() {
  stopSessionBeaconCheck();
  stopCountdownTimers();
  stopHeartbeat();
  hideAllWarningBanners();

  _tabSurrendered = true;
  _evictedByServer = true;

  var takeoverWall = document.getElementById('tab-takeover-wall');
  if (takeoverWall) {
    var takeoverText = takeoverWall.querySelector('p');
```

## DriveApp Quota Analysis

**Write operations (sign-in only):**
- `DriveApp.createFile()` or `File.setContent()`: 1 per sign-in per user
- `File.setSharing()`: 1 per first sign-in per user (subsequent sign-ins overwrite)
- `PropertiesService.setProperty()`: 1 per sign-in
- `CacheService.put()`: 1 per sign-in

**Read operations (per poll):** **Zero** — the client reads from Drive directly via `<script>` tag. No GAS server involvement.

**Daily DriveApp quota impact (10 users, 8hr day, average 2 sign-ins per user per day):**
- DriveApp calls: ~20/day (negligible)
- GAS execution quota for polling: **0** (the entire polling cost is offloaded to the browser + Drive CDN)

**This is the key advantage of this approach** — the polling quota cost is truly zero.

## Alternatives Considered Within This Plan

### Alternative: New File Per Sign-In (Instead of Overwriting)

**What:** Instead of overwriting the same Drive file, create a new file each sign-in. Pass the new file ID to the client. The old device's `<script>` tag loads the old file (unchanged), so it sees the old timestamp. Wait — this doesn't work. The old device is loading the *old* file, which still has the old timestamp. The eviction signal would need to come from the old file being deleted or modified, not from a new file being created.

**Verdict:** Overwriting the same file is the correct approach. New-file-per-sign-in doesn't help with the CDN caching problem and doesn't provide the eviction signal.

### Alternative: Dual-Channel (Drive Beacon + `google.script.run` Fallback)

**What:** Use the Drive beacon as the primary check (zero server cost). If the beacon has been unchanged for >2 minutes (suggesting CDN staleness), fall back to a single `google.script.run.checkSessionValid()` call to verify.

**This is the most robust option if pursuing the Drive approach.** It gives you:
- Zero server cost in the common case (CDN serves fresh content)
- Guaranteed detection within 2 minutes in the worst case (fallback to server check)
- Self-healing — if CDN caching resolves, the fallback never fires

**Implementation:** add a `_lastBeaconChangeTime` variable. If the beacon timestamp hasn't changed for `2 * 60 * 1000` ms (2 minutes), fire a single `google.script.run.checkSessionValid()` call. If it returns `{ valid: false }`, evict. If valid, reset the timer. This requires the GAS iframe to have the `checkSessionValid` function from Plan 9.1 Phase 1 — making this a hybrid of plans 9.1 and 9.1.1.

- [ ] **XSS resistance:** Manually edit the Drive beacon file to contain `alert('XSS')` instead of the expected content. - [ ] **NOTE:** This test will **fail** — the `<script>` tag executes whatever JavaScript is in the file. This is the fundamental XSS risk of this approach. The mitigation is controlling access to the Drive file (only the GAS service account can write to it). If the Drive account is compromised, the attacker has full XSS capability. **Document this as an accepted risk if proceeding.**
- [ ] **Beacon content validation:** Modify the Drive file to return malformed JSON. Verify the polling function handles it gracefully (structure validation rejects it, no eviction)
- [ ] **Run the security test suite.** New test for beacon-based session check should pass

## What This Plan Does NOT Address

1. **CDN cache invalidation guarantees** — Google does not document Drive's CDN TTL for public files. The detection latency of this approach is fundamentally unbounded. Empirical testing may show acceptable behavior, but there is no SLA
2. **Drive file security** — the beacon file is publicly readable. It contains a truncated email and a timestamp. An attacker who discovers the file URL can monitor when a specific user signs in. The truncated email limits exposure but doesn't eliminate it
3. **Drive storage cleanup** — beacon files persist until explicitly deleted. Long-running deployments with many users will accumulate files. A periodic cleanup mechanism (time-triggered Apps Script) would be needed
4. **Items 2-6 from Plan 9** (real-time eviction, multiple concurrent sessions, cross-tab transfer, handoff notification, quota monitoring) — same limitations apply
