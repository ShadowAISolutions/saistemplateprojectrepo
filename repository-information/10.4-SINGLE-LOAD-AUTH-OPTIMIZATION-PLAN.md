# 10.4 — Single-Load Auth Optimization Plan

> **Goal:** Reduce the GAS login flow from 2 `doGet()` executions to 1, halving quota consumption per sign-in while maintaining full HIPAA compliance and all existing security guarantees.

> **Status:** Ready to implement
> **Prerequisite:** Phases 1-7, 9 complete (Phase 8 CSP not yet active — this optimization should be implemented BEFORE Phase 8, since it modifies the inline `<script>` block)
> **Affects:** `testauth1.gs` (GAS), `testauth1.html` (HTML)
> **HIPAA impact:** Neutral — no new data exposure; reduces attack surface by eliminating one network round-trip

## 1. Problem Statement

Every user login currently triggers **two separate GAS `doGet()` executions**:

| Load | URL | Purpose | GAS calls |
|------|-----|---------|-----------|
| Load 1 | `?exchangeToken=<googleToken>` | Exchange Google OAuth token for session | CacheService ×3-5, SpreadsheetApp ×2-4 |
| Load 2 | `?session=<sessionToken>` | Validate session, return app HTML + signed `gas-auth-ok` | CacheService ×2, `google.script.run` ×1 (signAppMessage) |

**The second load (`?session=`) exists solely because the first load returns a tiny postMessage-only HTML stub** (lines 1352-1354 in `testauth1.gs`) instead of the full app HTML. The HTML parent receives `gas-session-created`, saves the session locally, then reloads the iframe with `?session=` to get the actual app UI.

### Why this matters

- **Quota cost:** Each `doGet()` counts as one execution against the per-account daily quota. Two loads per login means ~2× quota consumption for the most critical operation
- **Latency:** The second iframe load requires a full HTTP round-trip to Google's servers (~1-3 seconds), during which the user sees nothing
- **Network visibility:** Two IDLE→BUSY→IDLE cycles in DevTools, which the developer noticed and flagged
- **Attack surface:** Each network round-trip is an opportunity for interception — fewer round-trips = smaller attack surface (HIPAA principle: minimize ePHI exposure windows)

### What we can do

The `?exchangeToken` path already has all the information needed to build the full app HTML — it just discards it and returns a stub. If we make that path return the full app HTML (with the signed `gas-auth-ok` message), the `?session=` reload becomes unnecessary.

The `?session=` path is still needed for:
- **Page refresh** (session resume from localStorage)
- **"Use Here" tab reclaim**
- **Tab duplicate recovery**

These continue to use `?session=` as before — only the initial OAuth login flow changes

## 3. Proposed Architecture (1-Load Flow)

### Core idea

After `exchangeTokenForSession()` succeeds, **build and return the full app HTML directly** — don't return a postMessage stub. The `?exchangeToken` path and `?session=` path converge into the same app HTML builder, eliminating the iframe reload.

### URL Exchange Path — Optimized

```
User clicks "Sign In" → Google OAuth → handleTokenResponse()
    │
    ▼
┌─ SINGLE LOAD: iframe.src = baseUrl + '?exchangeToken=' + googleToken ────┐
│  GAS doGet():                                                             │
│    exchangeTokenForSession(googleToken)                                   │
│      ├─ (all existing checks — rate limit, token validation, ACL, etc.)   │
│      └─ Return: {success, sessionToken, email, displayName, messageKey}   │
│                                                                           │
│  IF success:                                                              │
│    validateSession(sessionToken)  ← validates the just-created session    │
│    Build full app HTML (same code as ?session= path)                      │
│      ├─ Embeds _sessionToken, user email, version                         │
│      ├─ Inline script sends BOTH messages:                                │
│      │   1. postMessage({type: 'gas-session-created', ...})               │
│      │   2. google.script.run.signAppMessage() → gas-auth-ok             │
│      └─ Registers all message listeners                                   │
│                                                                           │
│  IF failure:                                                              │
│    Return: postMessage stub with error (same as current)                  │
└───────────────────────────────────────────────────────────────────────────┘
    │
    ▼  HTML receives gas-session-created:
    │    ├─ Import HMAC key
    │    ├─ Save session to localStorage
    │    ├─ Set _pendingSessionShow (deferred)
    │    ├─ SKIP iframe reload (new: detect that app HTML is already loaded)
    │    └─ Wait for gas-auth-ok
    │
    ▼  HTML receives gas-auth-ok (from same iframe, no reload):
    │    ├─ showApp(email, displayName)
    │    ├─ startCountdownTimers()
    │    ├─ startHeartbeat()
    │    └─ broadcastTabClaim()
    │
    ▼  ✅ APP VISIBLE (one load instead of two)
```

**Result: 1 `doGet()` + 1 `google.script.run` = 2 GAS executions (down from 3)**

### postMessage Exchange Path (HIPAA) — Optimized

```
User clicks "Sign In" → Google OAuth → handleTokenResponse()
    │
    ▼
┌─ LOAD 0: iframe.src = baseUrl (no params) ───────────────────────────────┐
│  GAS doGet(): returns postMessage listener page                           │
│    Posts: {type: 'gas-ready-for-token'}                                   │
│    Listens for: {type: 'exchange-token', accessToken, nonce}              │
│    On receive: google.script.run.exchangeTokenForSession(token)           │
│    On success: google.script.run.buildSignedAppPayload(sessionToken)      │
│      └─ Returns: {sessionCreatedData, signedAuthOk, appHtml}             │
│    Reconstructs page with app HTML + posts both messages                  │
└───────────────────────────────────────────────────────────────────────────┘
    │
    ▼  HTML receives gas-session-created + gas-auth-ok from same load
    │    ├─ Import HMAC key, save session, show app
    │    └─ No iframe reload needed
    │
    ▼  ✅ APP VISIBLE
```

**Wait — the HIPAA path has a constraint.** The `google.script.run` call happens inside the GAS iframe, which is a sandboxed environment. The iframe can't rewrite itself with new HTML after `exchangeTokenForSession()` returns — `google.script.run` returns data, not HTML.

### HIPAA path — revised approach

For the HIPAA path, the optimization is different. Instead of trying to serve app HTML from inside `google.script.run`, we **have `exchangeTokenForSession()` return a flag telling the HTML parent that app HTML is already available**, and the HTML parent sets the iframe to `?session=<token>&skipValidation=true` — wait, that introduces a security hole.

**Better approach for HIPAA path:** Keep the existing 2-load behavior for HIPAA mode. The HIPAA `google.script.run` architecture inherently requires a second load because the first load is a listener page (not controlled by `doGet` URL params). The optimization applies cleanly to the **URL exchange path only**.

However, there IS an optimization available for HIPAA: **have `exchangeTokenForSession()` also call `signAppMessage()` server-side and return the signed `gas-auth-ok` payload alongside the session data**. Then the listener page can post BOTH `gas-session-created` AND the signed `gas-auth-ok` to the parent. The parent receives both messages, imports the key, saves the session, and immediately shows the app — **but still needs the `?session=` reload to get the app HTML with its inline scripts (version polling, user-activity reporting, save-note, etc.)**.

The app HTML can't be delivered via postMessage — it needs to be the actual iframe content for `google.script.run` to work from within it.

**Conclusion: The HIPAA path stays at 2 loads but reduces from 3 GAS executions to 2** (the `signAppMessage` call is folded into the exchange). The URL path goes from 2 loads to 1 load (2 GAS executions to 2, but one `doGet()` eliminated).

### Final optimized architecture per path

| Path | Before | After | Savings |
|------|--------|-------|---------|
| URL exchange | 2 `doGet()` + 1 `google.script.run` = 3 exec | 1 `doGet()` + 1 `google.script.run` = 2 exec | 1 `doGet()` saved (33%) |
| postMessage (HIPAA) | 2 `doGet()` + 1 `google.script.run` + 1 `google.script.run` = 4 exec | 2 `doGet()` + 1 `google.script.run` = 3 exec | 1 `google.script.run` saved (25%) |
| Session resume (both) | 1 `doGet()` + 1 `google.script.run` = 2 exec | unchanged | 0 |

### What changes where

**GAS (`testauth1.gs`):**
1. URL exchange path: after successful `exchangeTokenForSession()`, call `validateSession()` + build full app HTML (reuse existing code block) instead of returning the postMessage stub
2. postMessage listener page: after `exchangeTokenForSession()` succeeds, also call `signAppMessage()` and return the signed `gas-auth-ok` alongside the session data — so the listener page can post both messages

**HTML (`testauth1.html`):**
1. `gas-session-created` handler: add a flag/signal check — if the iframe already contains the app HTML (i.e. came from the optimized `?exchangeToken` path), skip the iframe reload. The presence of `gas-auth-ok` arriving shortly after from the same iframe confirms this
2. The simplest signal: include an `appLoaded: true` field in the `gas-session-created` payload from the optimized path. When `appLoaded` is true, skip the `gasApp.src = baseUrl + '?session=...'` reload

## 4. HIPAA Compliance Analysis

### Does this optimization change what data is transmitted?

**No.** The exact same data flows between the exact same parties (parent HTML ↔ GAS iframe). The only difference is *when* and *how many times* the GAS iframe is loaded — the data payloads are identical.

| Data element | Before (2-load) | After (1-load) | Change? |
|-------------|-----------------|-----------------|---------|
| Google OAuth token | Sent in URL param to GAS | Same | No |
| Session token | Returned via postMessage, then sent in URL param | Returned via postMessage, then available in-page | **Slightly better** — token no longer appears in a second URL |
| HMAC messageKey | Returned via postMessage | Same | No |
| User email/displayName | Returned via postMessage | Same | No |
| Signed `gas-auth-ok` | Via `google.script.run` → postMessage | Same | No |

### HIPAA Security Rule checklist

| Safeguard | Requirement | Current status | After optimization |
|-----------|-------------|----------------|-------------------|
| **Transmission security** (§164.312(e)(1)) | ePHI encrypted in transit | All communication over HTTPS (GAS enforces TLS). postMessage uses targeted origin | **Unchanged** — same HTTPS transport |
| **Access controls** (§164.312(a)(1)) | Unique user identification, emergency access | OAuth + session token + ACL check | **Unchanged** — same auth chain |
| **Audit controls** (§164.312(b)) | Record and examine activity | `auditLog()` called during `exchangeTokenForSession()` | **Unchanged** — same audit calls |
| **Integrity** (§164.312(c)(1)) | Protect ePHI from improper alteration | HMAC-SHA256 message signing | **Unchanged** — same HMAC mechanism |
| **Authentication** (§164.312(d)) | Verify identity of person seeking access | Google OAuth + domain restriction + ACL | **Unchanged** |
| **Minimum necessary** (§164.502(b)) | Limit PHI exposure to what's needed | Only email + display name transmitted (no PHI) | **Improved** — one fewer network round-trip = one fewer exposure window |

### What about the session token appearing in the URL?

Currently, the `?session=<token>` reload puts the session token in the iframe's URL. This was flagged as a concern in Phase 7 for heartbeat/signout (which were migrated to postMessage). For the initial login flow:

- **Before optimization:** session token appears in the `?session=` URL on the second load
- **After optimization (URL path):** session token never appears in any URL during the login flow — it's only in the `gas-session-created` postMessage payload (targeted origin, not logged in browser history)
- **After optimization (HIPAA path):** session token still appears in `?session=` URL on the second load (no change for HIPAA mode)

**This is a net positive for HIPAA compliance** — the URL path eliminates one instance of token-in-URL exposure during login.

### postMessage security — is it safe for session data?

The existing postMessage architecture already handles sensitive data (session tokens, HMAC keys, user info) via postMessage with origin-validated delivery. This optimization doesn't change the postMessage security model — the same origin checks, nonce verification, and first-write-wins guards apply.

Per OWASP and web security best practices:
- postMessage with a targeted origin (`PARENT_ORIGIN`, not `'*'`) is secure against cross-origin interception
- The GAS sandbox origin is unpredictable, so the HTML→GAS direction uses `'*'` — but this is a constraint of the GAS platform, not a new risk
- The GAS→HTML direction always uses `PARENT_ORIGIN` — only the embedding page receives the messages

### HIPAA verdict

**This optimization is HIPAA-neutral to HIPAA-positive:**
- No new data exposure paths
- One fewer network round-trip (URL path) = smaller attack surface
- Session token removed from one URL (URL path) = better token hygiene
- All existing security mechanisms (HMAC, nonce, origin validation, audit logging) remain unchanged
- The `?session=` path continues to exist for page resume/tab reclaim — only the initial login flow is optimized

## 5. GAS Quota Impact Analysis

### Google Apps Script quota structure (verified via web search)

Per the official [Google Apps Script Quotas page](https://developers.google.com/apps-script/guides/services/quotas) (last updated March 13, 2026):

- **`doGet()`/`doPost()` are NOT subject to trigger quotas** — they are web app entry points, not triggers
- **Quotas are per-account** (not per-script) — all scripts under one Google account share the same pool
- **URL Fetch quota:** Consumer (gmail.com) = 20,000/day; Workspace = 100,000/day
- **Execution time limit:** 6 minutes per execution (both consumer and workspace)
- **Trigger runtime:** 90 minutes cumulative/day (but `doGet`/`doPost` don't count as triggers)

### Quota consumption per login

| Operation | Type | Before | After (URL path) | After (HIPAA path) |
|-----------|------|--------|-------------------|-------------------|
| `doGet(?exchangeToken=)` | Web app entry | 1 | 1 | N/A |
| `doGet()` no params (HIPAA listener) | Web app entry | N/A | N/A | 1 |
| `google.script.run.exchangeTokenForSession()` | Client-side call | N/A | N/A | 1 |
| `doGet(?session=)` | Web app entry | 1 | **0 (eliminated)** | 1 |
| `google.script.run.signAppMessage()` | Client-side call | 1 | 1 | **0 (folded into exchange)** |
| **Total GAS executions** | | **3** | **2** | **3 → 2** |

### Projected savings at scale

For a deployment with N users signing in per day:

| Metric | Before | After | Savings |
|--------|--------|-------|---------|
| GAS executions per login (URL) | 3 | 2 | 33% |
| GAS executions per login (HIPAA) | 4 | 3 | 25% |

### CacheService and SpreadsheetApp calls

The optimization adds one `validateSession()` call to the `?exchangeToken` path (to validate the just-created session before building app HTML). This is a CacheService read + write — negligible cost, and it's offset by eliminating the entire second `doGet()` which did the same thing.

**Net CacheService calls per login: unchanged** (the validation just moves from Load 2 to Load 1)

**Changes:**
- Extract lines 1422-1607 into `buildAppHtml(sessionToken, session, appMsgKey)`
- Both call sites pass the same parameters
- No behavioral change — pure refactor

### Step 2: GAS — URL exchange path returns full app HTML on success

**File:** `testauth1.gs` (lines 1328-1358)
**Why:** Instead of returning the postMessage stub, return the full app HTML with an additional `gas-session-created` postMessage embedded.

**Before (current code, lines 1342-1357):**
```javascript
var payload = JSON.stringify({
  type: "gas-session-created",
  success: result.success,
  sessionToken: result.sessionToken || "",
  // ...
});
var exchangeHtml = '<!DOCTYPE html><html><body><script>'
  + 'try { window.top.postMessage(' + payload + ', '
  + JSON.stringify(PARENT_ORIGIN) + '); } catch(e) {}'
  + '</' + 'script></body></html>';
return HtmlService.createHtmlOutput(exchangeHtml)...
```

**After (new logic):**
```javascript
if (result.success) {
  // Validate the just-created session
  var session = validateSession(result.sessionToken);

  // Retrieve messageKey
  var appMsgKey = result.messageKey || '';

  // Build the session-created payload for the parent page
  var sessionCreatedPayload = JSON.stringify({
    type: "gas-session-created",
    success: true,
    sessionToken: result.sessionToken,
    email: result.email || "",
    displayName: result.displayName || "",
    absoluteTimeout: result.absoluteTimeout || 0,
    messageKey: result.messageKey || "",
    appLoaded: true  // ← NEW FLAG: tells parent to skip iframe reload
  });

  // Build full app HTML — same as ?session= path
  // but prepend a gas-session-created postMessage before the app script
  return buildAppHtml(result.sessionToken, session, appMsgKey, sessionCreatedPayload);
} else {
  // Error path — return stub with error (unchanged)
  var errorPayload = JSON.stringify({...});
  var exchangeHtml = '<!DOCTYPE html>...' // same as current error stub
  return HtmlService.createHtmlOutput(exchangeHtml)...
}
```

**Key detail:** The `buildAppHtml` function needs a new optional parameter `sessionCreatedPayload`. When provided, the generated HTML includes an additional inline script that posts `gas-session-created` to the parent **before** the `signAppMessage` call. This ensures the parent receives `gas-session-created` first (to import the HMAC key and save the session) and then `gas-auth-ok` (to show the app).

### Step 3: GAS — Modify `buildAppHtml` to optionally include `gas-session-created`

**File:** `testauth1.gs`

Add a `sessionCreatedPayload` parameter to `buildAppHtml()`. When non-null, inject a script block at the top of the inline `<script>` that posts this payload:

```javascript
function buildAppHtml(sessionToken, session, appMsgKey, sessionCreatedPayload) {
  var html = `<html><head>...</head><body>
    ...app markup...
    <script>
      var _sessionToken = '${escapeJs(sessionToken)}';

      // If this is a fresh login (not session resume), notify the parent
      // that a new session was created — BEFORE signAppMessage fires
      ${sessionCreatedPayload
        ? 'try { window.top.postMessage(' + sessionCreatedPayload
          + ', ' + JSON.stringify(PARENT_ORIGIN) + '); } catch(e) {}'
        : '// Session resume — no gas-session-created needed'}

      // Notify wrapper that auth is OK (same as current)
      google.script.run
        .withSuccessHandler(function(signed) {
          window.top.postMessage(signed, '${PARENT_ORIGIN}');
        })
        .withFailureHandler(function(err) {
          // Fallback (same as current)
          window.top.postMessage({type: 'gas-auth-ok', ...}, '${PARENT_ORIGIN}');
        })
        .signAppMessage(_sessionToken, 'gas-auth-ok');

      // ... rest of listeners (version-check, user-activity, save-note) ...
    </script>
  </body></html>`;

  return HtmlService.createHtmlOutput(html)
    .setTitle(TITLE)
    .setXFrameOptionsMode(HtmlService.XFrameOptionsMode.ALLOWALL);
}
```

**Important ordering:** The `gas-session-created` postMessage must fire BEFORE `google.script.run.signAppMessage()`. Since inline scripts execute top-to-bottom synchronously, and `postMessage` is synchronous while `google.script.run` is async, the ordering is guaranteed:

1. `gas-session-created` → parent receives, imports HMAC key
2. `google.script.run.signAppMessage()` fires async → GAS returns signed `gas-auth-ok`
3. Parent receives `gas-auth-ok`, verifies signature (key already imported), shows app

### Step 4: HTML — Skip iframe reload when `appLoaded` flag is set

**File:** `testauth1.html` (lines 1508-1516 in `gas-session-created` handler)

**Before (current code):**
```javascript
// Normal sign-in flow (OAuth) — reload iframe with session token
var gasApp = document.getElementById('gas-app');
var baseUrl = (gasApp && gasApp.dataset.baseUrl) || '';
if (gasApp && baseUrl) {
  _expectingSession = true;
  gasApp.src = baseUrl + '?session=' + encodeURIComponent(data.sessionToken);
}
```

**After:**
```javascript
// If appLoaded flag is set, the iframe already contains the full app HTML
// (optimized single-load path) — skip the reload. gas-auth-ok will arrive
// from the same iframe via google.script.run.signAppMessage().
if (data.appLoaded) {
  // App HTML already loaded — just wait for gas-auth-ok
  _expectingSession = false;  // No iframe reload happening
} else {
  // Legacy path (HIPAA postMessage exchange) or fallback — reload iframe
  var gasApp = document.getElementById('gas-app');
  var baseUrl = (gasApp && gasApp.dataset.baseUrl) || '';
  if (gasApp && baseUrl) {
    _expectingSession = true;
    gasApp.src = baseUrl + '?session=' + encodeURIComponent(data.sessionToken);
  }
}
```

### Step 5: GAS — HIPAA path optimization (fold `signAppMessage` into exchange)

**File:** `testauth1.gs` (lines 1361-1399, postMessage listener page)

For the HIPAA path, we can't eliminate the `?session=` reload (the listener page can't transform into app HTML). But we CAN fold the `signAppMessage` call into the exchange result, so the `?session=` path doesn't need to call `signAppMessage` separately.

**Modified listener page script:**
```javascript
'google.script.run'
+ '  .withSuccessHandler(function(result) {'
+ '    if (result.success) {'
+ '      // Also get the signed gas-auth-ok to include with session data'
+ '      google.script.run'
+ '        .withSuccessHandler(function(signedAuthOk) {'
+ '          result.signedAuthOk = signedAuthOk;'  // piggyback on same message
+ '          window.top.postMessage({...result, nonce: nonce}, ORIGIN);'
+ '        })'
+ '        .withFailureHandler(function() {'
+ '          // Proceed without pre-signed auth — ?session= path will sign it'
+ '          window.top.postMessage({...result, nonce: nonce}, ORIGIN);'
+ '        })'
+ '        .signAppMessage(result.sessionToken, "gas-auth-ok");'
+ '    } else {'
+ '      window.top.postMessage({...result, nonce: nonce}, ORIGIN);'
+ '    }'
+ '  })'
+ '  .exchangeTokenForSession(token);'
```

**Wait — this adds complexity to the HIPAA path for marginal gain** (saving one `google.script.run` call, not a `doGet()`). The `?session=` path will still need `signAppMessage` for session resume, so the call can't be removed from the app HTML.

**Revised recommendation:** Skip Step 5. The HIPAA path optimization is marginal and adds complexity. Focus on the URL path optimization (Steps 1-4) which provides the clear win of eliminating a full `doGet()`.

## 7. Security Invariants (Must Not Break)

Every security mechanism from Phases 1-9 must remain functional after this optimization. This is the compliance checklist:

| # | Invariant | Phase | How to verify | Risk if broken |
|---|-----------|-------|---------------|----------------|
| 1 | HMAC key import via `_importHmacKey()` succeeds | Phase 1 | `console.log(_hmacKeySet, _hmacKey)` → `true, CryptoKey` | Messages unverifiable |
| 2 | First-write-wins: duplicate `gas-session-created` rejected | Phase 5 | Post forged message → `_hmacKey` unchanged | Key overwrite attack |
| 3 | Nonce verification on HIPAA path | Phase 2 | Nonce mismatch → message rejected | Replay attack |
| 4 | Origin validation on all postMessages | Phase 2 | Forged message → rejected at Layer 0 | Cross-origin injection |
| 5 | `gas-auth-ok` signature verification | Phase 1 | HMAC-signed message → verified by Web Crypto | Message forgery |
| 6 | Session timer immutability | Phase 6 | `window.SERVER_SESSION_DURATION = 999` → ignored | Timer tampering |
| 7 | Deferred `showApp()` until `gas-auth-ok` | Design | Forged `gas-session-created` → app stays hidden | UI spoofing |
| 8 | `_expectingSession` guard | Design | Stale `gas-needs-auth` during exchange → ignored | Auth wall flash |
| 9 | BroadcastChannel: no credentials | Phase 4 | BC message has only `type` + `tabId` | Token leak |
| 10 | Token-in-URL elimination (heartbeat, signout) | Phase 7 | Network tab: no tokens in URLs | History exposure |
| 11 | `window._r` removed | Phase 9 | `typeof window._r === 'undefined'` → `true` | GAS URL exposure |
| 12 | Session resume still works | Existing | Refresh page → app loads from localStorage | Login loop |
| 13 | "Use Here" tab reclaim still works | Existing | Tab B claims → Tab A reclaims → app visible | Multi-tab broken |
| 14 | Sign-out clears HMAC key | Phase 5 | After sign-out: `_hmacKey === null` | Key persistence |

### Ordering constraint

The `gas-session-created` postMessage MUST arrive at the parent BEFORE `gas-auth-ok`. The parent needs to:
1. Import the HMAC key from `gas-session-created.messageKey`
2. Use that key to verify the signature on `gas-auth-ok`

If `gas-auth-ok` arrives first, signature verification fails (no key imported yet).

**This ordering is guaranteed** because:
- `window.top.postMessage()` is synchronous (fires immediately)
- `google.script.run.signAppMessage()` is async (requires server round-trip)
- The inline script executes `postMessage` first, then calls `google.script.run`
- Therefore `gas-session-created` is delivered before `gas-auth-ok` even starts

### Edge case: `gas-auth-ok` fallback path

The `signAppMessage` failure handler sends an unsigned `gas-auth-ok` with the `messageKey` included. This fallback is needed when `google.script.run` fails (network issue, GAS quota exceeded).

In the optimized flow, the parent already has the HMAC key from `gas-session-created`. The fallback `gas-auth-ok` carries `messageKey` again, but `_hmacKeySet` is already `true`, so the duplicate import is safely skipped. The unsigned message is accepted because it matches the existing "first message after key import" pattern.

**No change needed** — the fallback path is already safe

## 9. Rollback Plan

The optimization is fully backwards-compatible. To rollback:

1. **GAS:** Revert the `?exchangeToken` path to return the postMessage stub (remove the `buildAppHtml` call, restore lines 1342-1357)
2. **HTML:** Remove the `appLoaded` check in the `gas-session-created` handler (always reload iframe)
3. Redeploy GAS

The `?session=` path is unchanged throughout — session resume, tab reclaim, and all non-login flows continue to work regardless of whether the optimization is active.

**Zero-downtime rollback:** Since the HTML checks for `data.appLoaded` and falls back to the reload path when it's absent, rolling back the GAS side alone is sufficient — the HTML will automatically use the 2-load flow when it doesn't see the `appLoaded` flag. This means you can rollback by reverting only `testauth1.gs` and redeploying, without touching `testauth1.html`.

### Feature flag option

If you want a server-side toggle, add a config flag:

```javascript
// In AUTH_CONFIG presets:
SINGLE_LOAD_AUTH: true,  // true = return full app HTML on exchange, false = legacy stub
```

Then in the `?exchangeToken` path:
```javascript
if (result.success && AUTH_CONFIG.SINGLE_LOAD_AUTH) {
  // Optimized: build app HTML
  return buildAppHtml(result.sessionToken, session, appMsgKey, sessionCreatedPayload);
} else if (result.success) {
  // Legacy: postMessage stub (current behavior)
  return HtmlService.createHtmlOutput(exchangeHtml)...
}
```

This allows toggling the optimization per-preset without code changes
