## Why This Document Exists

This is the **single comprehensive reference** for the testauth1 authentication system. Previous plans (01–11) each addressed a specific aspect — pattern design, security hardening, cross-device enforcement, EMR data access — but no single document captured the complete picture: how all the pieces fit together, what's implemented, what's planned, what the security audit found, and what remains to be done.

This document serves three audiences:

1. **Implementers** — developers who need to understand the full system before making changes. Every architectural decision, constraint, and gotcha is documented here so the same mistakes aren't repeated (see the [Known Constraints](KNOWN-CONSTRAINTS-AND-FIXES.md) file for the painful history)

2. **Security auditors** — reviewers who need to assess the system's security posture against OWASP, NIST, and HIPAA standards. The [Security Audit Findings](#security-audit-findings--remediation-status) section maps every finding to its remediation status

3. **Compliance officers** — stakeholders who need to verify HIPAA Technical Safeguard coverage. The [HIPAA 2026 Compliance Readiness](#hipaa-2026-compliance-readiness) section provides a requirement-by-requirement assessment

**Relationship to other documents:** This document is a reference synthesis — it does not replace the individual plans. Plans 7–11 contain the detailed implementation code (exact line numbers, current/modified code blocks, test procedures). This document provides the architectural context that makes those plans comprehensible as a unified system.

## Executive Summary

The testauth1 authentication system is a **client-side HTML host page + server-side Google Apps Script (GAS)** architecture designed for Google OAuth-based authentication with optional HIPAA-level security hardening. It is deployed as a static HTML page on GitHub Pages that embeds a GAS web app in an iframe.

### What's Implemented (Plans 6–9.2)

| Capability | Implementation | Plan |
|-----------|---------------|------|
| **Google OAuth Sign-In** | Google Identity Services (GIS) token client with `select_account` prompt | Plan 6 |
| **Server-Side Sessions** | GAS CacheService with UUID-based opaque session tokens (not raw OAuth tokens) | Plan 6 |
| **Session Token Exchange** | Dual-mode: URL parameter (standard) or postMessage (HIPAA) | Plan 6 |
| **Rolling Session Timeout** | Configurable idle timeout (15min HIPAA / 30min standard) with activity-gated heartbeat | Plan 6 |
| **Absolute Session Timeout** | Maximum session lifetime (8hr HIPAA / 16hr standard) regardless of activity | Plan 6 |
| **HMAC Message Integrity** | HMAC-SHA256 signing of GAS→HTML postMessages using per-session key derived from server secret | Plans 7–8 |
| **Message Type Allowlist** | Only known message types processed; unknown types silently dropped + reported | Plan 7 |
| **Security Event Reporting** | Blocked attacks reported to GAS backend with rate limiting | Plan 7 |
| **Domain Restriction** | Optional allowlist of permitted Google Workspace domains (HIPAA preset) | Plan 6 |
| **Cross-Device Enforcement** | Heartbeat-piggyback tombstone detection — new sign-in evicts old session | Plan 9.2 |
| **Single-Tab Enforcement** | BroadcastChannel-based tab coordination — only one active tab per session | Plan 8 |
| **ACL-Based Access Control** | Master ACL spreadsheet with per-page, per-user access permissions | Plan 6 |
| **IP Format Validation** | `_validateIp()` prevents log injection attacks | Plan 8 |

| **Referrer Policy** | `no-referrer` prevents token leakage via HTTP Referer header | Plan 7 |

### Key Architectural Decisions

| Decision | Rationale | Reference |
|----------|-----------|-----------|
| **Opaque session tokens (not raw OAuth tokens)** | OAuth tokens grant access to Google APIs; session tokens only grant access to the app. Limits blast radius of token theft | Plan 3 |
| **Server-side session storage (CacheService)** | Sessions exist only on Google's servers. Client holds only a lookup key. No session data in browser storage except the token string and email | Plan 3 |
| **Dual storage: localStorage (standard) vs sessionStorage (HIPAA)** | localStorage persists across tabs/restarts for convenience; sessionStorage is per-tab and clears on close for HIPAA compliance | Plan 6 |
| **Activity-gated heartbeat** | Heartbeat only fires when user is actively interacting. Idle users don't consume GAS quota. Addresses the 20K/day execution limit on consumer accounts | Plan 9.2 |
| **HTML auth layer = front door only** | No PHI stored, displayed, or processed at the HTML layer. All patient data lives exclusively in GAS/Google Sheets. The HTML layer handles sign-in, session state, and UI chrome | Audit clarification |
| **Config-driven security levels** | One codebase, two security profiles (`standard`/`hipaa`). Every security feature is behind a toggle. Switching is a code change, not a runtime toggle — intentional for compliance | Plan 6 |

## Hard Constraints

These are architectural rules derived from production failures. **Every constraint has broken the system when violated.** See [KNOWN-CONSTRAINTS-AND-FIXES.md](KNOWN-CONSTRAINTS-AND-FIXES.md) for the full history.

### Constraint A — postMessage `'*'` targetOrigin (HTML → GAS)

**Rule:** When the HTML page sends a `postMessage` to the GAS iframe, the `targetOrigin` **must be `'*'`**.

**Why:** Google Apps Script uses a double-iframe sandbox:
1. The `<iframe>` `src` points to `script.google.com/macros/s/.../exec`
2. That URL serves a shell page that creates an **inner** iframe at `*.googleusercontent.com`
3. The actual GAS listener code runs inside the inner sandbox frame

The sandbox origin is dynamically generated and unpredictable. Setting `targetOrigin` to `'https://script.google.com'` causes the browser to silently drop the message.

**Mitigation:** The token sent via `'*'` is single-use (`_pendingToken` nulled after sending), and `<meta name="referrer" content="no-referrer">` prevents leakage via HTTP headers.

**Times violated:** 3+ (each time broke sign-in silently with zero error messages)

### Constraint B — `event.source` for HTML → GAS replies

**Rule:** Replies to GAS sandbox messages must use `event.source.postMessage(...)` — **not** `gasApp.contentWindow.postMessage(...)`.

**Why:** `gasApp.contentWindow` references the outer `script.google.com` shell frame. `event.source` targets the actual inner sandbox frame where the listener runs.

### Constraint C — GAS deploy webhook must stay unauthenticated

**Rule:** The `doPost(action=deploy)` handler **must not** require session authentication. Adding a `DEPLOY_SECRET` check in v02.79r bricked the entire auto-deploy pipeline, freezing GAS at v01.15g for an extended period.

**Defense:** The deploy action only calls `pullAndDeployFromGitHub()` which overwrites the script with GitHub's source of truth. Security comes from securing the GitHub repo (branch protection, 2FA).

### Constraint D — Google Sign-In flow cannot be short-circuited

**Rule:** The complete chain must execute in order:
```
Google OAuth popup → token callback → token exchange → GAS session creation
→ gas-session-created → HTML saves session → iframe loads authenticated content
```
Each step depends on the previous step's output. Skipping or reordering any step causes silent failure.

### Constraint E — GAS → HTML messages use `PARENT_ORIGIN`

**Rule:** GAS→HTML direction uses the known GitHub Pages origin as `targetOrigin`. This is the **opposite** of Constraint A — the asymmetry exists because GAS knows where the HTML page lives (stable GitHub Pages origin), but HTML cannot know where the GAS sandbox lives (dynamic googleusercontent.com origin).

### Constraint F — Cross-tab sign-out requires two mechanisms

**Rule:** Standard preset uses `localStorage` `storage` event (fires automatically cross-tab). HIPAA preset uses `BroadcastChannel` (because `sessionStorage` changes never fire cross-tab events). Both are correct for their storage type — do not consolidate.

## Authentication Flow — Complete Walkthrough

### Phase 1: Page Load & Initialization

```
1. Browser loads testauth1.html from GitHub Pages (static)
2. CSP meta tag enforced by browser
3. Google Identity Services (GIS) library loaded from accounts.google.com
4. initGoogleSignIn() called:
   a. google.accounts.oauth2.initTokenClient() configured with:
      - client_id: CLIENT_ID (Google OAuth Client ID — public, per Google's design)
      - scope: 'openid email profile'
      - prompt: 'select_account' (always show account chooser)
      - callback: handleTokenResponse
5. Existing session check:
   a. Read session token from storage (localStorage or sessionStorage per preset)
   b. If token exists → load GAS iframe with ?session=TOKEN
   c. If no token → show "Sign In Required" auth wall
```

### Phase 2: Google OAuth Sign-In

```
6. User clicks "Sign In with Google" button
7. Google OAuth popup opens (hosted by Google — separate origin)
8. User selects Google account and grants permissions
9. handleTokenResponse(response) receives:
   - response.access_token: Google OAuth access token (single-use for our purposes)
   - response.error: set if user cancelled or error occurred
10. If error → show error message, return
11. Access token temporarily stored as _pendingToken (nulled after use)
```

### Phase 3: Token Exchange (Standard Preset — URL Mode)

```
12. TOKEN_EXCHANGE_METHOD = 'url' (standard preset default)
13. exchangeToken(accessToken):
    a. Set iframe src = GAS_DEPLOYMENT_URL + '?exchangeToken=' + accessToken
    b. GAS doGet(e) receives the access token as URL parameter
    c. GAS calls Google's userinfo API to validate the token:
       - UrlFetchApp.fetch('https://www.googleapis.com/oauth2/v3/userinfo',
         { headers: { 'Authorization': 'Bearer ' + accessToken } })
       - Returns: { email, name, picture, hd (hosted domain) }
    d. GAS validates:
       - ACL check: is this email in the Master ACL spreadsheet for this page?
       - Domain check: if ENABLE_DOMAIN_RESTRICTION, is the domain in ALLOWED_DOMAINS?
       - Rate limit check: is this email rate-limited?
    e. If valid → exchangeTokenForSession(email, displayName):
       - Generate UUID session token (crypto-random)
       - Create session data: { email, displayName, createdAt, lastActivity,
         absoluteCreatedAt, tokenObtainedAt }
       - If ENABLE_HMAC_INTEGRITY → generate HMAC signature over session data
       - Store in CacheService with SESSION_DURATION TTL
       - Invalidate any existing sessions for this email (cross-device enforcement)
       - Write tombstone for evicted sessions (5min TTL)
       - Generate per-session messageKey for HMAC signing of postMessages
       - Audit log: session_created
    f. GAS returns HTML page containing:
       - postMessage to parent: { type: 'gas-session-created',
         sessionToken, email, displayName, messageKey }
```

### Phase 3a: Token Exchange (HIPAA Preset — postMessage Mode)

```
12a. TOKEN_EXCHANGE_METHOD = 'postMessage' (HIPAA preset default)
13a. exchangeToken(accessToken):
     a. Set iframe src = GAS_DEPLOYMENT_URL (no token in URL — cleaner)
     b. GAS iframe loads and sends: { type: 'gas-ready-for-token' }
     c. HTML receives gas-ready-for-token via message listener
     d. HTML replies via event.source.postMessage(
        { type: 'exchange-token', accessToken: _pendingToken }, '*')
        ← MUST use event.source (Constraint B) and '*' (Constraint A)
     e. _pendingToken nulled immediately (single-use)
     f. GAS receives exchange-token, validates, creates session (same as 13c-f above)
```

### Phase 4: Session Establishment

```
14. HTML receives gas-session-created postMessage
15. HMAC verification (if _messageKey is NOT set — first-write-wins):
    - First gas-session-created sets _messageKey
    - Subsequent gas-session-created messages with different key are rejected
    - 30-second replay window: reject if >30s after iframe load
16. saveSession(token, email, displayName):
    a. Store session token in storage (localStorage or sessionStorage)
    b. Store email and display name
    c. _pendingSessionShow = true (waiting for server confirmation)
17. Load authenticated app: iframe src = GAS_DEPLOYMENT_URL + '?session=' + token
18. GAS doGet(e) with session parameter:
    a. Validate session token against CacheService
    b. If valid → return app HTML with postMessage({ type: 'gas-auth-ok' })
    c. If invalid → return HTML with postMessage({ type: 'gas-needs-auth' })
19. HTML receives gas-auth-ok:
    a. Hide auth wall overlay
    b. Show app content
    c. Start heartbeat timer
    d. _pendingSessionShow = false
```

### Phase 5: Session Maintenance (Heartbeat)

```
20. Heartbeat timer fires every HEARTBEAT_INTERVAL (30s test / 5min production)
21. Activity gate: heartbeat only fires if user has interacted since last heartbeat
    (keydown, click, mousemove, touchstart, scroll events tracked)
22. Heartbeat execution:
    a. iframe src = GAS_DEPLOYMENT_URL + '?heartbeat=' + sessionToken
    b. GAS doGet(e) with heartbeat parameter:
       - Validate session exists in CacheService
       - If exists:
         · Extend session TTL (rolling timeout)
         · Check for cross-device eviction tombstone
         · Return gas-heartbeat-ok (or gas-heartbeat-expired with reason)
       - If not exists:
         · Check for tombstone (evicted_TOKEN_PREFIX)
         · If tombstone found → return gas-heartbeat-expired with reason 'evicted'
         · If no tombstone → return gas-heartbeat-expired with reason 'expired'
    c. HTML processes response:
       - gas-heartbeat-ok → session still valid, reset activity flag
       - gas-heartbeat-expired (reason: expired) → show "Session expired" overlay
       - gas-heartbeat-expired (reason: evicted) → show "Session Active Elsewhere" overlay
```

### Phase 6: Sign-Out

```
23. User clicks sign-out button
24. performSignOut():
    a. Send sign-out to GAS: iframe src = GAS_DEPLOYMENT_URL + '?signOut=' + token
    b. GAS invalidates session in CacheService
    c. GAS audit log: session_signout
    d. clearSession():
       - Remove token, email, displayName from storage
       - Null _messageKey (critical — see Fix 3 in Known Constraints)
       - Null _pendingToken
    e. Broadcast sign-out to other tabs:
       - Standard: localStorage clear triggers storage event
       - HIPAA: BroadcastChannel 'auth-sign-out' broadcast
    f. iframe src = 'about:blank' (DOM clearing — destroys any cached content)
    g. Show auth wall
```

### Session Timeout Configuration

| Parameter | Standard Preset | HIPAA Preset | Test Values (⚡) |
|-----------|----------------|-------------|-----------------|
| `SERVER_SESSION_DURATION` (rolling) | 1800s (30 min) | 900s (15 min) | 180s (3 min) |
| `ABSOLUTE_SESSION_DURATION` | 57600s (16 hr) | 28800s (8 hr) | 300s (5 min) |
| `HEARTBEAT_INTERVAL` | 300000ms (5 min) | 300000ms (5 min) | 30000ms (30s) |
| `ACTIVITY_DEBOUNCE` | 5000ms (5s) | 5000ms (5s) | 5000ms (5s) |

**Design rationale:**
- **Rolling timeout (idle):** HIPAA recommends 15 minutes for automatic logoff (§164.312(a)(2)(iii)). Standard uses 30 minutes for convenience.
- **Absolute timeout:** Maximum session lifetime regardless of activity. Prevents sessions from living indefinitely through continuous heartbeats. HIPAA preset is shorter (8hr — one work shift).
- **Activity-gated heartbeat:** The heartbeat timer fires at the configured interval, but the actual server call only executes if the user has interacted since the last heartbeat. This prevents idle tabs from consuming GAS quota while still maintaining accurate session state.

### Browser Storage Strategy

| Aspect | Standard Preset | HIPAA Preset |
|--------|----------------|-------------|
| **Storage API** | `localStorage` | `sessionStorage` |
| **Persistence** | Survives tab close, page reload | Cleared on tab close |
| **Cross-tab visibility** | Yes — all same-origin tabs share it | No — per-tab isolation |
| **Cross-tab sync** | `storage` event fires automatically | Requires `BroadcastChannel` |
| **Tab duplicate behavior** | New tab reads same storage | Duplicate tab clones storage (HTML spec) |
| **Stored values** | `gas_session_token`, `gas_user_email`, `gas_display_name` | Same keys, per-tab |

**NIST SP 800-63B §7.1 note:** "Secrets used for session binding SHOULD NOT be placed in insecure locations such as HTML5 Local Storage." Both `localStorage` and `sessionStorage` are accessible to any JavaScript on the same origin — neither provides HttpOnly-equivalent protection. This is an **architectural limitation** of client-side web apps without a traditional backend. Mitigations: CSP restricts script sources, DOM clearing on expiry destroys content, and the GAS layer independently validates every server operation.

## Cryptographic Message Integrity (HMAC)

### Overview

All GAS→HTML postMessages are signed with HMAC-SHA256 using a per-session key derived from a server-side secret. This prevents message forgery — an attacker who achieves XSS cannot fabricate valid `gas-auth-ok` or `gas-heartbeat-ok` messages without knowing the HMAC secret stored in GAS Script Properties.

**This replaced the original DJB2 hash** (Audit finding C-1), which was a non-cryptographic 32-bit hash vulnerable to brute-force collision in ~65K attempts (~milliseconds on modern hardware).

### How It Works

**Server side (testauth1.gs):**
```
1. HMAC_SECRET stored in GAS Script Properties (never exposed to client)
2. On session creation → generateSessionHmac(sessionData):
   - payload = email|createdAt|lastActivity|absoluteCreatedAt|displayName|tokenObtainedAt
   - signature = Utilities.computeHmacSha256Signature(payload, HMAC_SECRET)
   - Returns base64-encoded signature
3. On every GAS→HTML postMessage:
   - messageKey = per-session random key (delivered via gas-session-created)
   - Each message includes: { type, ...data, _sig: HMAC(messageKey, type|timestamp) }
```

**Client side (testauth1.html):**
```
1. _messageKey received via gas-session-created (first-write-wins)
2. On every incoming postMessage:
   - If _messageKey is set AND message type is not gas-session-created/gas-needs-auth:
     · _verifyMessageSignature(data, _messageKey)
     · Invalid → silently drop + report security event
   - If _messageKey is NOT set → message passes through (bootstrap phase)
```

### Security Properties

| Property | Implementation | OWASP/NIST Reference |
|----------|---------------|---------------------|
| **Algorithm** | HMAC-SHA256 (256-bit) | NIST SP 800-107r1 approved |
| **Key storage** | GAS Script Properties (server-side, never in client code) | OWASP: "Never hardcode secrets" |
| **Key length** | 64-character hex string (256 bits) recommended | NIST: key length ≥ hash output length |
| **Replay protection** | 30-second validity window on gas-session-created | IETF OAuth BCP: timestamp-based replay prevention |
| **First-write-wins** | Only first gas-session-created message accepted per session | Defense against key injection |
| **Fail behavior** | Currently fail-open when secret missing; **Plan 10 Phase 1 changes to fail-closed** | OWASP: fail-closed for security controls |

### Known Gap — Fail-Open Without Secret

**Current behavior:** If `HMAC_SECRET` is not configured in Script Properties, `generateSessionHmac()` returns `''` and `verifySessionHmac()` returns `true`. This means HMAC is silently a no-op — sessions are created and validated without integrity protection.

**Plan 10 Phase 1 fix:** Change to fail-closed. If `ENABLE_HMAC_INTEGRITY` is `true` but `HMAC_SECRET` is missing, throw an error and refuse to create sessions. Log a `security_alert` audit entry.

### Web Crypto API — Best Practices Applied

Per [W3C Web Cryptography API Level 2](https://w3c.github.io/webcrypto/) and current best practices:

- **Use SHA-256 or stronger** — SHA-1 is deprecated for HMAC. The system uses SHA-256 via GAS `Utilities.computeHmacSha256Signature()` (server-side) and `crypto.subtle` (client-side verification if implemented)
- **Include timestamps for replay protection** — the 30-second replay window on session creation prevents captured `gas-session-created` messages from being replayed
- **Rotate secrets periodically** — `HMAC_SECRET` should be rotated on a regular schedule. Rotation invalidates all existing sessions (they'll fail HMAC verification), forcing re-authentication
- **Constant-time comparison** — GAS's `===` comparison is not constant-time, making theoretical timing attacks possible. In practice, timing attacks against server-side comparisons over network latency are infeasible, but `Utilities.computeHmacSha256Signature` output comparison should ideally use a constant-time method

## Content Security Policy (CSP)

### Current CSP (Meta Tag)

The CSP is delivered via `<meta>` tag (not HTTP header — GitHub Pages does not allow custom headers for static sites):

```
default-src 'none';
script-src 'self' 'unsafe-inline' https://accounts.google.com/gsi/client https://apis.google.com;
style-src 'self' 'unsafe-inline';
img-src 'self' data: https:;
font-src 'self' https://fonts.gstatic.com;
connect-src 'self' https://accounts.google.com/gsi/ https://api.ipify.org;
frame-src https://script.google.com https://accounts.google.com/gsi/;
base-uri 'none';
form-action 'none';
upgrade-insecure-requests;
```

### CSP Analysis

| Directive | Current Value | Security Assessment |
|-----------|--------------|-------------------|
| `default-src` | `'none'` | **Excellent** — deny-by-default baseline |
| `script-src` | `'self' 'unsafe-inline' ...` | **Weakened** — `unsafe-inline` allows inline script injection if HTML injection is found (Audit H-7). Nonce-based CSP would be ideal but requires server-side nonce generation (impossible with static GitHub Pages hosting) |
| `style-src` | `'self' 'unsafe-inline'` | **Acceptable** — inline styles are lower risk than inline scripts |
| `connect-src` | Includes `https://api.ipify.org` | **Problem** — third-party IP lookup without BAA (Audit C-3). Must be removed when IP lookup is moved server-side (Plan 10 Phase 7) |
| `frame-src` | `https://script.google.com ...` | **Correct** — only allows GAS and Google auth iframes |
| `base-uri` | `'none'` | **Excellent** — prevents base tag injection |
| `form-action` | `'none'` | **Excellent** — prevents form hijacking |

### CSP Improvement Path

**The ideal strict CSP** (per [Google's CSP guide](https://csp.withgoogle.com/docs/strict-csp.html) and [OWASP CSP Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Content_Security_Policy_Cheat_Sheet.html)):

```
script-src 'nonce-{RANDOM}' 'strict-dynamic' https: 'unsafe-inline';
```

**Why we can't fully achieve this:** GitHub Pages serves static HTML — there is no server-side template engine to inject unique nonces per response. The `'unsafe-inline'` fallback is necessary because:
1. The inline `<script>` block contains the entire auth logic
2. Moving it to an external `.js` file would work but increases attack surface (the file itself could be replaced via GitHub compromise)
3. Hash-based CSP (`'sha256-...'`) is an alternative — compute the SHA-256 of the inline script block and use it in CSP. This eliminates `unsafe-inline` for known scripts while blocking injected scripts

**Recommended next step:** Switch from `'unsafe-inline'` to hash-based CSP. The inline script content is known at build time, so its hash can be precomputed and embedded in the CSP meta tag. Modern browsers ignore `'unsafe-inline'` when a hash is present, per the [CSP spec](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/CSP).

### Meta Tag vs HTTP Header Limitations

| Feature | HTTP Header | Meta Tag |
|---------|-------------|----------|
| `frame-ancestors` | ✅ Supported | ❌ Not supported |
| `report-uri` / `report-to` | ✅ Supported | ❌ Not supported |
| Nonce injection | ✅ Server generates per-request | ❌ Static — same nonce every time (defeats purpose) |
| Hash-based | ✅ Supported | ✅ Supported |
| `sandbox` | ✅ Supported | ❌ Not supported |

**Implication:** We cannot use CSP reporting or `frame-ancestors` with GitHub Pages hosting. For clickjacking protection, the GAS side sets `X-Frame-Options` on its responses.

## postMessage Security Model

### Message Flow Diagram

```
    HTML Host Page                                    GAS Sandbox
    (GitHub Pages origin)                        (googleusercontent.com)
         │                                              │
         │  ────── gas-ready-for-token ──────────────►  │
         │         (unsigned — bootstrap)               │
         │                                              │
         │  ◄────── exchange-token ──────────────────   │
         │          targetOrigin: '*' (Constraint A)    │
         │          via event.source (Constraint B)     │
         │                                              │
         │  ────── gas-session-created ──────────────►  │
         │         { sessionToken, messageKey }         │
         │         targetOrigin: PARENT_ORIGIN          │
         │         (Constraint E)                       │
         │                                              │
         │  ◄────── ?session=TOKEN (URL reload) ─────   │
         │                                              │
         │  ────── gas-auth-ok ──────────────────────►  │
         │         HMAC signed with messageKey          │
         │         targetOrigin: PARENT_ORIGIN          │
         │                                              │
         │  ────── gas-heartbeat-ok ─────────────────►  │
         │         HMAC signed                          │
         │         (repeats every heartbeat interval)   │
         │                                              │
         │  ────── gas-user-activity ─────────────────► │
         │         (iframe detects user interaction)     │
```

### Message Type Allowlist

Only these message types are processed — all others are silently dropped and reported:

| Message Type | Direction | Signed? | Purpose |
|-------------|-----------|---------|---------|
| `gas-ready-for-token` | GAS → HTML | No (bootstrap) | GAS iframe loaded, ready for token exchange |
| `gas-session-created` | GAS → HTML | No (delivers key) | Session created, provides token + messageKey |
| `gas-auth-ok` | GAS → HTML | Yes | Server confirmed session is valid |
| `gas-needs-auth` | GAS → HTML | No (pre-auth) | Session invalid, re-authentication needed |
| `gas-heartbeat-ok` | GAS → HTML | Yes | Heartbeat confirmed session alive |
| `gas-heartbeat-expired` | GAS → HTML | Yes | Session expired or evicted |
| `gas-user-activity` | GAS iframe → HTML | No (informational) | User interacted within GAS iframe |
| `exchange-token` | HTML → GAS | N/A | Delivers OAuth access token for exchange |
| `host-client-ip` | HTML → GAS | N/A | Sends client IP to GAS (if collected) |

### Origin Validation

**Current state (Audit finding H-2):** The HTML-side message listener checks `data.type` against the allowlist but does **not** check `event.origin`.

**Why origin validation is complex here:**
- GAS messages come from `*.googleusercontent.com` — the subdomain is dynamic and unpredictable
- Validating against `script.google.com` would reject all legitimate messages (they come from the inner sandbox, not the outer shell)
- Validating against a regex like `/\.googleusercontent\.com$/` is possible but could be bypassed if an attacker controls a `*.googleusercontent.com` subdomain (unlikely but not impossible)

**Best practice (per [OWASP HTML5 Security Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/HTML5_Security_Cheat_Sheet.html)):** "Check the origin attribute of the sender to verify the data is originating from the expected location." The IETF OAuth 2.0 Security BCP (2025) reinforces: "Both the initiator and receiver of the in-browser message MUST be strictly verified."

**Recommended implementation:** Add origin validation using a regex allowlist:
```javascript
var ALLOWED_ORIGINS = [
  /^https:\/\/[a-z0-9-]+\.googleusercontent\.com$/,
  /^https:\/\/script\.google\.com$/
];
// In message listener:
if (!ALLOWED_ORIGINS.some(function(re) { return re.test(event.origin); })) {
  _reportSecurityEvent('postmessage_unknown_origin', { origin: event.origin });
  return;
}
```

### postMessage Security Best Practices Applied

Per [Microsoft's 2025 postMessage research](https://www.microsoft.com/en-us/msrc/blog/2025/08/postmessaged-and-compromised) and [OWASP guidance](https://cheatsheetseries.owasp.org/cheatsheets/HTML5_Security_Cheat_Sheet.html):

| Best Practice | Status | Notes |
|--------------|--------|-------|
| Specify exact `targetOrigin` | **Partial** — GAS→HTML uses PARENT_ORIGIN; HTML→GAS must use `'*'` (Constraint A) | Architectural constraint — mitigated by single-use token |
| Validate `event.origin` | **Not implemented** — allowlist by message type only | Recommended for implementation |
| Never evaluate messages as code | **Compliant** — no `eval()` or `innerHTML` on message data | |
| Treat all messages as untrusted | **Compliant** — type allowlist + HMAC verification | |
| Use HTTPS | **Compliant** — `upgrade-insecure-requests` in CSP | |
| Sandbox untrusted iframes | **Handled by Google** — GAS iframe is sandboxed by Google's architecture | |

## Cross-Device Single-Session Enforcement

### Mechanism (Plan 9.2 — Heartbeat Piggyback)

When a user signs in on Device B, the system ensures Device A's session is terminated:

```
1. Device B: user signs in → exchangeTokenForSession()
2. Server: invalidateAllSessions(email)
   a. Find all CacheService keys for this email
   b. Delete each session
   c. For each deleted session, write a tombstone:
      Key: 'evicted_' + tokenPrefix
      Value: { reason: 'new_session', timestamp, newDevice: userAgent }
      TTL: 300 seconds (5 minutes)
3. Device A: next heartbeat fires (within 30s test / 5min production)
   a. GAS checks session → not found
   b. GAS checks tombstone → found with reason 'new_session'
   c. Returns gas-heartbeat-expired with reason 'evicted'
4. Device A: shows "Session Active Elsewhere" overlay
   a. Distinct messaging from timeout: "Your account was signed in on another device"
   b. Offers "Sign In Here Instead" button (triggers new sign-in, evicting Device B)
   c. iframe src = 'about:blank' (DOM clearing)
```

### Detection Latency

| Scenario | Detection Time | Acceptable? |
|----------|---------------|-------------|
| Active user, test mode | ≤30 seconds | Yes |
| Active user, production | ≤5 minutes | Yes — session already invalidated server-side |
| Idle user (no activity) | Until next interaction + heartbeat | Yes — no reason to alert idle user |
| Background tab (throttled) | ≤60s test / 5min production | Yes — Chrome throttles setInterval to 1/min after 5min |

### Why Not a Dedicated Polling Loop?

Plans 9, 9.1, and 9.1.1 each proposed dedicated cross-device detection mechanisms (hidden iframe, `google.script.run` polling, Drive file polling). All were rejected because the heartbeat **already does the same job** — it queries the session store on every fire. The tombstone mechanism adds ~60 lines of code vs ~200-300 for a dedicated system, zero additional server calls, and zero new attack surface.

## Cross-Tab Session Coordination

### Single-Tab Enforcement

Only one tab is "active" (showing the app) at a time. When a second tab opens with the same session:

```
1. New tab loads, reads session from storage
2. New tab broadcasts on BroadcastChannel 'auth-sign-out':
   { type: 'tab-claim', token, email, messageKey }
3. Existing active tab receives tab-claim:
   a. Shows "Session Active in Another Tab" overlay
   b. Offers "Use Here" button (re-claims the tab)
4. New tab becomes the active tab
```

**Audit finding H-1:** BroadcastChannel transmits `{token, email, messageKey}` in plaintext. Any same-origin tab (including one with XSS) can intercept all three values. **Recommended fix:** Broadcast events only (e.g., `{type: 'session-changed'}`), not credentials. Each tab reads its own storage independently.

### Cross-Tab Sign-Out (Constraint F)

Two mechanisms, matched to storage type:

**Standard preset (localStorage):**
- `performSignOut()` clears `localStorage`
- Browser fires `storage` event to all other same-origin tabs
- Other tabs detect cleared session key → show auth wall

**HIPAA preset (sessionStorage):**
- `performSignOut()` broadcasts via `BroadcastChannel('auth-sign-out')`
- Other tabs receive broadcast → clear their own `sessionStorage` → show auth wall
- `sessionStorage` changes never fire cross-tab events (by spec), so BroadcastChannel is necessary

## Rate Limiting & Account Lockout

### Current Implementation

**Per-email rate limiting (GAS server-side):**
- Tracks failed authentication attempts per email address
- CacheService key: `rate_limit_EMAIL`
- Limit: 5 failures within 5 minutes
- After limit reached: reject all attempts for that email for 5 minutes
- Audit log: `rate_limit_exceeded`

**Per-session token rate limiting:**
- Tracks requests per session token
- Prevents brute-force session token guessing

### Plan 10 Phase 6 — Escalating Lockout (Not Yet Implemented)

```
Tier 1: 5 failures in 5 min  → 5 min lockout
Tier 2: 10 failures in 1 hr  → 30 min lockout
Tier 3: 20 failures in 24 hr → 24 hr lockout
```

**Architectural constraint (D from Plan 8):** GAS `doGet(e)` does not expose client IP. Rate limiting is per-email (post-authentication) and per-session-token (pre-authentication), not per-IP.

```javascript
var PROJECT_OVERRIDES = {
  ENABLE_DATA_AUDIT_LOG: true,  // Enable audit logging on standard preset
  SERVER_SESSION_DURATION: 600  // Custom 10-minute timeout
};
```

Overrides take precedence over preset values. This allows incremental security adoption — a standard deployment can enable individual hardening features without committing to the full HIPAA preset.

### Preset Transition Rules

**Standard → HIPAA:**
1. Set `ACTIVE_PRESET = 'hipaa'`
2. Configure Script Properties: `HMAC_SECRET`, `ALLOWED_DOMAINS`
3. Redeploy GAS
4. All existing sessions invalidated (HMAC verification fails against sessions created without HMAC enforcement)
5. Users re-authenticate with HIPAA-hardened sessions

**HIPAA → Standard:**
1. Set `ACTIVE_PRESET = 'standard'`
2. Redeploy
3. Hardening features deactivate immediately
4. Existing HIPAA sessions continue to work (HMAC check skipped when disabled)
5. Audit logs retained permanently in Google Sheets

## HIPAA 2026 Compliance Readiness

The HIPAA Security Rule NPRM (published December 2024, expected final May 2026) eliminates "addressable" safeguards — **everything becomes required**. Compliance deadline is ~240 days after finalization (~December 2026 / early 2027).

### Technical Safeguards Assessment (45 CFR § 164.312)

| Requirement | Section | Current State | Gap | Remediation |
|-------------|---------|--------------|-----|-------------|
| **Access Control** | §164.312(a)(1) | ✅ ACL-based per-user per-page. Unique sessions via UUID | Role-based access (RBAC) not implemented | Plan 11, Phase 1 |
| **Unique User ID** | §164.312(a)(2)(i) | ✅ Google OAuth provides unique email identity | None | — |
| **Emergency Access** | §164.312(a)(2)(ii) | ❌ Config flags exist, code not implemented | Full break-glass procedure needed | Plan 10, Phase 5 |
| **Automatic Logoff** | §164.312(a)(2)(iii) | ✅ Rolling timeout (15min HIPAA) + absolute timeout (8hr) + server-side enforcement + activity-gated heartbeat | None | — |
| **Encryption** | §164.312(a)(2)(iv) | ✅ HTTPS enforced via CSP `upgrade-insecure-requests`. Google encrypts Sheets at rest | sessionStorage not encrypted at rest by browser. CacheService encryption status undocumented | Architectural limitation |
| **Audit Controls** | §164.312(b) | ⚠️ Session lifecycle logged. Data-level access not logged | Per-operation audit trail needed for PHI access | Plans 10–11 |
| **Integrity — PHI** | §164.312(c)(1) | ⚠️ HMAC protects session metadata. No data integrity checks | Data-level integrity verification needed | Plan 11, Phase 3 |
| **Authentication** | §164.312(d) | ⚠️ Google OAuth = strong identity. No MFA. No session binding | MFA mandatory under 2026 NPRM. Session binding recommended | Organizational + Plan 10 |
| **Transmission Security** | §164.312(e)(1) | ✅ TLS everywhere (HTTPS + Google internal) | Token-in-URL pattern (H-5, H-6) weakens this | Recommended fix |
| **Integrity Controls** | §164.312(e)(2)(i) | ⚠️ HMAC enabled but silently no-op without secret | Must fail closed | Plan 10, Phase 1 |

## Attack Surface Analysis

### Threat Model

| Attacker Profile | Capability | Primary Targets |
|-----------------|-----------|-----------------|
| **External unauthenticated** | Web browser, network position | OAuth flow, public GAS endpoints, GitHub Pages content |
| **XSS attacker (same-origin)** | JavaScript execution on the page | sessionStorage tokens, BroadcastChannel, postMessage interception, DOM manipulation |
| **Compromised Google account** | Legitimate OAuth token | ACL bypass (if email is in ACL), session hijacking |
| **Insider with GAS edit access** | Full server-side code access | Deploy pipeline manipulation, data exfiltration, HMAC secret access |
| **Network MITM** | Traffic interception | Token-in-URL (H-5/H-6), though TLS prevents actual interception |

### Attack Vectors & Mitigations

| Vector | Risk | Current Mitigation | Remaining Gap |
|--------|------|-------------------|---------------|
| **Forge postMessages** | HIGH → LOW | HMAC-SHA256 signing with server-derived key | Origin validation not implemented (H-2) |
| **Steal session token via XSS** | HIGH | CSP restricts script sources; no inline event handlers | `unsafe-inline` weakens CSP (H-7); sessionStorage readable by any JS (C-5) |
| **Brute-force session tokens** | LOW | UUID v4 tokens (122 bits of entropy); per-token rate limiting | — |
| **Replay gas-session-created** | MEDIUM → LOW | 30-second replay window; first-write-wins on messageKey | — |
| **Cross-device session reuse** | MEDIUM → LOW | Server-side session invalidation on new sign-in; tombstone detection | Detection latency = heartbeat interval |
| **Token leakage via URL** | MEDIUM | Token in URL for heartbeat/signout (H-5/H-6). No referer leakage (meta tag) | Token appears in server logs, browser history, proxy logs |
| **IP lookup to non-BAA third party** | CRITICAL (HIPAA) | None — api.ipify.org receives user IP | Must move server-side or self-host |
| **Clickjacking** | LOW | CSP `frame-src` restrictive; GAS sets X-Frame-Options | Meta tag CSP can't set `frame-ancestors` |
| **OAuth token phishing** | MEDIUM | Google's redirect URI validation | CLIENT_ID is public (M-7) — attacker can create lookalike auth page |

## Testing & Verification Procedures

### Authentication Flow Test Checklist

| Test | Steps | Expected Result |
|------|-------|----------------|
| **Fresh sign-in** | Clear storage → load page → sign in | Auth wall shows → Google popup → successful sign-in → app visible |
| **Session persistence (standard)** | Sign in → close tab → reopen | Session restored from localStorage, no re-auth needed |
| **Session isolation (HIPAA)** | Sign in → open new tab | New tab shows auth wall (sessionStorage is per-tab) |
| **Sign-out** | Click sign-out | Auth wall shows, storage cleared, iframe blanked |
| **Re-sign-in after sign-out** | Sign out → sign in again (no page refresh) | Should work without getting stuck (Fix 3 — _messageKey cleared) |
| **Session timeout (rolling)** | Sign in → wait > SERVER_SESSION_DURATION with no activity | Session expires, auth wall shows |
| **Session timeout (absolute)** | Sign in → continuously interact > ABSOLUTE_SESSION_DURATION | Session expires despite activity |
| **Cross-device eviction** | Sign in on Device A → sign in on Device B | Device A shows "Session Active Elsewhere" within heartbeat interval |
| **Tab duplication** | Sign in → duplicate tab (Ctrl+Shift+D) | Only one tab active; duplicate shows "Use Here" option |
| **ACL denied** | Sign in with email NOT in ACL | Sign-in fails with access denied message |
| **Domain restricted** | HIPAA preset → sign in with email outside ALLOWED_DOMAINS | Sign-in fails with domain restriction message |
| **Rate limiting** | Fail authentication 5+ times rapidly | Lockout message for 5 minutes |
| **HMAC verification** | Inject forged postMessage with invalid signature | Message silently dropped, security event reported |
| **Invalid message type** | Send postMessage with unrecognized type | Message silently dropped, security event reported |

### Security Regression Test Checklist

| Constraint | Test | Verification |
|-----------|------|-------------|
| A | Set targetOrigin to `'https://script.google.com'` | Sign-in breaks silently — proves constraint is still load-bearing |
| B | Use `gasApp.contentWindow` instead of `event.source` | Token exchange fails silently — proves constraint is still load-bearing |
| C | Add auth check to deploy handler | Auto-deploy stops working — GAS freezes at current version |
| D | Skip any step in sign-in chain | Sign-in gets stuck on auth wall |
| E | Change GAS postMessage targetOrigin to `'*'` | Messages still work but security reduced — verify PARENT_ORIGIN is used |
| F | Remove BroadcastChannel sign-out (HIPAA) | Duplicate tabs don't receive sign-out — verify both mechanisms exist |
