### Phase 1: HMAC-SHA256 Message Integrity
- **Steps 1.1–1.5: Completed as documented.** No deviations from the guide.
- Dual-accept migration (Step 1.5) is in place — both HMAC-SHA256 and DJB2 legacy signatures accepted during transition.
- GAS-side HMAC uses a toggle gate (`ENABLE_HMAC_INTEGRITY`) for safe rollout.

### Phase 2: postMessage Origin Validation
- **Step 2.1 — Regex fix required:** The guide's recommended regex `[a-z0-9]+-script\.googleusercontent\.com` is **too restrictive**. Google's GAS sandbox subdomains contain hyphens throughout the subdomain (e.g. `n-abc123def-0lu-script.googleusercontent.com`). The `[a-z0-9]+` character class doesn't allow hyphens, causing all legitimate GAS messages to be silently dropped — sign-in gets stuck on "Signing in..." indefinitely. **Fixed regex:** `/^https:\/\/[a-z0-9][a-z0-9.-]*\.googleusercontent\.com$/` — matches any `*.googleusercontent.com` subdomain (all Google-controlled, so this is safe). This replaced both the strict sandbox regex AND the separate `script.googleusercontent.com` exact match with a single broader pattern.

- **Step 2.2: Completed as documented.** Nonce added to token exchange, GAS side echoes nonce in `gas-session-created` response (both success and failure paths).
- **Step 2.3 — Cannot restrict `host-client-ip` targetOrigin to `script.google.com`:** The guide recommends changing `'*'` to `'https://script.google.com'` for `host-client-ip` messages sent via `gasFrame.contentWindow.postMessage()`. This **does not work** — `contentWindow`'s origin depends on GAS load timing: it may still be at the parent origin (`shadowaisolutions.github.io`) when the IP fetch completes, causing `Failed to execute 'postMessage' on 'DOMWindow'` error.
  - **Solution — deferred forwarding with verified origin:** Instead of sending immediately on IP fetch, the IP is stored in `_clientIp` and forwarded to the GAS iframe on the **first valid message received from GAS** (which proves the iframe has loaded and its origin is known). The targetOrigin uses `event.origin` (the verified GAS sandbox origin) instead of `'*'`. A `_ipForwardedToGas` flag ensures the IP is sent exactly once. This is **more secure** than both the original `'*'` and the guide's recommended `'https://script.google.com'` — it uses the actual verified sandbox origin.
  - **Google warden console log eliminated:** Before this fix, sending `host-client-ip` via `postMessage('*')` before the iframe loaded triggered Google's internal `warden.js` to log `"dropping postMessage.. from host https://shadowaisolutions.github.io but expected host https://n-...googleusercontent.com"`. This exposed the internal GAS sandbox subdomain in the browser console — a HIPAA/security concern. The deferred approach eliminates this log entirely since the message only fires after the iframe is at the correct origin.
  - IP collection is removed entirely in Phase 3 anyway, but the deferred-forwarding pattern is worth noting for any future `postMessage` to the GAS iframe.

## Phase 1: HMAC-SHA256 Message Integrity

**Finding:** C-1 — DJB2 non-cryptographic hash used for message signing
**Severity:** CRITICAL
**HIPAA:** Fails §164.312(e)(1) — integrity controls
**OWASP:** Fails message integrity requirement (HMAC-SHA256 minimum)
**NIST:** Fails FIPS 198-1 — keyed hash message authentication

### Why This Is Phase 1

Every other postMessage security fix depends on message integrity. If an attacker can forge messages (which DJB2 allows — brute-force in ~65K attempts, ~milliseconds), then origin validation (Phase 2) and other controls are bypassed by simply crafting valid-looking messages. **Fix integrity first, then layer on other defenses.**

### The Problem

DJB2 is a **string hash function**, not a **message authentication code**. It was designed for hash table lookups, not cryptographic integrity:

| Property | DJB2 (Current) | HMAC-SHA256 (Target) |
|----------|----------------|----------------------|
| Output size | 32 bits | 256 bits |
| Collision resistance | ~2^16 (65,536 attempts) | ~2^128 (3.4 × 10^38) |
| Pre-image resistance | Trivial | Computationally infeasible |
| Keyed authentication | No (key concatenated, not bound) | Yes (key cryptographically integrated via HMAC construction) |
| NIST approved | No | Yes (FIPS 198-1) |
| HIPAA compliant | No | Yes |
| Length extension attack | Vulnerable | Immune (HMAC double-hash construction prevents this) |

The current implementation concatenates the key to the payload (`JSON.stringify(copy) + '|' + key`) then hashes — this is **not HMAC**. The key is not cryptographically bound to the hash computation. An attacker who observes a few signed messages can brute-force the key in seconds.

### Implementation Steps

#### Step 1.1: Add HMAC-SHA256 Signing to GAS (testauth1.gs)

Replace the inline DJB2 function with server-side HMAC-SHA256 signing. This is the signing side — GAS generates signatures, HTML verifies them.

```javascript
// ── GAS-Side HMAC-SHA256 Signing ──
// Uses Utilities.computeHmacSha256Signature() — native GAS API.
// This is the SAME algorithm as Web Crypto API HMAC-SHA256.
// The outputs are byte-for-byte identical when given the same key and payload.

/**
 * Sign a message object with HMAC-SHA256 using the session's messageKey.
 * @param {Object} msgObj - The message to sign (will have _sig added)
 * @param {string} messageKey - The session's HMAC key
 * @returns {Object} - The message with _sig field containing hex-encoded signature
 */
function signMessage(msgObj, messageKey) {
  if (!messageKey) return msgObj;

  // Create a deterministic payload — sorted keys, exclude _sig
  var copy = {};
  var keys = Object.keys(msgObj).sort();
  for (var i = 0; i < keys.length; i++) {
    if (keys[i] !== '_sig') copy[keys[i]] = msgObj[keys[i]];
  }
  var payload = JSON.stringify(copy);

  // Compute HMAC-SHA256 using GAS native API
  var signature = Utilities.computeHmacSha256Signature(payload, messageKey);

  // Convert signed byte array to hex string
  // GAS returns signed bytes (-128 to 127), must mask to unsigned (0-255)
  var hex = signature.map(function(byte) {
    return ('0' + (byte & 0xff).toString(16)).slice(-2);
  }).join('');

  msgObj._sig = hex;
  return msgObj;
}
```

**Critical detail — signed byte array:** `Utilities.computeHmacSha256Signature()` returns an array of **signed** bytes (Java-style, -128 to 127). You must mask with `& 0xff` to get unsigned values (0-255) before converting to hex. Without this, negative bytes produce wrong hex values and signatures will never verify.

#### Step 1.2: Replace Inline DJB2 in GAS `doGet()` Responses

The current GAS code embeds a minified DJB2 function in HTML output strings. Replace all of these with pre-signed messages:

```javascript
// ── BEFORE (DJB2 — vulnerable) ──
// The GAS side embeds a signing function in the HTML response:
+ 'function s(m){if(!k)return m;var p=JSON.stringify(m)+"|"+k;'
+ 'var h=0;for(var i=0;i<p.length;i++){h=((h<<5)-h)+p.charCodeAt(i);h|=0;}'
+ 'm._sig=h.toString(36);return m;}'

// ── AFTER (HMAC-SHA256 — server-side signing) ──
// Instead of embedding a signing function in the HTML, GAS pre-signs
// the messages server-side and embeds the signed result directly.
// The signing key NEVER leaves the GAS server.

// Example: heartbeat-ok response
var hbMsg = {
  type: 'gas-heartbeat-ok',
  expiresIn: AUTH_CONFIG.SESSION_EXPIRATION,
  absoluteRemaining: hbAbsRemaining
};
var signedMsg = signMessage(hbMsg, msgKey);

// The HTML response just sends the pre-signed message — no client-side signing
var hbOkHtml = '<!DOCTYPE html><html><body><script>'
  + 'window.top.postMessage(' + JSON.stringify(signedMsg) + ', '
  + JSON.stringify(PARENT_ORIGIN) + ');'
  + '</' + 'script></body></html>';
```

**Why this is better:** The HMAC key never appears in client-side code. Even if an attacker reads the entire HTML response, they see the signed message but not the key — they cannot forge new signatures.

#### Step 1.3: Add HMAC-SHA256 Verification to HTML (testauth1.html)

The HTML side imports the key and verifies signatures using the Web Crypto API:

```javascript
// ── Cryptographic Message Verification (Web Crypto API) ──
// The GAS side signs; the HTML side verifies.
// Key is imported as non-extractable — even XSS cannot read it back.

var _hmacKey = null;  // CryptoKey object — non-extractable

/**
 * Import the raw HMAC key string into a CryptoKey for verification.
 * Called once when gas-session-created delivers the messageKey.
 *
 * SECURITY: The key is imported as non-extractable (exportable: false).
 * This means even if an attacker achieves XSS, they cannot call
 * crypto.subtle.exportKey() to read the raw key material.
 * They CAN still use the key to verify messages (by calling verify()),
 * but they CANNOT extract it to sign forged messages.
 */
async function _importHmacKey(rawKey) {
  var encoder = new TextEncoder();
  var keyData = encoder.encode(rawKey);
  return crypto.subtle.importKey(
    'raw',
    keyData,
    { name: 'HMAC', hash: 'SHA-256' },
    false,  // non-extractable
    ['verify']  // only verify — cannot sign
  );
}

/**
 * Verify an HMAC-SHA256 signature on a postMessage payload.
 * @param {Object} data - The message data (with _sig field containing hex signature)
 * @returns {Promise<boolean>} - true if signature is valid
 */
async function _verifyMessageSignature(data) {
  if (!_hmacKey) return true;   // No key yet — pre-session messages are allowed
  var sig = data._sig;
  if (!sig) return false;        // Key exists but no signature — reject

  // Reconstruct the payload identically to how GAS computed it
  var copy = {};
  var keys = Object.keys(data).sort();  // MUST sort — deterministic order
  for (var i = 0; i < keys.length; i++) {
    if (keys[i] !== '_sig') copy[keys[i]] = data[keys[i]];
  }
  var payload = JSON.stringify(copy);

  // Decode hex signature to ArrayBuffer
  var sigBytes = new Uint8Array(sig.match(/.{1,2}/g).map(function(byte) {
    return parseInt(byte, 16);
  }));

  var encoder = new TextEncoder();
  var payloadBytes = encoder.encode(payload);

  return crypto.subtle.verify('HMAC', _hmacKey, sigBytes, payloadBytes);
}
```

#### Step 1.4: Update the postMessage Listener for Async Verification

The `_verifyMessageSignature` function is now async (returns a Promise). The postMessage listener must handle this:

## Phase 2: postMessage Origin Validation

**Finding:** C-2 — postMessage sends token with `'*'` targetOrigin; H-2 — No origin validation on incoming
**Severity:** CRITICAL (C-2), HIGH (H-2)
**OWASP:** "Always specify an exact target origin, not `*`" — HTML5 Security Cheat Sheet
**HIPAA:** Token exposure to unauthorized party violates §164.312(a)(1) access controls

### The Problem

Two separate but related issues:

**C-2 (Outgoing):** The parent page sends the OAuth access token to the GAS iframe with wildcard targetOrigin:
```javascript
event.source.postMessage({ type: 'exchange-token', accessToken: _pendingToken }, '*');
```
This means **any window** (not just the GAS iframe) can receive the token if it's listening.

**H-2 (Incoming):** The parent page accepts messages from **any origin** without checking `event.origin`. Any page that can reference your page (e.g., via iframe, window.open) can send it forged messages.

### The GAS Double-Iframe Constraint (Why `'*'` Exists)

This is not laziness — it's an architectural constraint:

```
┌── Your Page (github.io/org/repo) ────────────────────────────┐
│                                                                │
│  gasFrame.contentWindow ──► script.google.com (outer shell)    │
│                                ▼                               │
│                    [inner sandbox: *.googleusercontent.com]     │
│                    (randomly generated subdomain)               │
│                    THIS is where event.source points to         │
│                                                                │
│  The inner sandbox origin is unpredictable at coding time.     │
│  It changes per deployment and sometimes per session.          │
│  You CANNOT hardcode it as a targetOrigin.                     │
└────────────────────────────────────────────────────────────────┘
```

**What CAN be restricted:**
- Messages **from GAS → parent**: GAS knows the parent origin (`PARENT_ORIGIN`) and already uses it ✅
- Messages **from parent → GAS outer shell** (`contentWindow`): Use `'https://script.google.com'`
- Messages **from parent → GAS inner sandbox** (`event.source`): **Must use `'*'`** — unpredictable origin

**What CANNOT be restricted:**
- The `exchange-token` message replies to `event.source` (the inner sandbox) — `'*'` is required

### Implementation Steps

#### Step 2.1: Add Origin Validation on Incoming Messages (H-2 Fix)

This is the highest-impact fix — it blocks all cross-origin message injection:

```javascript
// ── Origin Validation for Incoming postMessages ──

/**
 * Validate that a postMessage origin is from a legitimate GAS source.
 * GAS uses two origin patterns:
 *   1. https://script.google.com (outer shell)
 *   2. https://<random>-script.googleusercontent.com (inner sandbox)
 *
 * The sandbox subdomain varies per deployment/session.
 * We use a regex that matches Google's known patterns.
 */
function _isValidGasOrigin(origin) {
  if (!origin) return false;

  // Exact match: GAS outer shell
  if (origin === 'https://script.google.com') return true;

  // Pattern match: GAS inner sandbox
  // Format: https://<hash>-script.googleusercontent.com
  // The hash is alphanumeric with hyphens, variable length
  if (/^https:\/\/[a-z0-9]+-script\.googleusercontent\.com$/.test(origin)) return true;

  // Some deployments use this alternate format
  if (/^https:\/\/script\.googleusercontent\.com$/.test(origin)) return true;

  // Google Accounts origin (for GIS callback messages)
  if (origin === 'https://accounts.google.com') return true;

  return false;
}

// ── In the postMessage listener, add as Layer 0 (before all other checks) ──
window.addEventListener('message', function(event) {
  var data = event.data;
  if (!data || typeof data !== 'object') return;

  // Layer 0: Origin validation — reject messages from unexpected origins
  if (!_isValidGasOrigin(event.origin)) {
    // Only log if it looks like an attack (has a type field), not random noise
    if (data.type) {
      _reportSecurityEvent('rejected_origin', {
        origin: event.origin,
        type: String(data.type).substring(0, 50)
      });
    }
    return;  // Silently drop
  }

  // Layer 1: Message type allowlist (existing)
  // Layer 2: HMAC signature verification (Phase 1)
  // Layer 3: Application logic
  // ... existing handler code ...
});
```

#### Step 2.2: Add Nonce to Token Exchange (C-2 Mitigation)

Since `'*'` is unavoidable for the token exchange, add compensating controls:

```javascript
// ── Token Exchange with Nonce + Immediate Clearing ──
// The token exchange is the highest-risk postMessage because:
// 1. It contains the OAuth access token
// 2. It must use '*' (GAS sandbox origin is unpredictable)
// Compensating controls: one-time nonce, immediate token clearing,
// and the token is already short-lived (Google OAuth token ~1 hour)

var _pendingNonce = null;  // One-time nonce for token exchange

function _exchangeTokenWithGas(event) {
  if (!_pendingToken) return;

  // Generate a cryptographic nonce for this exchange
  _pendingNonce = Array.from(crypto.getRandomValues(new Uint8Array(16)))
    .map(function(b) { return b.toString(16).padStart(2, '0'); }).join('');

  event.source.postMessage({
    type: 'exchange-token',
    accessToken: _pendingToken,
    nonce: _pendingNonce,
    timestamp: Date.now()
  }, '*');  // Constraint: unpredictable sandbox origin

  // Immediately clear the pending token — one-shot delivery
  _pendingToken = null;
}

// When GAS responds, verify the nonce matches:
if (data.type === 'gas-session-created' && data.success) {
  if (_pendingNonce && data.nonce !== _pendingNonce) {
    _reportSecurityEvent('nonce_mismatch', {
      expected: _pendingNonce.substring(0, 8) + '...',
      received: (data.nonce || '').substring(0, 8) + '...'
    });
    return;  // Reject — possible replay or injection
  }
  _pendingNonce = null;  // Consumed
  // ... proceed with session setup ...
}
```

#### Step 2.3: Restrict Non-Token Messages to Specific Origins

Messages that don't contain the OAuth token can use specific targetOrigin:


## Phase 4: BroadcastChannel Credential Leak

**Finding:** H-1 — BroadcastChannel transmits session token + messageKey
**Severity:** HIGH
**OWASP:** Never broadcast credentials over shared channels
**Attack:** Any same-origin tab (including one compromised by XSS) intercepts full session hijacking material

### The Problem

The single-tab enforcement mechanism broadcasts credentials to all same-origin tabs:

```javascript
// CURRENT (vulnerable):
function broadcastTabClaim() {
  _signOutChannel.postMessage({
    type: 'tab-claim',
    tabId: _tabId,
    token: session.token || null,      // ← Session hijacking material
    email: session.email || null,       // ← User identity
    messageKey: _messageKey || null     // ← HMAC signing key!
  });
}
```

**Why this is dangerous:** BroadcastChannel delivers messages to **every** same-origin context — every tab, every iframe, every service worker on `your-org.github.io`. If any other page on your GitHub Pages domain has an XSS vulnerability, the attacker gets:
1. The session token (can impersonate the user)
2. The email address (knows who they're impersonating)
3. The messageKey/HMAC key (can forge signed messages)

### Implementation Steps

#### Step 4.1: Broadcast Events, Not Secrets

The tab-claim broadcast only needs to signal "a new tab is claiming the session." Each tab can read its own credentials from sessionStorage — there's no need to send them over the channel:

```javascript
// ── AFTER (secure) ──
function broadcastTabClaim() {
  _signOutChannel.postMessage({
    type: 'tab-claim',
    tabId: _tabId
    // NO token, NO email, NO messageKey
    // The new tab reads credentials from its own sessionStorage
    // (which was populated by the sign-in flow)
  });
}
```

#### Step 4.2: Update the Tab Surrender Handler

The receiving tab (the one being "kicked out") must NOT expect credentials from the broadcast message:

```javascript
// ── Tab surrender handler ──
_signOutChannel.onmessage = function(e) {
  if (e.data && e.data.type === 'tab-claim' && e.data.tabId !== _tabId) {
    _tabSurrendered = true;

    // DON'T try to read credentials from e.data — they're not there
    // The surrendered tab already has its own session in sessionStorage
    // When the user clicks "Use Here", it reclaims using its stored token

    // Show the "session claimed by another tab" wall
    var wall = document.getElementById('tab-takeover-wall');
    if (wall) wall.style.display = 'flex';
  }
};
```

#### Step 4.3: Update "Use Here" Reclaim Handler

When the user clicks "Use Here" on a surrendered tab, it reclaims the session using its own stored credentials:

```javascript
// ── "Use Here" button handler ──
function reclaimSession() {
  // Read credentials from THIS tab's own sessionStorage
  var session = loadSession();
  if (!session.token) {
    // Session was cleared — force re-auth
    showAuthWall();
    return;
  }

  // Broadcast that this tab is reclaiming
  broadcastTabClaim();

  // Hide the takeover wall
  var wall = document.getElementById('tab-takeover-wall');
  if (wall) wall.style.display = 'none';

  _tabSurrendered = false;

  // Resume heartbeat
  startHeartbeat();
}
```

### What About Tab Duplication (M-1)?

When a user duplicates a tab (right-click → Duplicate), the new tab gets a **copy** of sessionStorage. Both tabs have valid credentials. The BroadcastChannel `tab-claim` mechanism catches this:

1. Duplicate tab loads → runs `broadcastTabClaim()` with its new `_tabId`
2. Original tab receives `tab-claim` with a different `_tabId` → surrenders
3. Race window: ~100ms to 1-3 seconds before the broadcast is received

This race window is acceptable because:
- Server-side session validation rejects concurrent requests from the same session
- The surrendered tab cannot make authenticated GAS calls after surrendering
- The window is too short for meaningful exploitation

## Phase 5: messageKey Lifecycle Hardening

**Finding:** H-3 — messageKey nullified in multiple places
**Severity:** HIGH
**Attack:** Each nullification point reopens the first-write-wins window, allowing an attacker to inject a new key

### The Problem

The `_messageKey` (now `_hmacKey` after Phase 1) is set to `null` in at least 4 places:
1. Reauth handler
2. "Use Here" tab reclaim handler
3. Tab-claim broadcast receiver
4. `clearSession()`

Each time the key is nullified, there's a window where an attacker can send a forged `gas-session-created` message with their own `messageKey`, hijacking the integrity mechanism. The "first-write-wins" guard (`if (!_hmacKey)`) becomes useless if the key keeps getting cleared.

### Implementation Steps

#### Step 5.1: Centralize Key Clearing

The HMAC key should ONLY be cleared in `clearSession()` — the function that tears down the entire session:

```javascript
// ── Key Lifecycle: Set Once, Clear Only on Session Teardown ──

// CLEAR: Only in clearSession()
function clearSession() {
  getStorage().removeItem(SESSION_KEY);
  getStorage().removeItem(ABSOLUTE_START_KEY);
  _hmacKey = null;         // ← ONLY place the key is cleared
  _pendingToken = null;
  _pendingNonce = null;
  // ... rest of session cleanup ...
}

// SET: Only in gas-session-created handler, with first-write-wins guard
if (data.type === 'gas-session-created' && data.success && data.messageKey) {
  if (!_hmacKey) {  // First-write-wins — cannot be overwritten
    _importHmacKey(data.messageKey).then(function(key) {
      _hmacKey = key;
    });
  }
}
```

#### Step 5.2: Remove Key Clearing From Other Handlers

```javascript
// ── Reauth handler: DON'T clear the key preemptively ──
// The reauth flow calls clearSession() first (which clears the key),
// then the new session's gas-session-created delivers a new key.
// NO additional key clearing needed.

// BEFORE (vulnerable):
function handleReauth() {
  _messageKey = null;  // ← REMOVE THIS
  clearSession();
  startReauth();
}

// AFTER:
function handleReauth() {
  clearSession();  // This already clears _hmacKey
  startReauth();
}

// ── "Use Here" handler: DON'T clear the key ──
// The reclaiming tab keeps its HMAC key — it needs it to verify
// messages from the GAS iframe for its existing session.

// BEFORE (vulnerable):
function reclaimSession() {
  _messageKey = null;  // ← REMOVE THIS
  // ...
}

// AFTER:
function reclaimSession() {
  // _hmacKey stays — this tab's session is still valid
  // ...
}

// ── Tab-claim receiver: DON'T clear the key ──
// The surrendering tab keeps its key until it either reclaims
// or the user signs out (which calls clearSession()).

// BEFORE (vulnerable):
if (e.data.type === 'tab-claim') {
  _messageKey = null;  // ← REMOVE THIS
  _tabSurrendered = true;
}

// AFTER:
if (e.data.type === 'tab-claim') {
  _tabSurrendered = true;
  // _hmacKey stays — tab might reclaim later
}
```

#### Step 5.3: Defense-in-Depth — Freeze the Key Object

After importing the CryptoKey, make the variable reference non-reassignable from external code:

```javascript
// After successful key import, protect the variable reference
// (Note: this doesn't prevent clearSession() from setting it to null —
//  clearSession() uses the module-scoped reference directly)
var _hmacKeySet = false;

async function _importHmacKey(rawKey) {
  if (_hmacKeySet) return _hmacKey;  // Already set — reject overwrites

  var key = await crypto.subtle.importKey(
    'raw',
    new TextEncoder().encode(rawKey),
    { name: 'HMAC', hash: 'SHA-256' },
    false,
    ['verify']
  );

  _hmacKey = key;
  _hmacKeySet = true;
  return key;
}

// In clearSession():
function clearSession() {
  _hmacKey = null;
  _hmacKeySet = false;  // Allow new key import for next session
  // ...
}
```

## Phase 6: Immutable Session Timers

**Finding:** H-4 — Client-side session expiry timers are authoritative
**Severity:** HIGH
**Attack:** User (or attacker) modifies timer variables via DevTools to prevent session expiry on a shared workstation
**HIPAA:** §164.312(a)(2)(iii) — automatic logoff requirement

### The Problem

Session timeout values are regular JavaScript variables that can be modified in the browser console:

```javascript
// An attacker (or careless user) on a shared workstation can do:
SERVER_SESSION_DURATION = 999999999;
ABSOLUTE_SESSION_DURATION = 999999999;
// Now the session never expires on the client side.
// The next person who sits down sees an authenticated screen.
```

**The good news:** The GAS server independently enforces session expiry. Even with modified client timers, the server will reject heartbeats after the real timeout. **The bad news:** Between heartbeats, the client UI shows an authenticated state — if someone walks up to the machine during that gap, they see the app as "logged in" until the next heartbeat fails.

### Implementation Steps

#### Step 6.1: Make Timer Constants Immutable

Use `Object.defineProperty` to prevent modification:

```javascript
// ── Immutable Session Timer Constants ──
// These values come from the GAS server via the session creation response.
// Once set, they cannot be changed by client-side code.

// Define immutable timer values
// (Call this once when processing gas-session-created)
function _lockSessionTimers(serverDuration, absoluteDuration) {
  // If already defined as non-configurable, skip
  var desc = Object.getOwnPropertyDescriptor(window, '_SERVER_SESSION_DURATION');
  if (desc && !desc.configurable) return;

  Object.defineProperty(window, '_SERVER_SESSION_DURATION', {
    value: serverDuration,
    writable: false,
    configurable: false,
    enumerable: true
  });

  Object.defineProperty(window, '_ABSOLUTE_SESSION_DURATION', {
    value: absoluteDuration,
    writable: false,
    configurable: false,
    enumerable: true
  });
}

// In gas-session-created handler:
if (data.type === 'gas-session-created' && data.success) {
  _lockSessionTimers(
    data.sessionDuration || 900,    // Default: 15 min
    data.absoluteDuration || 57600  // Default: 16 hours
  );
  // ... rest of session setup ...
}
```

#### Step 6.2: Use Closure-Scoped Timers (Belt-and-Suspenders)

`Object.defineProperty` can be circumvented by deleting the property first. For stronger protection, use closure-scoped variables:

```javascript
// ── Closure-Scoped Timer Enforcement ──
// The timer values live inside a closure — not accessible from the global scope.
// Even delete window._SERVER_SESSION_DURATION + redefine won't help.

var _sessionTimerEnforcer = (function() {
  var _serverDuration = null;
  var _absoluteDuration = null;
  var _locked = false;

  return {
    lock: function(serverDur, absoluteDur) {
      if (_locked) return;  // Already locked — ignore attempts to change
      _serverDuration = serverDur;
      _absoluteDuration = absoluteDur;
      _locked = true;
    },
    getServerDuration: function() { return _serverDuration; },
    getAbsoluteDuration: function() { return _absoluteDuration; },
    isLocked: function() { return _locked; },
    reset: function() {
      // Only called during clearSession() — not accessible externally
      // because this function reference is only stored in the closure
      _serverDuration = null;
      _absoluteDuration = null;
      _locked = false;
    }
  };
})();

// Usage:
// _sessionTimerEnforcer.lock(900, 57600);  // Set once
// _sessionTimerEnforcer.getServerDuration();  // Read anytime
// Cannot be modified from console — no global reference to the internals
```

#### Step 6.3: Ensure Server Is the True Authority

The client-side timers are **advisory** — they trigger UI changes (show expiry warning, show auth wall). The GAS server is the **authoritative** session enforcer:

```javascript
// The gas-heartbeat-expired response ALWAYS forces sign-out,
// regardless of what the client timers say.
// This code path should NEVER be conditional on client-side state:

if (data.type === 'gas-heartbeat-expired') {
  // Server says session is expired — this is NOT debatable
  clearSession();
  showAuthWall('Session expired. Please sign in again.');
  // The client-side timer is irrelevant — the server has spoken
}

// Even if an attacker prevents the client-side timer from firing,
// the next heartbeat will hit the server, which will respond with
// gas-heartbeat-expired, forcing the auth wall.
```


### Rollback Plan

Remove the `Object.defineProperty` calls and revert to regular variables. Timer immutability is a defense-in-depth measure — the server-side enforcement is the real protection. Reverting doesn't create an immediate vulnerability but weakens the shared-workstation scenario.

## Phase 7: Token-in-URL Elimination

**Finding:** H-5 — Heartbeat sends token in URL; H-6 — Sign-out sends token in URL; M-4 — Security events sent via URL params
**Severity:** HIGH (H-5, H-6), MEDIUM (M-4)
**OWASP:** "Session identifiers MUST NOT appear in URL parameters" — Session Management Cheat Sheet
**HIPAA:** §164.312(e)(1) — Transmission security; tokens in URLs are data at rest in uncontrolled locations

### The Problem

Three separate functions send sensitive data as URL query parameters:

```javascript
// H-5: Heartbeat — session token in URL
hbFrame.src = baseUrl + '?heartbeat=' + encodeURIComponent(session.token)
  + '&msgKey=' + encodeURIComponent(_messageKey || '');

// H-6: Sign-out — session token in URL
gasApp.src = baseUrl + '?signOut=' + encodeURIComponent(session.token);

// M-4: Security events — attack details in URL
frame.src = baseUrl + '?securityEvent=' + encodeURIComponent(eventType)
  + '&details=' + encodeURIComponent(JSON.stringify(details));
```

**Where these URLs end up:**
1. **Browser history** — `chrome://history` shows the full URL including tokens
2. **GAS server execution logs** — `doGet(e)` parameters are logged by Google
3. **Proxy/CDN logs** — if any intermediate proxy exists
4. **Referer header** — if the iframe navigates to another page
5. **Browser autocomplete/suggestions** — URL bar predictions may include tokens

These are **data at rest** in **uncontrolled locations**. You have no way to clear browser history or GAS server logs.

### The Solution: postMessage-Based Communication

Replace URL parameters with a two-step pattern:
1. Load the GAS iframe with a non-sensitive action parameter (`?action=heartbeat`)
2. Wait for GAS to signal ready, then send sensitive data via `postMessage`

### Architecture Change

```
BEFORE (vulnerable):
  Browser ──URL?token=xxx──► GAS iframe ──► GAS server
  (token in URL at every hop)

AFTER (secure):
  Browser ──URL?action=heartbeat──► GAS iframe
  Browser ◄──postMessage(ready)──── GAS iframe
  Browser ──postMessage(token)───► GAS iframe ──google.script.run──► GAS server
  (token never appears in a URL)
```

### Implementation Steps

#### Step 7.1: Create GAS Listener Pages

The GAS `doGet()` handler needs new action routes that return listener pages instead of processing URL parameters directly:

#### Step 7.2: Update Heartbeat (H-5)

```javascript
// ── In testauth1.html — postMessage-based heartbeat ──

function sendHeartbeat() {
  var session = loadSession();
  if (!session.token) return;

  var baseUrl = (document.getElementById('gas-app') || {}).dataset.baseUrl || '';
  if (!baseUrl) return;

  // Create or reuse heartbeat iframe
  var hbFrame = document.getElementById('gas-heartbeat');
  if (!hbFrame) {
    hbFrame = document.createElement('iframe');
    hbFrame.id = 'gas-heartbeat';
    hbFrame.style.cssText = 'display:none;width:0;height:0;border:0';
    document.body.appendChild(hbFrame);
  }

  // Load GAS with action parameter only — NO token in URL
  hbFrame.src = baseUrl + '?action=heartbeat';

  // One-time listener for the ready signal
  var handled = false;
  var hbHandler = function(event) {
    if (handled) return;
    if (!event.data || event.data.type !== 'gas-heartbeat-ready') return;

    handled = true;
    window.removeEventListener('message', hbHandler);

    // Send token via postMessage — never appears in any URL
    event.source.postMessage({
      type: 'heartbeat-token',
      token: session.token,
      fingerprint: _generateSessionFingerprint ? _generateSessionFingerprint() : null
    }, '*');  // Constraint: unpredictable GAS sandbox origin
  };
  window.addEventListener('message', hbHandler);

  // Timeout: clean up if GAS doesn't respond within 10 seconds
  setTimeout(function() {
    if (!handled) {
      handled = true;
      window.removeEventListener('message', hbHandler);
    }
  }, 10000);
}
```

## Phase 8: CSP Hardening

**Finding:** H-7 — `unsafe-inline` in CSP `script-src`
**Severity:** HIGH
**OWASP:** "Avoid 'unsafe-inline'" — CSP Cheat Sheet; Use hash-based or nonce-based CSP
**Impact:** `unsafe-inline` negates CSP's primary XSS protection — any injected `<script>` tag executes freely

### The Problem

The current CSP allows inline script execution:

```html
<meta http-equiv="Content-Security-Policy" content="
  script-src 'self' 'unsafe-inline' https://accounts.google.com/gsi/client https://apis.google.com;
  ...
">
```

**Why this matters:** CSP is your last line of defense against XSS. If an attacker finds a way to inject HTML (even through a DOM clobbering or template injection attack), `unsafe-inline` means their `<script>alert(document.cookie)</script>` executes without restriction. With a proper CSP, the injected script would be blocked.

### The Honest Constraint: Static Hosting

GitHub Pages is a **static file host**. It cannot set HTTP response headers (only `<meta>` tags in HTML). This means:

| CSP Method | How It Works | Static Site Compatible? | Security |
|-----------|-------------|----------------------|----------|
| **Nonce-based** | Random value per page load, matched in CSP header + script tags | ❌ No — nonce must change per request, requires dynamic server | Strongest |
| **Hash-based** | SHA-256 hash of inline script content baked into CSP | ✅ Yes — hash is static, matches static script content | Strong |
| **`strict-dynamic`** | Trusted root script can load other scripts | ✅ Partially — works with hash as root of trust | Strong |
| **External script file** | Move inline JS to `.js` file, use `'self'` in CSP | ✅ Yes — no inline scripts = no `unsafe-inline` needed | Strongest for static sites |

**Recommended approach:** Hash-based CSP. It works on static sites, provides strong protection, and doesn't require restructuring the entire codebase.

### Implementation Steps

#### Step 8.1: Extract Inline Script Hash

The inline `<script>` block in testauth1.html must have its SHA-256 hash computed:

```bash
# Step 1: Extract the inline script content
# The script is everything between <script> and </script> tags
# (excluding the tags themselves)

# Method: extract inline script content and compute hash
# IMPORTANT: The hash is computed on the EXACT content between the tags,
# including all whitespace, newlines, and formatting.

# Quick method using sed + openssl:
sed -n '/<script>/,/<\/script>/{ /<script>/d; /<\/script>/d; p; }' testauth1.html | \
  openssl dgst -sha256 -binary | openssl base64

# This outputs something like: "5K1oN2eGcMW9R3jHX..."
```

**Critical: the hash covers the EXACT byte-for-byte content.** A single space, newline, or character change invalidates it. This is why Phase 8 must be done **last** — all other phases that modify the inline script must be complete first.

#### Step 8.2: Update the CSP Meta Tag

```html
<!-- BEFORE (vulnerable): -->
<meta http-equiv="Content-Security-Policy" content="
  default-src 'none';
  script-src 'self' 'unsafe-inline' https://accounts.google.com/gsi/client https://apis.google.com;
  style-src 'self' 'unsafe-inline' https://accounts.google.com/gsi/style;
  ...
">

<!-- AFTER (hash-based): -->
<meta http-equiv="Content-Security-Policy" content="
  default-src 'none';
  script-src 'self' 'sha256-YOUR_COMPUTED_HASH_HERE' https://accounts.google.com/gsi/client https://apis.google.com;
  style-src 'self' 'unsafe-inline' https://accounts.google.com/gsi/style;
  ...
">
```

> **Note:** `style-src 'unsafe-inline'` is kept because Google Identity Services (GIS) injects inline styles. You can compute hashes for the known GIS styles, but they change between GIS versions — this would break sign-in when Google updates GIS. The risk from inline styles is much lower than inline scripts (CSS injection has limited exploitation potential compared to JS injection).

#### Step 8.3: Automate Hash Regeneration

Since every change to the inline script invalidates the hash, automate hash computation as part of the commit process:

```bash
#!/bin/bash
# compute-csp-hash.sh — Run before every commit that touches testauth1.html

FILE="live-site-pages/testauth1.html"

# Extract inline script content (between first <script> and </script>)
SCRIPT_CONTENT=$(sed -n '/<script>/,/<\/script>/{ /<script>/d; /<\/script>/d; p; }' "$FILE")

# Compute SHA-256 hash
HASH=$(echo -n "$SCRIPT_CONTENT" | openssl dgst -sha256 -binary | openssl base64)

echo "CSP hash: 'sha256-${HASH}'"
echo ""
echo "Update the CSP meta tag in $FILE with:"
echo "  script-src 'self' 'sha256-${HASH}' https://accounts.google.com/gsi/client https://apis.google.com;"
```

#### Step 8.4: Add `strict-dynamic` for Defense-in-Depth

`strict-dynamic` tells the browser: "any script loaded by a trusted (hashed) script is also trusted." This allows the inline script to dynamically load additional scripts if needed, without adding those scripts' hashes individually:

```html
<!-- With strict-dynamic: -->
<meta http-equiv="Content-Security-Policy" content="
  default-src 'none';
  script-src 'self' 'sha256-YOUR_HASH' 'strict-dynamic' https://accounts.google.com/gsi/client https://apis.google.com;
  ...
">
```

> **Caveat:** With `strict-dynamic`, the `https://accounts.google.com` and `https://apis.google.com` allowlist entries are **ignored** in browsers that support `strict-dynamic` (CSP Level 3). GIS scripts loaded by the hashed root script are trusted via the trust chain. The explicit URLs remain as a fallback for older browsers.

## Phase 9: GAS URL Exposure

**Finding:** M-3 — `window._r` exposes GAS deployment URL globally
**Severity:** MEDIUM
**Attack:** Any script (including injected XSS) can discover and target the GAS deployment endpoint
**Impact:** Combined with stolen credentials, attacker can make authenticated requests to the GAS backend

### The Problem

The GAS deployment URL is stored in `window._r` — a global variable accessible to any JavaScript on the page:

```javascript
// Current code — global variable
window._r = /* decoded and reversed base64 deployment URL */;
// ... used later in auth flow ...
var baseUrl = window._r || gasApp.dataset.baseUrl || '';
delete window._r;  // Deleted after use — but race window exists
```

The `delete` happens after the URL has been read and used. Between the page load and the `delete`, any script can read `window._r`. Even after deletion, the URL persists in `gasApp.dataset.baseUrl`.

### Implementation Steps

#### Step 9.1: Eliminate the Global Variable

Move the deployment URL into a closure-scoped variable that never touches `window`:

```javascript
// ── Closure-Scoped Deployment URL ──
// The URL is decoded inside an IIFE and NEVER assigned to window.
// External scripts cannot access it.

(function() {
  // Decode the deployment URL (same decoding logic as before)
  var _e = '...';  // Base64 + reversed URL
  var _deploymentUrl = '';
  if (_e) {
    try {
      _deploymentUrl = atob(_e).split('').reverse().join('');
    } catch(e) { /* invalid encoding */ }
  }

  // Pass the URL directly to the GAS iframe via dataset
  // (this happens once, immediately, no window exposure)
  var gasApp = document.getElementById('gas-app');
  if (gasApp && _deploymentUrl) {
    gasApp.src = _deploymentUrl;
    gasApp.dataset.baseUrl = _deploymentUrl;
  }

  // _deploymentUrl is now ONLY accessible via:
  // 1. gasApp.dataset.baseUrl (DOM property, not a global JS variable)
  // 2. gasApp.src (DOM property)
  // Both require DOM access — an attacker with DOM access already has XSS,
  // which is a much larger problem than knowing the deployment URL.
})();
```

#### Step 9.2: Update All References to `window._r`

Search the codebase for every `window._r` reference and replace with `gasApp.dataset.baseUrl`:

```javascript
// BEFORE:
var baseUrl = window._r || gasApp.dataset.baseUrl || '';

// AFTER:
var gasApp = document.getElementById('gas-app');
var baseUrl = (gasApp && gasApp.dataset.baseUrl) || '';
// No window._r fallback — it doesn't exist anymore
```

### Residual Risk Assessment

`gasApp.dataset.baseUrl` is still readable by any JS with DOM access. This is a much smaller surface than a global variable:
- **Global variable (`window._r`):** Readable by any script, including scripts in other frames
- **DOM dataset property:** Readable only by scripts that can access the specific DOM element

The deployment URL is inherently discoverable (it's in the iframe `src`). Hiding it completely is not possible in a client-side architecture. The goal is to remove **easy, scriptable access** — not to make it truly secret.

## Troubleshooting

### Phase 1: HMAC-SHA256

**Problem:** All postMessages rejected with `invalid_hmac_signature` after upgrade.

**Root causes and fixes:**

| Cause | Symptom | Fix |
|-------|---------|-----|
| Key sorting mismatch | Signatures never match | Use `Object.keys(obj).sort()` on BOTH sides — exact same order |
| GAS signed byte array | Hex values have negative numbers | Use `(byte & 0xff).toString(16)` — mask to unsigned |
| Unicode in payload | Works in English, fails with accented chars | Stick to ASCII in message payloads; use `TextEncoder` on HTML side |
| JSON.stringify differences | Different key order between GAS and browser | The `.sort()` step should prevent this — verify both sides sort |
| Non-extractable key used for signing | `crypto.subtle.sign()` fails | Import with `['verify']` only — GAS signs, HTML verifies |

**Debug script:**
```javascript
// HTML side — inspect the verification input:
async function debugHmac(data) {
  var copy = {};
  var keys = Object.keys(data).sort();
  for (var i = 0; i < keys.length; i++) {
    if (keys[i] !== '_sig') copy[keys[i]] = data[keys[i]];
  }
  console.log('Payload to verify:', JSON.stringify(copy));
  console.log('Signature received:', data._sig);
  console.log('Signature length:', data._sig ? data._sig.length : 'N/A');
  // HMAC-SHA256 hex = 64 chars. DJB2 base36 = 1-7 chars.
  // If sig length < 10, you're still receiving DJB2 signatures (GAS not updated).
}
```

| Timeout too short | Intermittent failures | Increase the 10-second timeout if GAS is slow |

**Problem:** Heartbeat works in testing but fails intermittently in production.

**Root cause:** The GAS iframe may be rate-limited by Google (especially in the free tier). If too many iframes load in quick succession, some may be throttled.

**Fix:** Ensure heartbeat interval is not too aggressive. The production value should be 5 minutes minimum, not the test value of 30 seconds.

### Phase 8: CSP Hash

**Problem:** Inline script blocked after every code change.

**Solution:** Run the hash computation script after every edit:
```bash
# Quick hash check — does the current CSP hash match the script content?
CURRENT_HASH=$(grep -oP "sha256-[A-Za-z0-9+/=]+" live-site-pages/testauth1.html | head -1)
COMPUTED_HASH="sha256-$(sed -n '/<script>/,/<\/script>/{ /<script>/d; /<\/script>/d; p; }' \
  live-site-pages/testauth1.html | openssl dgst -sha256 -binary | openssl base64)"

if [ "$CURRENT_HASH" = "$COMPUTED_HASH" ]; then
  echo "✅ CSP hash matches script content"
else
  echo "❌ CSP hash MISMATCH"
  echo "  Current: $CURRENT_HASH"
  echo "  Computed: $COMPUTED_HASH"
  echo "  Update the CSP meta tag with: $COMPUTED_HASH"
fi
```

### General: "Everything broke after Phase X"

**Recovery procedure:**
1. Identify which phase caused the break (revert that phase's changes)
2. Re-run that phase's individual verification checklist
3. Check the Known Interactions table (Phase 10) for dependencies
4. If the break is in Phase 1 (HMAC), verify both GAS and HTML sides are deployed
5. If the break is in Phase 8 (CSP), the hash needs recomputation — this is the most common issue
