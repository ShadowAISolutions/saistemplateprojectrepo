## Why This Plan Exists

Plans 9, 9.1, and 9.1.1 all introduce **new polling mechanisms** to detect cross-device session invalidation — a hidden iframe, a `google.script.run` call, or a Drive file check. Each adds complexity (new functions, new timers, new message types, new state variables) and quota/cost concerns.

But the detection mechanism **already exists**. The heartbeat system already does exactly what cross-device enforcement needs:

1. **The heartbeat already calls `doGet(?heartbeat=TOKEN)`** — which reads the session from CacheService
2. **`exchangeTokenForSession()` already calls `invalidateAllSessions()`** — which deletes old sessions from CacheService when a new device signs in
3. **The heartbeat already returns `gas-heartbeat-expired` when the session doesn't exist** — the old device already receives a "your session is gone" signal

The cross-device eviction detection is already happening. The only thing missing is:
- **Server-side:** the heartbeat response doesn't distinguish "session timed out" from "session was evicted by another device"
- **Client-side:** the `gas-heartbeat-expired` handler treats all expirations identically — it shows "Session expired. Please sign in again." regardless of cause

### The Insight

Instead of building a new polling system, **enhance the existing heartbeat with a reason code**. When `invalidateAllSessions()` deletes old sessions, it leaves a short-lived "tombstone" in CacheService that records the eviction. When the heartbeat finds no session but finds a tombstone, it tells the client *why* the session disappeared. The client then shows the appropriate UI — "Session Active Elsewhere" for cross-device eviction vs. "Session expired" for natural timeout.

### Accepted Tradeoffs

The user explicitly accepted these tradeoffs:

1. **Detection speed matches heartbeat interval** — 30s in test mode, 5 minutes in production. This is slower than the 10-second detection of plans 9 and 9.1. For most use cases, a 5-minute window is acceptable — the user must sign in on Device B, and Device A detects within 5 minutes. The session was already invalidated server-side the instant Device B signed in, so Device A can't do anything harmful during the detection window — their next server interaction (heartbeat, page load, API call) will fail.

2. **Activity-gated** — the heartbeat only fires when the user is active. If Device A's user is completely idle when Device B signs in, Device A won't detect the eviction until the user interacts with Device A again (triggering a heartbeat). This is actually *desirable* — why show an eviction overlay to a user who isn't looking at the screen? When they return and interact, the overlay appears immediately (within one heartbeat interval).

3. **Background tab throttling** — Chrome throttles `setInterval` to 1/minute after 5 minutes of background inactivity (per [Chrome's timer throttling](https://developer.chrome.com/blog/timer-throttling-in-chrome-88)). This extends the worst-case detection to ~60s in test mode (already below heartbeat production interval) or 5 minutes in production (unchanged — the heartbeat interval is already 5 minutes). This is identical to all other plans since they all use `setInterval`.

## Implementation Phases

### Phase 1: GAS — Eviction Tombstone in `invalidateAllSessions()`

**Risk:** Low
**Files:** `testauthgas1.gs`
**What:** When `invalidateAllSessions()` deletes sessions, leave a short-lived "tombstone" cache entry for each deleted token. The tombstone tells the heartbeat handler *why* the session is gone.

**Current code** (`testauthgas1.gs` lines 560–577):
```javascript
function invalidateAllSessions(email) {
  if (!email) return;
  var cache = CacheService.getScriptCache();
  var trackKey = "sessions_" + email.toLowerCase();
  var raw = cache.get(trackKey);
  if (!raw) return;
  try {
    var tokens = JSON.parse(raw);
    for (var i = 0; i < tokens.length; i++) {
      cache.remove("session_" + tokens[i]);
    }
    if (tokens.length > 0) {
      auditLog('session_management', email, 'all_sessions_invalidated',
        { count: tokens.length });
    }
  } catch (e) {}
  cache.remove(trackKey);
}
```

**Modified code:**
```javascript
function invalidateAllSessions(email) {
  if (!email) return;
  var cache = CacheService.getScriptCache();
  var trackKey = "sessions_" + email.toLowerCase();
  var raw = cache.get(trackKey);
  if (!raw) return;
  try {
    var tokens = JSON.parse(raw);
    for (var i = 0; i < tokens.length; i++) {
      cache.remove("session_" + tokens[i]);
      // Leave a tombstone so the heartbeat handler knows WHY the session
      // disappeared. Short TTL (5 minutes) — just needs to survive until
      // the old device's next heartbeat fires. After that, natural expiry
      // is assumed (no tombstone = timed out normally).
      cache.put("evicted_" + tokens[i], "new_sign_in", 300);
    }
    if (tokens.length > 0) {
      auditLog('session_management', email, 'all_sessions_invalidated',
        { count: tokens.length });
    }
  } catch (e) {}
  cache.remove(trackKey);
}
```

**Design decisions:**

1. **Tombstone key format: `evicted_TOKEN`** — mirrors the `session_TOKEN` key format. The heartbeat handler already has the token, so looking up `evicted_TOKEN` is a simple string concatenation.

2. **Tombstone value: plain string `"new_sign_in"`** — kept minimal. No JSON parsing needed. The heartbeat handler just needs to know the reason. Future reasons could include `"admin_revoked"` or `"security_lockout"` without changing the mechanism.

3. **TTL: 300 seconds (5 minutes)** — matches the production heartbeat interval. This guarantees the tombstone survives until the old device's next heartbeat in production. In test mode (30s heartbeat), the tombstone outlives ~10 heartbeat cycles, which is more than enough. After 5 minutes with no heartbeat claim, the tombstone expires naturally — any subsequent discovery of a missing session defaults to "natural timeout" (the correct assumption after 5+ minutes).

4. **One additional `cache.put()` per eviction** — the only new CacheService operation. This runs during sign-in (rare event), not during polling. Zero ongoing cost.

5. **No tombstone on explicit sign-out** — when a user clicks "Sign Out", `invalidateSession()` runs (not `invalidateAllSessions()`). No tombstone is needed because the sign-out is communicated through `gas-signed-out` postMessage, which the client already handles. Tombstones are only for the case where the old device doesn't know its session was killed.

6. **No tombstone on natural expiry** — when CacheService TTL expires and the session disappears naturally, there's no tombstone (nothing triggered the deletion). The heartbeat handler's fallback — no session AND no tombstone → "natural timeout" — handles this correctly.

### Phase 2: GAS — Heartbeat Handler Enhancement

**Risk:** Low
**Files:** `testauthgas1.gs`
**What:** When the heartbeat finds no session, check for an eviction tombstone. Include the reason in the `gas-heartbeat-expired` postMessage.

**Current heartbeat expired response** (`testauthgas1.gs` lines 768–775):
```javascript
var raw = cache.get("session_" + heartbeatToken);
if (!raw) {
  var hbExpiredHtml = '<!DOCTYPE html><html><body><script>'
    + 'window.top.postMessage({type:"gas-heartbeat-expired"}, '
    + JSON.stringify(PARENT_ORIGIN) + ');'
    + '</' + 'script></body></html>';
  return HtmlService.createHtmlOutput(hbExpiredHtml)
    .setTitle(TITLE)
    .setXFrameOptionsMode(HtmlService.XFrameOptionsMode.ALLOWALL);
}
```

**Modified code:**
```javascript
var raw = cache.get("session_" + heartbeatToken);
if (!raw) {
  // Check for eviction tombstone — tells us WHY the session disappeared
  var evictionReason = cache.get("evicted_" + heartbeatToken) || 'timeout';
  // Clean up the tombstone after reading (one-time use)
  if (evictionReason !== 'timeout') {
    cache.remove("evicted_" + heartbeatToken);
  }
  var hbExpiredHtml = '<!DOCTYPE html><html><body><script>'
    + 'var k=' + JSON.stringify(msgKey) + ';'
    + 'function s(m){if(!k)return m;var p=JSON.stringify(m)+"|"+k;var h=0;'
    + 'for(var i=0;i<p.length;i++){h=((h<<5)-h)+p.charCodeAt(i);h|=0;}'
    + 'm._sig=h.toString(36);return m;}'
    + 'window.top.postMessage(s({type:"gas-heartbeat-expired"'
    + ',reason:' + JSON.stringify(evictionReason) + '}), '
    + JSON.stringify(PARENT_ORIGIN) + ');'
    + '</' + 'script></body></html>';
  return HtmlService.createHtmlOutput(hbExpiredHtml)
    .setTitle(TITLE)
    .setXFrameOptionsMode(HtmlService.XFrameOptionsMode.ALLOWALL);
}
```

**Design decisions:**

1. **HMAC signing the expired message** — the current `gas-heartbeat-expired` response does NOT include the signing function. This is a security gap — an attacker could inject a fake `gas-heartbeat-expired` postMessage to force a sign-out. The modified version includes the HMAC signing function `s()` (same pattern used by `gas-heartbeat-ok` at line 835). The `msgKey` is already available in the heartbeat handler scope (extracted from URL parameter at line 747). This is a security improvement regardless of cross-device enforcement.

2. **Tombstone cleanup after read** — `cache.remove("evicted_" + heartbeatToken)` ensures the tombstone is consumed once. If the heartbeat fires again (e.g., due to a timing race or retry), subsequent reads return `null`, and the reason defaults to `'timeout'`. This prevents stale tombstones from confusing later sessions.

3. **Default reason `'timeout'`** — when no tombstone exists, the reason defaults to `'timeout'`. This covers: (a) natural session expiry (CacheService TTL elapsed), (b) tombstone already expired (>5 minutes since eviction), (c) tombstone already consumed by a previous heartbeat. All these cases should show the standard "Session expired" message.

4. **Reason as a string, not a code** — using readable strings (`"new_sign_in"`, `"timeout"`) instead of numeric codes. Easier to debug, easier to extend, and the heartbeat response is already HTML-in-a-string so there's no size concern.

5. **All other heartbeat-expired paths** — the heartbeat handler has multiple paths that return `gas-heartbeat-expired`: HMAC failure (line 789), absolute timeout (line 803), rolling timeout (line 818), and parse error (line 780). These should also include a `reason` field for consistency. Suggested reasons:
   - Parse error → `reason: "corrupt_session"`
   - HMAC failure → `reason: "integrity_violation"`
   - Absolute timeout → `reason: "absolute_timeout"`
   - Rolling timeout → `reason: "timeout"`
   - Rate limited → already returns `gas-heartbeat-error` (different message type)

   **Implementation note:** adding the reason + HMAC signing to all expired paths is a mechanical find-and-replace. Each currently returns a bare `gas-heartbeat-expired` without signing — all should be updated to use the `s()` signing function and include a `reason` field. This is a net security improvement.

```javascript
if (data.type === 'gas-heartbeat-expired') {
  _heartbeatInFlight = false;
  stopHeartbeat();
  var hbEl2 = document.getElementById('timer-heartbeat');
  if (hbEl2) {
```

```javascript
          + 'Only one session can be active at a time.';

      if (useHereBtn) {
        useHereBtn.textContent = 'Sign In Here';
      }
      takeoverWall.style.display = 'flex';
    }
    return;  // Don't show auth wall — show takeover overlay instead
  }

  // Normal expiration (timeout, absolute timeout, corrupt session, etc.)
  clearSession();

```

**Design decisions:**

1. **Reuses the existing `tab-takeover-wall` overlay** — the same overlay used by BroadcastChannel single-tab enforcement. The text and button label are changed to reflect cross-device eviction. This is consistent with plans 9.1 and 9.1.1 — the visual experience is identical.

2. **`_evictedByRemote` flag** — a new boolean state variable that distinguishes cross-device eviction from same-browser tab claim. The "Use Here" button handler checks this flag to determine the correct reclaim behavior (see Phase 5).

3. **`_tabSurrendered = true`** — reuses the existing tab-surrender mechanism. This prevents the heartbeat from firing again (the heartbeat's activity listener checks `_tabSurrendered`).

4. **`clearSession()` before overlay** — the session is already invalid on the server. Clearing it locally prevents stale session data from being used in any subsequent flow.

5. **`return` after showing overlay** — prevents falling through to the `showAuthWall()` path. The eviction overlay is a different UX flow — it shows a "Sign In Here" button that triggers a fresh sign-in, not the standard "sign in again" auth wall.

6. **No `sessionStorage.setItem(SESSION_EXPIRED_FLAG, '1')`** — the expired flag is used to show "Session expired" on the auth wall after page reload. For cross-device eviction, we show the takeover overlay instead, so the flag is not set.

**Modified handler:**
```javascript
document.getElementById('tab-use-here-btn').addEventListener('click', function() {
  _tabSurrendered = false;
  document.getElementById('tab-takeover-wall').style.display = 'none';

  // Cross-device eviction: session was invalidated by another device.
  // Cannot reclaim — must sign in fresh.
  if (_evictedByRemote) {
    _evictedByRemote = false;
    _messageKey = null;  // Clear HMAC key — old session's key is invalid
    clearSession();
    showAuthWall();
    return;
  }

  // Same-browser tab claim: session still exists, just reclaim it.
  if (!HTML_CONFIG.SINGLE_TAB_ENFORCEMENT) return;
  var session = loadSession();
  if (session.token) {
    showApp(session.email);
    startCountdownTimers();
    startHeartbeat();
    var gasApp = document.getElementById('gas-app');
    var baseUrl = window._r || (gasApp && gasApp.dataset.baseUrl) || '';
    if (gasApp && baseUrl) {
      _expectingSession = true;
      gasApp.src = baseUrl + '?session=' + encodeURIComponent(session.token);
    }
  } else {
    showAuthWall();
  }
  broadcastTabClaim();
});
```

**Design decisions:**

1. **Cross-device: show auth wall, not reclaim** — when evicted by another device, the session is gone on the server. There's no session to reclaim. The user must sign in fresh, which will in turn evict the other device.

2. **`_messageKey = null`** — the HMAC message key was bound to the old session. It's invalid now. Clearing it ensures the next `gas-session-created` message can bootstrap a new key (first-write-wins logic at line 1143).

3. **Removed the `if (!HTML_CONFIG.SINGLE_TAB_ENFORCEMENT) return;` guard from the top** — the "Use Here" button can now be triggered by two separate flows: same-browser tab claim (requires `SINGLE_TAB_ENFORCEMENT`) and cross-device eviction (independent of `SINGLE_TAB_ENFORCEMENT`). The guard now only protects the same-browser reclaim path.

## CacheService Consistency

**Question:** When `invalidateAllSessions()` deletes a session via `cache.remove()` and writes a tombstone via `cache.put()`, is there a risk that the old device's heartbeat reads the session key between the two operations (seeing the session as "still valid" before the tombstone is written)?

**Answer: No — this is not a meaningful risk.** Here's why:

1. **Sequential execution** — `invalidateAllSessions()` runs within a single GAS execution context. `cache.remove()` and `cache.put()` execute sequentially within that context. There is no async gap between them.

2. **Server-side concurrency model** — GAS uses a single-threaded execution model per invocation. The heartbeat from Device A is a separate `doGet()` invocation running on a different server instance. CacheService uses Memcache (Google's distributed cache) under the hood, which provides eventual consistency but with extremely low propagation delay (typically <10ms within a datacenter).

3. **Race window is harmless** — even in the unlikely scenario where the heartbeat reads the session key in the microseconds between `cache.remove()` and `cache.put()`:
   - The heartbeat sees the session as valid → extends it → returns `gas-heartbeat-ok`
   - On the *next* heartbeat (30s/5min later), the session is definitely gone (the new session's `invalidateAllSessions` already ran, and CacheService has long since propagated the delete)
   - The tombstone is still there (5-minute TTL)
   - The old device detects eviction one heartbeat interval later — at most 30s/5min slower than the ideal case

4. **Even without the tombstone, detection works** — if the tombstone somehow doesn't get written (cache.put fails, quota exceeded, etc.), the heartbeat still detects the missing session on the next tick and returns `gas-heartbeat-expired` with `reason: "timeout"`. The user sees "Session expired" instead of "Session Active Elsewhere" — a slightly less informative message, but the session is still correctly terminated. The tombstone is an enhancement for UX messaging, not a prerequisite for security.

**Sources:**
- [Google Apps Script CacheService docs](https://developers.google.com/apps-script/reference/cache/cache-service) — single-threaded execution model
- [Memcache consistency model](https://cloud.google.com/appengine/docs/standard/python/memcache/) — eventual consistency with sub-10ms typical propagation
- [CacheService Eviction and Limits](https://dev.to/googleworkspace/apps-script-cacheservice-eviction-and-other-limits-1p6d) — 1,000-item limit, FIFO eviction of ~10% when exceeded

### Alternative Considered: Overwrite Instead of Delete + Tombstone

The Spring Security pattern for concurrent session control uses **overwrite with eviction flag** instead of **delete + tombstone**. In that model, `invalidateAllSessions()` would not delete the old session but instead overwrite its content to `{"status": "evicted", "evictedAt": Date.now()}`. The heartbeat would read the session, find the `"evicted"` status, and know why it was evicted — no separate tombstone needed.

**Why we chose delete + tombstone instead:**
1. **Minimal diff** — the current `invalidateAllSessions()` already uses `cache.remove()`. Adding a single `cache.put()` for the tombstone is less invasive than changing the deletion logic to an overwrite. The tombstone is additive, not a refactor.
2. **No TTL confusion** — an overwritten session would inherit the same CacheService TTL slot. If the original session had 2 minutes of TTL remaining, the eviction flag would expire in 2 minutes, potentially before the old device's heartbeat fires (5-minute production interval). A separate tombstone with its own 300-second TTL avoids this — it's guaranteed to survive one full production heartbeat cycle.
3. **Cleaner separation of concerns** — the session key (`session_TOKEN`) stores session data. The tombstone key (`evicted_TOKEN`) stores eviction metadata. They have different TTLs, different purposes, and different consumers. Mixing them into one key adds complexity to every function that reads session data (validateSession, heartbeat, etc.) — they'd all need to check for the `"evicted"` status.

The overwrite approach is equally valid and may be worth revisiting if CacheService `remove()` proves unreliable in practice. The tombstone approach was chosen for implementation simplicity.

## Edge Cases and Behavior

### Edge Case 1: User idle on Device A when Device B signs in

**Scenario:** User has Device A open but idle (no mouse/keyboard activity). User signs in on Device B.

**Behavior:** `invalidateAllSessions()` runs, deletes Device A's session, writes tombstone. Device A's heartbeat does NOT fire (activity-gated — no activity since last tick). The tombstone sits in CacheService for up to 5 minutes (its TTL). When the user returns to Device A and moves their mouse/presses a key:
1. `_onHeartbeatActivity()` fires → sets `_lastHeartbeatActivity = Date.now()`
2. Next heartbeat tick (within one interval) → activity detected → `sendHeartbeat()` fires
3. Heartbeat finds no session → finds tombstone → returns `gas-heartbeat-expired` with `reason: "new_sign_in"`
4. Client shows "Session Active Elsewhere" overlay

**If the user returns AFTER the tombstone expires (>5 minutes idle):**
1. Heartbeat fires → no session → no tombstone → returns `reason: "timeout"`
2. Client shows "Session expired. Please sign in again."
3. This is acceptable — after 5+ minutes of inactivity, the session would have expired naturally anyway (production `SESSION_EXPIRATION` is 1 hour, but the CacheService TTL is set to `SESSION_EXPIRATION` in seconds, and the heartbeat didn't fire to extend it). The message is slightly inaccurate ("expired" vs "evicted") but the outcome is identical — user must sign in.

### Edge Case 2: Device A in background tab

**Scenario:** User has Device A in a background tab (minimized or behind another window). Device B signs in.

**Behavior:** Chrome throttles `setInterval` to 1/minute after 5 minutes of being backgrounded ([Chrome timer throttling](https://developer.chrome.com/blog/timer-throttling-in-chrome-88)). Additionally, the heartbeat is activity-gated — if there's no user activity in the background tab, the heartbeat skips sending.

**Worst case for test mode (30s interval):**
- Background throttling extends interval to ~60s
- Activity gating may skip additional ticks
- Detection happens when the tab is foregrounded (triggers activity) or when a throttled tick runs and the last activity was recent
- Tombstone (300s TTL) outlives the throttled interval comfortably

**Worst case for production mode (5min interval):**
- Background throttling has no effect (interval already > 1 minute)
- Activity gating is the dominant factor
- Detection: same as Edge Case 1 — when the user returns to the tab

### Edge Case 3: Both devices sign in simultaneously

**Scenario:** Two devices sign in at the exact same time for the same user.

**Behavior:** `exchangeTokenForSession()` runs on both. Both call `invalidateAllSessions()`. Due to CacheService's eventual consistency:
- Each invalidates whatever sessions exist at their read time
- Each creates a new session
- Each writes a tombstone for whatever it invalidated
- The last session to complete `cache.put("session_" + ...)` wins
- The earlier session's token becomes invalid (removed by the later `invalidateAllSessions`)

**Result:** One device ends up with a valid session, the other with an invalid one. The invalid device's next heartbeat detects the problem. This is the correct behavior — `MAX_SESSIONS_PER_USER: 1` means last-write-wins.

### Edge Case 4: Heartbeat in flight when eviction happens

**Scenario:** Device A's heartbeat HTTP request is already in transit when Device B's `invalidateAllSessions()` runs.

**Behavior:** Two sub-cases:
1. **Heartbeat arrives at server BEFORE `invalidateAllSessions` runs:** Session is found → heartbeat extends it → returns `gas-heartbeat-ok`. Then `invalidateAllSessions` runs and deletes the session + writes tombstone. Device A's *next* heartbeat detects the eviction. Detection delayed by one heartbeat interval.

2. **Heartbeat arrives at server AFTER `invalidateAllSessions` runs:** Session is not found → tombstone is found → returns `gas-heartbeat-expired` with `reason: "new_sign_in"`. Device A shows eviction overlay immediately.

Both are correct. Sub-case 1 adds at most one interval of delay, which is within the accepted detection window.

### Edge Case 5: Multiple rapid sign-ins (Device B, C, D...)

**Scenario:** Devices B, C, and D sign in rapidly in sequence for the same user.

**Behavior:** Each sign-in calls `invalidateAllSessions()`, which tombstones whatever sessions exist. Each overwrites the previous tombstone for the same token (same cache key). The last tombstone value is still `"new_sign_in"`. Device A's heartbeat finds the tombstone and shows the eviction overlay. Only the last device to sign in has a valid session.

**Result:** Correct. The tombstone is per-token, and only Device A's old token matters. The reason is always `"new_sign_in"` regardless of how many subsequent sign-ins happened.

### Edge Case 6: User clicks "Sign In Here" on Device A

**Scenario:** Device A shows "Session Active Elsewhere" overlay. User clicks "Sign In Here".

**Behavior:**
1. `_evictedByRemote = false` — clear the eviction flag
2. `_messageKey = null` — clear the old session's HMAC key
3. `clearSession()` — remove any local session data
4. `showAuthWall()` — show the sign-in UI
5. User signs in → `exchangeTokenForSession()` → `invalidateAllSessions(email)` → Device B's session is invalidated with a tombstone
6. Device B's next heartbeat detects eviction → shows overlay on Device B

**Result:** The session ping-pongs between devices. This is the expected behavior for `MAX_SESSIONS_PER_USER: 1` — the last sign-in wins. The overlay message and "Sign In Here" button make the experience clear.

## What This Plan Does NOT Address

1. **Real-time eviction notification** — detection is bounded by the heartbeat interval (30s test / 5min production). Plans 9 and 9.1 offer ~10s detection but at higher cost. If faster detection is needed in the future, Plan 9.1 can be layered on top of this plan (the tombstone mechanism and reason codes are compatible).

2. **Multiple concurrent sessions** — this plan enforces `MAX_SESSIONS_PER_USER: 1`. Supporting multiple concurrent sessions (e.g., 2 devices allowed but not 3) would require a different mechanism.

3. **Session transfer/handoff** — there is no mechanism to transfer a session from one device to another without re-authenticating. The user must sign in fresh on the new device.

4. **Active notification to Device B** — when Device A reclaims by clicking "Sign In Here" and signing in, Device B learns about the eviction passively via heartbeat. There's no push notification.

5. **Idle session vs. active eviction distinction after tombstone expiry** — if the tombstone expires (>5min), the client can't distinguish between natural timeout and cross-device eviction. This is acceptable — after 5 minutes of complete inactivity, "session expired" is a reasonable message regardless of cause.

6. **Heartbeat not enabled** — if `ENABLE_HEARTBEAT: false`, this plan does not work. Cross-device enforcement depends on the heartbeat mechanism. The `ENABLE_CROSS_DEVICE_ENFORCEMENT` toggle should be documented as requiring `ENABLE_HEARTBEAT: true`.
