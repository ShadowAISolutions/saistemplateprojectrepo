# DJB2 → HMAC-SHA256 Migration Plan — GAS Session HTML Messages

**Created:** 2026-03-18
**Based on:** [10.2-CATEGORY3-CODE-IMPLEMENTATION-GUIDE.md](10.2-CATEGORY3-CODE-IMPLEMENTATION-GUIDE.md) — Phase 1 residual
**Target:** `testauthgas1.gs` (GAS-served session HTML inline script) + `testauthgas1.html` (host page verification)
**Prerequisite:** Must be completed BEFORE Phase 8 (CSP Hardening) — changes to inline `<script>` invalidate CSP hash

> **Purpose:** The Phase 1 HMAC-SHA256 implementation replaced DJB2 signing for all server-side GAS responses
> (`signMessage()` in `.gs`). However, four message types are signed **client-side** inside the GAS-served
> session HTML using `_s()` — a DJB2 hash function embedded in the inline `<script>`. These messages pass
> through the host page's dual-accept verification (`_verifyMessageSignature()`), which currently falls back
> to DJB2 for non-HMAC signatures. This plan migrates those four messages to proper HMAC-SHA256 signing.

> **Implementation workflow:** Implement one step at a time. After each step, describe what to test and
> **wait** for the developer to confirm results before proceeding. Do not advance unprompted.

## Problem Statement

### What's happening now

The GAS-served session HTML (the app page shown after authentication) contains an inline `_s()` function that signs outgoing `postMessage` calls using DJB2:

```javascript
// Current: DJB2 signing in GAS session HTML (testauthgas1.gs:~1530)
var _mk = '${escapeJs(appMsgKey)}';
function _s(m) {
  if (!_mk) return m;
  var p = JSON.stringify(m) + '|' + _mk;
  var h = 0;
  for (var i = 0; i < p.length; i++) { h = ((h << 5) - h) + p.charCodeAt(i); h |= 0; }
  m._sig = h.toString(36);
  return m;
}
```

Four messages use `_s()`:

| Message Type | When Sent | Trigger |
|-------------|-----------|---------|
| `gas-auth-ok` | Immediately on session HTML load | Notifies host that GAS auth succeeded |
| `gas-version` | On `gas-version-check` request | Responds with current GAS VERSION |
| `gas-user-activity` | On keydown/click/input in GAS iframe | Debounced activity detection (5s) |
| `gas-session-invalid` | On `saveNote()` server error | Session validation failed during data op |

### Why it matters

- **DJB2 is not cryptographic** — 32-bit hash, trivially reversible, collision-prone. An attacker who obtains the `messageKey` (or guesses a collision) can forge any of these four message types
- **Inconsistency** — server-side GAS functions (`processHeartbeat`, `processSignOut`, etc.) already use HMAC-SHA256 via `signMessage()`. Only the session HTML's client-side messages use the weaker DJB2
- **Phase 8 dependency** — CSP hardening (hash-based `script-src`) locks the inline script content. Any changes to `_s()` or its replacement must happen BEFORE Phase 8 computes the CSP hash

### Why not just use Web Crypto in the GAS iframe?

The GAS-served session HTML runs in Google's sandboxed iframe (`*.googleusercontent.com`). While Web Crypto API is available, using it would mean:
- The signing key (`messageKey`) is available in JavaScript — same exposure as DJB2
- The client-side code would need to reimplement HMAC-SHA256 in a way that produces byte-identical output to the GAS server's `Utilities.computeHmacSha256Signature()`
- Any discrepancy in JSON serialization, key encoding, or byte handling between Web Crypto and GAS would cause signature mismatches

**The better approach:** move the signing server-side using `google.script.run`, which is already available in the GAS iframe and is exactly how Phase 7 solved the same problem for heartbeat/signout.

## Migration Strategy

### Approach: Server-Side Signing via `google.script.run`

Replace client-side `_s()` calls with `google.script.run` calls to a new server-side function that:
1. Accepts a message type and any required parameters
2. Constructs the message object server-side
3. Signs it with `signMessage()` (HMAC-SHA256)
4. Returns the signed message object
5. The client-side code `postMessage`s the signed result to the host

This is the same pattern used by Phase 7's `processHeartbeat()` and `processSignOut()`.

### New server-side function

```javascript
// New: Server-side message signing for GAS session HTML
// Called via google.script.run from the session HTML inline script.
function signAppMessage(sessionToken, messageType, params) {
  // Validate session first — reject if session is invalid
  var session = validateSession(sessionToken);
  if (session.status !== 'authorized') {
    return signMessage({ type: 'gas-session-invalid', reason: 'SESSION_EXPIRED' }, session.messageKey || '');
  }

  var msgKey = session.messageKey || '';

  switch (messageType) {
    case 'gas-auth-ok':
      return signMessage({
        type: 'gas-auth-ok',
        version: VERSION,
        needsReauth: session.needsReauth || false,
        messageKey: msgKey
      }, msgKey);

    case 'gas-version':
      var appData = getAppData();
      return signMessage({
        type: 'gas-version',
        version: appData.version
      }, msgKey);

    case 'gas-user-activity':
      return signMessage({
        type: 'gas-user-activity'
      }, msgKey);

    case 'gas-session-invalid':
      return signMessage({
        type: 'gas-session-invalid',
        reason: (params && params.reason) || 'unknown'
      }, msgKey);

    default:
      return { type: 'error', error: 'unknown_message_type' };
  }
}
```

**File:** `googleAppsScripts/Testauthgas1/testauthgas1.gs`

Add the `signAppMessage()` function near the other Phase 7 server-side functions (`processHeartbeat`, `processSignOut`, `processSecurityEvent`). This function:

Currently handled via `google.script.run.getAppData()` → client constructs and signs the response. Migration: the client calls `signAppMessage(_sessionToken, 'gas-version')` which calls `getAppData()` server-side and returns a signed `gas-version` message directly.

### Step 2: Rewrite `gas-auth-ok` to use server-side signing

**File:** `googleAppsScripts/Testauthgas1/testauthgas1.gs` (session HTML template)

**Before (DJB2):**
```javascript
window.top.postMessage(_s({type: 'gas-auth-ok', version: '${escapeJs(VERSION)}',
  needsReauth: ${session.needsReauth || false},
  messageKey: '${escapeJs(appMsgKey)}'}), '${PARENT_ORIGIN}');
```

**After (HMAC via google.script.run):**
```javascript
google.script.run
  .withSuccessHandler(function(signed) {
    window.top.postMessage(signed, '${PARENT_ORIGIN}');
  })
  .withFailureHandler(function(err) {
    // Fallback: send unsigned gas-auth-ok so the host page at least knows
    // the session is valid (verification will pass because no key is set yet)
    window.top.postMessage({type: 'gas-auth-ok', version: '${escapeJs(VERSION)}',
      needsReauth: ${session.needsReauth || false},
      messageKey: '${escapeJs(appMsgKey)}'}, '${PARENT_ORIGIN}');
  })
  .signAppMessage(_sessionToken, 'gas-auth-ok');
```

**Timing consideration:** The original `_s()` call fires synchronously on page load. The `google.script.run` call is asynchronous — there will be a small delay (~100-500ms) before the host page receives `gas-auth-ok`. The host page's `_pendingSessionShow` pattern already defers `showApp()` until `gas-auth-ok` arrives, so this delay is architecturally expected and should not cause issues.

**Verification:**
- [ ] `gas-auth-ok` arrives at host page with a 64-char hex `_sig` (not base36 DJB2)
- [ ] Host page's `_verifyMessageSignature` validates via HMAC path (not DJB2 fallback)
- [ ] `messageKey` is still included in the message for host-page key import
- [ ] No visible delay or flash during page load

### Step 3: Rewrite `gas-version` to use server-side signing

**File:** `googleAppsScripts/Testauthgas1/testauthgas1.gs` (session HTML template)

**Before (DJB2 + separate getAppData call):**
```javascript
if (e.data && e.data.type === 'gas-version-check') {
  google.script.run
    .withSuccessHandler(function(data) {
      top.postMessage(_s({type: 'gas-version', version: data.version}), '${PARENT_ORIGIN}');
    })
    .withFailureHandler(function() {
      top.postMessage(_s({type: 'gas-version', version: null}), '${PARENT_ORIGIN}');
    })
    .getAppData();
}
```

**After (HMAC via signAppMessage):**
```javascript
if (e.data && e.data.type === 'gas-version-check') {
  google.script.run
    .withSuccessHandler(function(signed) {
      top.postMessage(signed, '${PARENT_ORIGIN}');
    })
    .withFailureHandler(function() {
      // Don't send an unsigned response — the version poll is periodic and will retry.
      // A missing response is safer than an unsigned one that gets dropped by HMAC verify.
    })
    .signAppMessage(_sessionToken, 'gas-version');
}
```

**Why no fallback:** The host page's `_verifyMessageSignature` will drop unsigned messages once `_hmacKey` is set. Sending an unsigned `gas-version` would be silently rejected anyway. The version polling is periodic (~10s interval) and tolerates missing responses — it simply retries on the next poll cycle.

**Verification:**
- [ ] `gas-version` arrives with a 64-char hex `_sig`
- [ ] Version polling still works (check version display updates after GAS deploy)
- [ ] Failure case: no message sent, next poll retries successfully

### Step 4: Rewrite `gas-user-activity` to use server-side signing

**File:** `googleAppsScripts/Testauthgas1/testauthgas1.gs` (session HTML template)

**Before (DJB2):**
```javascript
function _notifyActivity() {
  var now = Date.now();
  if (now - _lastActivityNotify < 5000) return;
  _lastActivityNotify = now;
  window.top.postMessage(_s({type: 'gas-user-activity'}), '${PARENT_ORIGIN}');
}
```

**After (HMAC via signAppMessage):**
```javascript
var _pendingActivity = false;
function _notifyActivity() {
  var now = Date.now();
  if (now - _lastActivityNotify < 5000) return;
  if (_pendingActivity) return; // Prevent stacking server calls
  _lastActivityNotify = now;
  _pendingActivity = true;
  google.script.run
    .withSuccessHandler(function(signed) {
      _pendingActivity = false;
      window.top.postMessage(signed, '${PARENT_ORIGIN}');
    })
    .withFailureHandler(function() {
      _pendingActivity = false;
      // Silently drop — next activity event will retry
    })
    .signAppMessage(_sessionToken, 'gas-user-activity');
}
```

**Performance consideration:** `gas-user-activity` fires on every user interaction (debounced to 5s). Each `google.script.run` call is a round-trip to Google's servers (~200-800ms). The 5-second debounce provides ample headroom — even if the round-trip takes a full second, it won't stack up. The `_pendingActivity` guard prevents concurrent requests.

**Alternative considered:** Pre-sign a batch of `gas-user-activity` messages server-side at page load and consume them one per event. Rejected because:
- Adds complexity (counter tracking, batch size, refill logic)
- The 5s debounce already limits volume to ~12 messages/minute max
- Pre-signed messages would need unique nonces to prevent replay, adding more complexity

**Verification:**
- [ ] Activity detection still triggers heartbeat in host page
- [ ] No perceptible delay on keydown/click/input
- [ ] Console shows no errors during rapid typing
- [ ] `gas-user-activity` arrives with 64-char hex `_sig`

### Step 5: Rewrite `gas-session-invalid` to use server-side signing

**File:** `googleAppsScripts/Testauthgas1/testauthgas1.gs` (session HTML template)

**Before (DJB2):**
```javascript
.withFailureHandler(function(err) {
  var msg = err.message || '';
  if (msg.indexOf('SESSION_') === 0) {
    statusEl.textContent = 'Session invalid — re-authenticate to save.';
    statusEl.style.color = '#c62828';
    window.top.postMessage(_s({type: 'gas-session-invalid', reason: msg}), '${PARENT_ORIGIN}');
  } else {
    statusEl.textContent = 'Error: ' + msg;
    statusEl.style.color = '#c62828';
  }
})
```

**After (HMAC via signAppMessage):**
```javascript
.withFailureHandler(function(err) {
  var msg = err.message || '';
  if (msg.indexOf('SESSION_') === 0) {
    statusEl.textContent = 'Session invalid — re-authenticate to save.';
    statusEl.style.color = '#c62828';
    google.script.run
      .withSuccessHandler(function(signed) {
        window.top.postMessage(signed, '${PARENT_ORIGIN}');
      })
      .withFailureHandler(function() {
        // Double failure — can't even sign the invalid notification.
        // User already sees "re-authenticate" text in the GAS iframe.
        // Don't send unsigned — host page would drop it anyway.
      })
      .signAppMessage(_sessionToken, 'gas-session-invalid', {reason: msg});
  } else {
    statusEl.textContent = 'Error: ' + msg;
    statusEl.style.color = '#c62828';
  }
})
```

**Edge case — signing when session is invalid:** `gas-session-invalid` is sent precisely when the session has expired. The `signAppMessage()` function's session validation at the top will detect this and return a signed `gas-session-invalid` response from the validation check itself. This is the correct path — the function should handle this case in its validation block:

```javascript
// In signAppMessage():
var session = validateSession(sessionToken);
if (session.status !== 'authorized') {
  // Session is invalid — this is expected for 'gas-session-invalid' callers.
  // Recover messageKey from the session data if possible (cache may still have it).
  var cache = CacheService.getScriptCache();
  var raw = cache.get('session_' + sessionToken);
  var recoveredKey = '';
  if (raw) { try { recoveredKey = JSON.parse(raw).messageKey || ''; } catch(e) {} }
  return signMessage({
    type: 'gas-session-invalid',
    reason: 'SESSION_EXPIRED'
  }, recoveredKey);
}
```

The key recovery is important — when a session expires, the `messageKey` may still be in the cache even though `validateSession` returns `unauthorized`. If the key is gone too, the message is sent unsigned and will be dropped by the host page — but the user already sees the "re-authenticate" text in the GAS iframe, so the UX is preserved.

**Verification:**
- [ ] Save Note with an expired session triggers `gas-session-invalid` at host page
- [ ] Message arrives with a 64-char hex `_sig` (when key is recoverable)
- [ ] Host page handles the session invalidation (clears session, shows sign-in)
- [ ] UI feedback ("re-authenticate") still appears in the GAS iframe regardless of signing success

## Ordering and Phase 8 Dependency

```
Step 1: Add signAppMessage() to GAS
    ↓
Steps 2-5: Migrate each message type (can be done individually with testing between)
    ↓
Step 6: Remove _s(), DJB2 legacy code from both files
    ↓
Step 7: Update docs
    ↓
Phase 8: CSP Hardening (compute hash of final inline script)
```

**Critical:** Steps 2-5 each modify the inline `<script>` content. Phase 8 MUST wait until all steps are complete and verified, because any further script changes would invalidate the CSP hash.

**Dual-accept during migration:** Steps 2-5 can be done incrementally — the host page's `_verifyMessageSignature()` accepts both HMAC and DJB2 until Step 6 removes DJB2. This means partially-migrated states are safe: migrated messages use HMAC, un-migrated ones still use DJB2, and both pass verification.

## Rollback Procedure

Each step is independently rollbackable:

1. **Revert `signAppMessage()`** — remove the function from `testauthgas1.gs`
2. **Revert individual messages** — restore `_s()` call for the specific message type
3. **Restore DJB2 verification** — re-add `_verifyDjb2Legacy()` and the fallback path in `_verifyMessageSignature()`

**Full rollback:** Revert to the pre-migration commit. The dual-accept system in the host page means old DJB2-signed messages will still be accepted even after a GAS rollback — there is no "signature mismatch" window during rollback.
