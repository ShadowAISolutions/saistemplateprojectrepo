## Why This Revision Exists

The original plan (document 9) proposed a **hidden iframe + `doGet(?check=)` polling** mechanism. While functionally correct, quota analysis revealed a critical cost problem:

**Current steady-state `doGet` load:** ~0.2 invocations/min/user (heartbeat at 5min production interval)
**Original plan's load:** ~6.2 invocations/min/user (session check at 10s + heartbeat)
**Impact:** A **~30x increase** in `doGet` frequency per user

With Google Apps Script consumer accounts limited to ~20,000 daily executions, the original plan would exhaust quotas at ~5 concurrent users in production — unacceptable for any deployment beyond a single-user prototype.

### The Insight

The existing **version check** mechanism already demonstrates a zero-`doGet` pattern: the GAS app iframe (served by `doGet(?session=TOKEN)`) contains an inline `<script>` that listens for `gas-version-check` postMessages and calls `google.script.run.getAppData()` — an internal RPC channel to the GAS server that does **not** go through `doGet`. This is the same internal channel used by sidebars, dialogs, and add-ons.

**The session check can use the exact same pattern.** Instead of creating a second hidden iframe that hits `doGet(?check=TOKEN)`, we embed a `setInterval` + `google.script.run.checkSessionValid(token)` call inside the existing GAS app iframe. The server function performs a single `CacheService.get()` and returns the result. The iframe relays it to the parent via `postMessage`.

**Result:** Zero additional `doGet` invocations. The steady-state `doGet` load stays at ~0.2/min/user.

## Hard Constraints (Inherited + New)

All constraints from the original plan (document 9) remain in effect. This section documents them with annotations specific to this revised approach.

### Constraint A: No new external dependencies
Unchanged. The `google.script.run` channel is native to GAS — no new dependencies.

### Constraint B: GAS quota safety
**Improved.** `google.script.run` calls count against the same overall daily execution quota as `doGet`, but they avoid the **web app request overhead** (HTTP connection, cold start risk, response serialization to HTML). The per-client concurrent call limit for `google.script.run` is **10 simultaneous calls** ([Google Apps Script communication docs](https://developers.google.com/apps-script/guides/html/communication)), and the per-project limit is **30 simultaneous executions** ([Google Apps Script quotas](https://developers.google.com/apps-script/guides/services/quotas)). A single session check consuming 1 of 10 available slots every 5-10 seconds is well within bounds.

**Quota math (revised):**

| Metric | Original Plan (`doGet`) | Revised Plan (`google.script.run`) |
|--------|------------------------|-----------------------------------|
| `doGet` calls/min/user (production) | ~6.2 | ~0.2 (heartbeat only — unchanged) |
| Server function executions/min/user | ~6.2 | ~6.2 (same frequency, different channel) |
| Consumer daily quota at 10 users (8hr day) | ~89,280 (exceeds 20k limit) | ~14,880 (heartbeat ~960 + session checks ~13,920) — **within limit** |
| Cold start exposure | Every check can trigger cold start | Only heartbeat triggers cold start; `google.script.run` runs in already-warm runtime |

**Important nuance:** `google.script.run` calls still count against the daily execution quota. The difference is:
1. They avoid the `doGet` web app request path (no HTTP overhead, no cold start)
2. They run inside an already-initialized GAS runtime (the app iframe)
3. They are significantly faster (~100-400ms vs 400-1500ms for `doGet` warm calls)
4. They are not subject to the undocumented "Service invoked too many times" throttle that applies to `doGet` endpoints, since they route through a different internal channel

**Consumer account feasibility:** At 10 concurrent users with 10s polling:
- Session checks: 6/min × 10 users × 480 min = 28,800/day
- Heartbeats: 0.2/min × 10 users × 480 min = 960/day
- **Total: ~29,760/day** — exceeds the 20k consumer limit

At 15s polling: ~19,200 session checks + 960 heartbeats = **~20,160/day** — borderline. At 20s polling: ~14,400 + 960 = **~15,360/day** — safe with margin.

**Recommendation:** Use 10s for HIPAA preset (faster detection, typically paid Workspace with higher limits) and 15s for standard preset (consumer-safe). Document the quota tradeoff in the toggle description.

### Constraint C: Background tab throttling tolerance
Unchanged. The `setInterval` timer inside the GAS iframe is subject to the same browser throttling as any timer in a background tab:
- After 5 minutes in background: Chrome throttles to once per minute ([Chrome dev blog](https://developer.chrome.com/blog/timer-throttling-in-chrome-88))
- The GAS iframe is cross-origin (`*.googleusercontent.com`), which means Chrome may apply even more aggressive throttling — cross-origin iframes in background tabs are throttled to once per second initially, then once per minute after 5 minutes ([Chromium background tab docs](https://docs.google.com/document/d/18_sX-KGRaHcV3xe5Xk_l6NNwXoxm-23IOepgMx4OlE4/pub))
- When the user returns to the tab, timers restore immediately
- This is acceptable — see original plan document 9 section "Browser Background Tab Throttling Analysis" for the full rationale

**Additional consideration for `google.script.run` in background tabs:** unlike `doGet` calls (which fire via iframe `src` changes and complete regardless of tab state), `google.script.run` callbacks (`withSuccessHandler`, `withFailureHandler`) execute JavaScript in the iframe, which may be delayed by throttling. However, the callback only needs to fire `postMessage` — a nearly instantaneous operation — so even if the callback is delayed by throttling, the delay is the timer interval itself (already throttled), not an additional delay on top of it.

### Constraint D: Reuse existing UI
Unchanged. The "Session Active Elsewhere" overlay (`#tab-takeover-wall`) is reused with dynamic text based on eviction source.

### Constraint E: HMAC message signing must be maintained
**Simplified.** In the original plan, the `doGet(?check=)` endpoint had to construct HTML with inline JavaScript to compute an HMAC signature and send it via `postMessage`. The eviction message was **unsigned** because the signing key was deleted with the session — this was identified as a trade-off.

With `google.script.run`, the server function returns data directly to the iframe's JavaScript context (via the success/failure handler). The iframe code already has the message key (`_mk` variable, set during the session validation HTML render at line ~982 of testauth1.gs). The iframe signs the `postMessage` to the parent using the same `_s()` signing function already used for `gas-auth-ok` and `gas-version` messages.

**Critical improvement:** unlike the original plan, **both** the "session valid" and "session evicted" messages can be signed — because the signing key is already in the iframe's JavaScript memory. The key isn't re-read from the cache; it was embedded in the iframe HTML when the session was established. This eliminates the unsigned-eviction-message attack vector identified in the original plan.

### Constraint F: Toggleable and independent
Unchanged. The feature is independently toggleable via `HTML_CONFIG.CROSS_DEVICE_SESSION_CHECK` and `AUTH_CONFIG.ENABLE_SESSION_CHECK`, separate from `SINGLE_TAB_ENFORCEMENT`.

### Constraint G: No spreadsheet access in the hot path
Unchanged. `checkSessionValid()` only uses `CacheService.get()`.

### NEW Constraint H: GAS iframe lifecycle dependency
The session check timer lives inside the GAS app iframe. If the iframe is reloaded (e.g., by the heartbeat mechanism, sign-out flow, or "Use Here" reclaim), the timer is destroyed and must be re-established. This is the primary trade-off of this approach vs. the original plan's separate hidden iframe.

**Mitigation:** the iframe is only reloaded on explicit lifecycle events (sign-in, sign-out, reclaim) — all of which already restart or stop the heartbeat. The session check lifecycle mirrors the heartbeat lifecycle exactly: start after sign-in, stop on sign-out, restart on reclaim. The iframe is NOT reloaded during normal heartbeat polling (heartbeat uses its own hidden iframe `gas-heartbeat`, not `gas-app`).

**Verification:** the `gas-app` iframe's `src` is set in these scenarios:
1. After `gas-session-created` — `gasApp.src = baseUrl + '?session=' + token` (line ~1192 of testauth1.html). This is the initial load; the session check timer starts in the rendered HTML
2. After "Use Here" button — same reload pattern (line ~1827). Timer restarts in the new render
3. After sign-out — `gasApp.src = baseUrl + '?signOut=' + token` (line ~1736). Timer is irrelevant (signing out)
4. Normal heartbeat — uses `gas-heartbeat` iframe, NOT `gas-app`. `gas-app` is untouched

**Conclusion:** the GAS app iframe is stable during normal operation. The timer lifecycle matches existing patterns and requires no new lifecycle management beyond what the heartbeat already does.

## Implementation Phases

### Phase 1: GAS — Add `checkSessionValid` Server Function

**Risk:** Low
**Files:** `testauth1.gs`
**What:** Add a new server-side function that performs a read-only session validity check via `CacheService`. This function is called via `google.script.run` from inside the GAS app iframe — it is NOT a `doGet` parameter handler.

**New function (add after `getAppData()`, near line ~171):**

```javascript
/**
 * Lightweight session validity check for cross-device enforcement.
 * Called from the app iframe via google.script.run on a timer.
 * Read-only — does NOT extend the session (unlike heartbeat).
 *
 * @param {string} token — the session token to check
 * @return {Object} — { valid: boolean }
 */
function checkSessionValid(token) {
  if (!token || typeof token !== 'string' || token.length < 32) {
    return { valid: false };
  }

  // Rate limit: max 60 checks per session per 5-minute window
  // (supports ~12/min sustained, covering 5s polling with margin)
  var cache = CacheService.getScriptCache();
  var rlKey = 'sc_ratelimit_' + token.substring(0, 16);
  var attempts = cache.get(rlKey);
  var count = attempts ? parseInt(attempts, 10) : 0;
  if (count >= 60) {
    return { valid: true, rateLimited: true };
    // Return valid:true when rate limited — failing open is safer than
    // incorrectly evicting a user. The rate limit protects the server,
    // not the session validity signal. The client should back off on
    // rateLimited:true to reduce load.
  }
  cache.put(rlKey, String(count + 1), 300);

  // Read-only check — does the session still exist in cache?
  var raw = cache.get('session_' + token);
  return { valid: !!raw };
}
```

**Design decisions:**

1. **Returns `{ valid: boolean }` not HTML** — unlike `doGet` handlers that return `HtmlService.createHtmlOutput()`, this function returns a plain JavaScript object. `google.script.run` handles serialization automatically — the success handler receives the object directly. No HTML construction, no inline `<script>` tags, no `postMessage` inside the response. This is dramatically simpler than the original plan's approach.

2. **No HMAC verification on the session data** — the check only tests whether the session key exists in CacheService, not whether it's valid/uncorrupted. HMAC verification would require parsing the session JSON and computing a hash — unnecessary for a simple existence check. If the session data was corrupted, the next heartbeat or page reload will catch it through the existing validation path.

3. **No session extension** — identical to the original plan. The check does NOT reset `createdAt`, update `lastActivity`, or rewrite the session. It's pure observation. This means:
   - Session check requests do not extend the session (only heartbeats do)
   - Session check requests do not count as user activity
   - If a user's session expires naturally (idle timeout), the session check correctly reports eviction

4. **Rate limit returns `valid: true`** — failing open when rate-limited is safer than failing closed. If the server is overwhelmed, evicting users would compound the problem. The `rateLimited` flag tells the client to back off without disrupting the user's session.

5. **Token validation** — basic length/type check before hitting CacheService. Session tokens are 48-character hex strings (line 434-435 of testauth1.gs: `generateToken(48)`). The `length < 32` check rejects obvious junk without being so strict it breaks on future token format changes.

6. **Rate limit at 60/5min** — identical to the original plan. Supports a 5-second polling interval with ~50% headroom (12 checks/min × 5 min = 60). The heartbeat uses a separate rate limit key (`hb_ratelimit_`), so the combined load on CacheService is 80 rate-limit operations per 5 minutes per user — well within limits.

**Security considerations:**

- **Session enumeration risk:** identical to the original plan. 48 hex chars = 192 bits of entropy — brute-force is infeasible. The rate limiter adds further protection.
- **Caller authentication:** `google.script.run` can only be called from within a GAS-served iframe. An attacker cannot call `checkSessionValid()` from an external page — Google's infrastructure restricts `google.script.run` to iframes served by the same GAS project. This is a **stronger security boundary** than the original plan's `doGet(?check=)` endpoint, which is a public web URL accessible to anyone with the deployment URL.
- **Return value interception:** the return value travels through Google's internal channel to the iframe's success handler. It does not traverse the public internet as a separate HTTP response — it's part of the existing iframe-to-server session. An attacker would need to compromise the GAS iframe itself (same-origin with `*.googleusercontent.com`) to intercept the response.

### Phase 3: GAS — Embed Session Check Timer in App Iframe HTML

**Risk:** Medium (new polling loop inside the GAS iframe — must coexist with existing version check listener)
**Files:** `testauth1.gs`
**What:** Add a `setInterval` + `google.script.run.checkSessionValid()` loop inside the authenticated app HTML (the HTML returned by `doGet(?session=TOKEN)` when the session is valid).

**Location:** Inside the `<script>` block of the app HTML (lines ~978-1007 of testauth1.gs), after the existing `gas-version-check` message listener.

**Add the following code block:**

```javascript
// Cross-device session check: poll server for session validity
// Uses google.script.run (internal RPC) — no doGet overhead
(function() {
  var _scToken = '${escapeJs(sessionToken)}';
  var _scInterval = ${AUTH_CONFIG.ENABLE_SESSION_CHECK ? (AUTH_CONFIG.SESSION_CHECK_INTERVAL || 10) : 0};
  var _scInFlight = false;
  var _scTimerId = null;
  var _scEvicted = false;

  if (_scInterval <= 0) return;  // Feature disabled

  _scTimerId = setInterval(function() {
    if (_scInFlight) return;   // Don't stack requests
    if (_scEvicted) return;    // Already evicted — stop checking

    _scInFlight = true;
    google.script.run
      .withSuccessHandler(function(result) {
        _scInFlight = false;
        if (!result) return;

        if (result.rateLimited) {
          // Server is rate-limiting us — back off by stopping the timer
          if (_scTimerId) { clearInterval(_scTimerId); _scTimerId = null; }
          return;
        }

        if (!result.valid) {
          // Session was invalidated — notify parent
          _scEvicted = true;
          if (_scTimerId) { clearInterval(_scTimerId); _scTimerId = null; }
          window.top.postMessage(
            _s({ type: 'gas-session-evicted', source: 'cross-device' }),
            '${PARENT_ORIGIN}'
          );
        }
        // If valid — do nothing, continue polling
      })
      .withFailureHandler(function(err) {
        _scInFlight = false;
        // Server error — don't evict, just skip this cycle
        // The next interval will retry
      })
      .checkSessionValid(_scToken);
  }, _scInterval * 1000);

  // Listen for stop signal from parent (on sign-out or tab surrender)
  window.addEventListener('message', function(e) {
    if (e.data && e.data.type === 'stop-session-check') {
      _scEvicted = true;
      if (_scTimerId) { clearInterval(_scTimerId); _scTimerId = null; }
    }
  });
})();
```

**Design decisions:**

1. **Self-contained IIFE** — the session check logic is wrapped in an immediately-invoked function expression, isolating all variables from the rest of the iframe's scope. This prevents conflicts with existing code (version check listener, message signing, etc.).

2. **Uses existing `_s()` signing function** — the eviction message is signed using `_s()`, which is already defined in the iframe HTML (lines ~982-987 of testauth1.gs). The signing key `_mk` is set when the iframe HTML is rendered with the session's `messageKey`. This means **both valid and evicted messages are signed** — a significant security improvement over the original plan where eviction messages were unsigned.

3. **`_scInFlight` prevents stacking** — identical pattern to the existing heartbeat. If `google.script.run` takes longer than the polling interval (e.g., due to GAS cold start on the first call), the next interval is skipped rather than stacking a second concurrent call.

4. **`_scEvicted` flag** — once eviction is detected, the timer stops permanently. There's no scenario where we'd want to resume checking after eviction — the user must re-authenticate.

5. **Failure handler does nothing** — if `google.script.run` throws a server error, we skip the cycle and retry on the next interval. Evicting a user due to a server error would be wrong — the session might still be valid; the server just had a transient issue. This is a fail-open design.

6. **`stop-session-check` message listener** — the parent page needs a way to stop the session check when the user signs out or the tab is surrendered. The parent sends `{type: 'stop-session-check'}` to the iframe via `postMessage`. The listener sets `_scEvicted = true` and clears the timer. This message does not need signing — it's a command to stop work, not a security-sensitive data exchange. An attacker sending a spoofed `stop-session-check` would only cause the session check to stop (a denial of the enforcement feature), and the BroadcastChannel enforcement still operates independently.

7. **Server-rendered interval** — `_scInterval` is set via template literal from `AUTH_CONFIG.SESSION_CHECK_INTERVAL`, not passed from the parent page. This prevents the client from overriding the server's intended interval. The server controls how frequently it wants to be polled.

8. **Token is server-rendered** — `_scToken` is set via template literal from the validated `sessionToken`, not from a parent postMessage. This ensures the iframe checks the exact session token the server knows about, not a potentially stale or spoofed token from the parent.

```javascript
        + 'Only one session can be active at a time.';
    }
    var useHereBtn = document.getElementById('tab-use-here-btn');
    if (useHereBtn) {
      useHereBtn.textContent = 'Sign In Here';
    }
    takeoverWall.style.display = 'flex';
  }
}
```

**Security note:** the `gas-session-evicted` message is **signed** in this revised plan (the iframe uses `_s()` to sign it). The existing HMAC verification layer (lines ~1148-1150 of testauth1.html) will validate the signature before this handler runs. An attacker cannot spoof an eviction by sending an unsigned `gas-session-evicted` postMessage — it will be silently dropped by the signature check.

#### 4e. `stopSessionCheck()` helper

Add near the existing `stopHeartbeat()` function:

```javascript
function stopSessionCheck() {
  if (!HTML_CONFIG.CROSS_DEVICE_SESSION_CHECK) return;
  // Tell the GAS iframe to stop its internal timer
  var gasApp = document.getElementById('gas-app');
  if (gasApp && gasApp.contentWindow) {
    try {
      gasApp.contentWindow.postMessage(
        { type: 'stop-session-check' }, '*'
      );
    } catch (e) {
      // Cross-origin access may throw — iframe might already be navigating
    }
  }
}
```

**Why `'*'` for the target origin:** the GAS iframe is served from `*.googleusercontent.com` — a Google-controlled domain that varies per deployment. The `stop-session-check` message is a non-sensitive command (it only stops the timer), so origin restriction is not critical. The iframe's listener checks `e.data.type`, which is sufficient for this purpose.

#### 4f. Lifecycle wiring — start

The session check timer starts automatically when the GAS app iframe renders (it's embedded in the HTML via Phase 3). No explicit `startSessionCheck()` call is needed from the parent page. This is simpler than the original plan, which required the parent to manage the polling lifecycle.

**Verification:** the timer starts in these scenarios:
1. **After successful sign-in:** `gas-session-created` handler sets `gasApp.src = baseUrl + '?session=' + token` (line ~1192). The GAS server renders the app HTML with the embedded session check timer. Timer starts automatically.
2. **After "Use Here" reclaim:** the button handler sets `gasApp.src = baseUrl + '?session=' + token` (line ~1827). Same result — timer starts in the new render.
3. **After page load with existing session:** the initialization flow sets `gasApp.src = baseUrl + '?session=' + token`. Timer starts in the render.

In all cases, the timer is a side effect of the iframe loading its authenticated HTML — no parent-side coordination needed.

## What Changed From the Original Plan

| Aspect | Original Plan (Doc 9) | Revised Plan (Doc 9.1) |
|--------|----------------------|----------------------|
| **Polling mechanism** | Hidden iframe + `doGet(?check=TOKEN)` | `google.script.run.checkSessionValid(token)` inside existing GAS app iframe |
| **`doGet` impact** | +6/min/user (30x increase) | 0 additional `doGet` calls |
| **New iframes** | 1 (`gas-session-check`) | 0 |
| **New `doGet` parameters** | `?check=` | None |
| **Eviction message signing** | Unsigned (signing key deleted with session) | **Signed** (key already in iframe memory) |
| **Parent-side lifecycle** | `startSessionCheck()` / `stopSessionCheck()` managed by parent | Timer auto-starts in iframe render; `stopSessionCheck()` sends stop signal |
| **Rate limit response** | Separate `gas-session-check-error` message type | `{ valid: true, rateLimited: true }` return value — iframe handles internally |
| **New message types needed** | 3 (`gas-session-valid`, `gas-session-evicted`, `gas-session-check-error`) | 1 (`gas-session-evicted`) |
| **Server function** | Inline in `doGet` handler | Standalone `checkSessionValid()` function |
| **Cold start exposure** | Every check can trigger cold start | Runtime already warm from iframe load |
| **Consumer quota safety (10 users)** | Exceeds 20k limit | Within limit at 15s interval |

## Alternatives Considered

### Alternative A: Original plan's hidden iframe + `doGet(?check=)` approach (Document 9)

**Why superseded:** 30x increase in `doGet` invocations per user. Quota-unsafe for consumer accounts. More complex (requires new iframe, new `doGet` handler, HTML response construction with inline script). Unsigned eviction messages.

**When it might still be preferred:** if the GAS app iframe is unstable (frequently reloaded by other mechanisms), the separate hidden iframe provides more isolation. In the current architecture, the GAS app iframe is stable during normal operation, so this concern doesn't apply.

### Alternative B-E: See original plan document 9

Firebase (violates constraint A), long polling (consumes execution quota), localStorage cross-tab (same-browser only), heartbeat piggybacking (too slow, activity-gated). All remain rejected for the same reasons.

### Alternative F: Web Worker for throttle-resistant polling

**What:** move the `google.script.run` polling timer to a Web Worker, which is not subject to background tab throttling.

**Why not implemented (yet):** Web Workers cannot call `google.script.run` — they don't have access to the GAS iframe's DOM context. The Worker would need to communicate with the iframe via `postMessage` to trigger the `google.script.run` call, which adds complexity for marginal benefit (60s background detection is acceptable per the original plan's analysis). **Future enhancement** if sub-minute background detection becomes a requirement.

## Implementation Order

**Recommended sequence:** Phases 1 → 2 → 3 → 4 → 5

All phases can be implemented in a single commit. The feature is entirely toggleable. Setting `CROSS_DEVICE_SESSION_CHECK: false` in `HTML_CONFIG` prevents the parent from handling eviction messages, and setting `ENABLE_SESSION_CHECK: false` in `AUTH_CONFIG` prevents the GAS iframe from starting the polling timer.

**Dependencies:**
- Phase 3 depends on Phase 1 (iframe calls `checkSessionValid` function)
- Phase 3 depends on Phase 2 (iframe reads `AUTH_CONFIG.SESSION_CHECK_INTERVAL`)
- Phase 4 depends on Phase 3 (parent handles messages from iframe's timer)
- Phase 5 depends on Phase 4 (tests verify functions/variables from Phase 4)
