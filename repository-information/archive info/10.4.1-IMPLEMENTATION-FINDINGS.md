# 10.4.1 — HIPAA Single-Load Auth: Implementation Findings

> **Status:** Reverted — implementation attempt revealed issues that need further investigation
> **Date:** 2026-03-18
> **Versions attempted:** v04.89r through v04.91r (all reverted)
> **Affects:** `testauthgas1.gs`, `testauthgas1.html`

---

## 1. What Was Attempted

The goal was to reduce the HIPAA postMessage login from 2 `doGet()` calls to 1, per the plan in `10.4.1-HIPAA-SINGLE-LOAD-AUTH-OPTIMIZATION-PLAN.md`.

### Implementation approach

1. **`exchangeTokenAndBuildApp()`** — new server function wrapping `exchangeTokenForSession()` + `validateSession()` + `buildAppHtmlString()`. Returns session data + app HTML markup in a single `google.script.run` response
2. **`buildAppHtmlString()`** — extracted the app HTML body content from `doGet()`'s `?session=` path into a shared builder function
3. **Listener page modification** — the HIPAA listener page's success handler:
   - Posts `gas-session-created` with `appLoaded: true` flag to the parent
   - Injects app HTML via `document.body.innerHTML`
   - Directly calls `google.script.run.signAppMessage()` and sets up event listeners (version-check, activity detection, save-note)
4. **HTML `appLoaded` guard** — `gas-session-created` handler skips the `?session=` iframe reload when `appLoaded: true`

### Changes across 3 commits

| Commit | Version | What changed |
|--------|---------|-------------|
| `cf3247d` | v04.89r | Initial implementation — all 4 steps |
| `e167397` | v04.90r | Fix: replaced `createElement('script')` + `appendChild` with direct `google.script.run` calls (scripts don't auto-execute via innerHTML) |
| `4351e50` | v04.91r | Fix: split `buildAppHtmlString()` into `{ markup, script }` to prevent double `signAppMessage` execution from innerHTML-injected `<script>` tags |

## 2. What Worked

- **Login succeeded** — the single-load flow completed successfully. The user was authenticated, the app UI appeared, session timers started, and all interactive features (save note, version check, activity detection) were functional
- **`google.script.run` survived `innerHTML` replacement** — confirmed the core premise of the plan: replacing `document.body.innerHTML` does NOT destroy the `google.script.run` bridge. The listener page's success handler was able to call `signAppMessage()` after the innerHTML injection
- **`gas-session-created` with `appLoaded: true` was received** — the parent page correctly received the flag and skipped the `?session=` iframe reload
- **`gas-auth-ok` arrived from the single-load path** — `signAppMessage()` returned a signed `gas-auth-ok` that was verified by the parent's HMAC logic
- **Session resume (`?session=` path) was unaffected** — page refresh and "Use Here" tab reclaim continued to work via the legacy path

## 3. Issues Encountered

### Issue 1: Stuck on "Signing in..." (v04.89r)

**Symptom:** Backend showed login success, but the UI was stuck on the "Signing in... Setting up your session" spinner.

**Root cause:** The initial approach used `createElement('script')` + `appendChild` to manually execute `<script>` tags injected via `innerHTML`. Two problems:
1. `getElementsByTagName("script")` returns a **live HTMLCollection** — appending new `<script>` elements to the body grows the collection mid-loop, causing an infinite loop
2. Even with the live collection fix (snapshot to static array), script execution via `createElement` was unreliable in the GAS sandbox

**Fix (v04.90r):** Replaced the script execution approach entirely. The listener page's success handler now directly calls `google.script.run.signAppMessage()` and sets up all event listeners, bypassing innerHTML script execution.

### Issue 2: Double state changes / visual "blink" (v04.90r → v04.91r)

**Symptom:** Console showed 6 IDLE→BUSY→IDLE net state transitions instead of the expected 4 (2 round-trips). The GAS iframe visually "blinked" — the app appeared, then briefly flickered.

**Diagnosis:** `buildAppHtmlString()` returned HTML that included `<script>` tags containing `google.script.run.signAppMessage()`. While standard browsers do NOT execute scripts injected via `innerHTML`, the GAS sandbox (`googleusercontent.com` iframe) may have non-standard behavior that executes them. This caused `signAppMessage` to fire **twice**:
1. From the listener's direct call (our fix from v04.90r)
2. From the innerHTML-injected `<script>` tag (GAS sandbox non-standard behavior)

**Fix attempt (v04.91r):** Split `buildAppHtmlString()` into `{ markup, script }`. The innerHTML path receives markup only (zero `<script>` tags). The `?session=` path receives both.

### Issue 3: Persistent visual "blink" (v04.91r — NOT RESOLVED)

**Symptom:** Even after removing all `<script>` tags from the innerHTML injection, the visual "blink" persists.

**Possible causes (not yet confirmed):**
1. **innerHTML visual flash** — replacing `document.body.innerHTML` causes a brief blank frame as the browser destroys the old DOM and paints the new one. This is inherent to the technique and may not be avoidable without a different approach (e.g. hiding the iframe during the transition)
2. **GAS sandbox re-initialization** — the innerHTML replacement may trigger internal GAS framework behavior (sandbox re-initialization, permission policy re-evaluation) that causes a visual flash
3. **`gas-auth-ok` processing** — the parent page receives `gas-auth-ok` and calls `showApp()`, which hides the auth wall overlay. The transition from auth wall → visible iframe may cause a flash if the iframe content isn't fully painted yet
4. **CSS/layout recalculation** — the injected markup uses inline styles, not `<style>` blocks. The initial render without styles followed by style application could cause a brief layout shift

**What was NOT the cause:**
- A second `doGet()` call — the `appLoaded: true` guard correctly prevented the `?session=` reload
- `signAppMessage` double execution — resolved by removing `<script>` tags from innerHTML path

| Step | Type | Count |

**Net savings: 1 `doGet()` per login (25% reduction)** — this part worked correctly.

However, the `exchangeTokenAndBuildApp()` function is heavier than the original `exchangeTokenForSession()` because it also calls `validateSession()` internally. The added ~0.5s overhead is offset by eliminating the ~1-3s `doGet()` round-trip.

## 5. Key Technical Learnings

1. **`google.script.run` survives `innerHTML` replacement** — CONFIRMED. The GAS communication bridge is a JavaScript runtime object, not a DOM node. This is the foundation the plan was built on, and it works
2. **`innerHTML` does NOT auto-execute `<script>` tags** — TRUE in standard browsers, but the **GAS sandbox may behave differently**. The `googleusercontent.com` sandbox iframe has non-standard behavior that may execute innerHTML-injected scripts. Always strip `<script>` tags from innerHTML-injected content in GAS
3. **`getElementsByTagName` returns a live collection** — appending new elements of the same type to the DOM during iteration causes an infinite loop. Always snapshot with `Array.prototype.slice.call()` first
4. **Visual flash from `innerHTML` replacement may be unavoidable** — even with clean markup (no scripts), replacing `document.body.innerHTML` causes a brief visual transition that users perceive as a "blink." This may require a different approach (CSS transitions, opacity animation, or a pre-paint hidden swap)
5. **The `document.write()` violation in GAS consoles is from Google's own code** (`ae_html_user.js:315`), not from application code. It fires on every GAS iframe load

## 6. Recommendations for Future Attempt

If this optimization is revisited, consider these approaches to address the visual blink:

### Option A: Pre-paint hidden swap
1. Before innerHTML replacement, set `document.body.style.visibility = 'hidden'`
2. Replace innerHTML
3. Use `requestAnimationFrame` to wait for paint, then restore visibility
4. This hides the transition completely

### Option B: Overlay technique
1. Create a full-screen white overlay div positioned on top
2. Replace innerHTML behind the overlay
3. After paint, fade out the overlay
4. Smoother transition with no blink

### Option C: Fragment-based approach
1. Instead of `innerHTML`, build the new DOM in a `DocumentFragment`
2. Clear the body, then append the fragment
3. May avoid the flash since the fragment is pre-built before insertion

### Option D: Abandon innerHTML, use srcdoc
1. Instead of innerHTML replacement, use `iframe.srcdoc` to set new content
2. This is a full page replacement but without a network round-trip
3. However, this destroys `google.script.run` (same as `document.write`)
4. NOT viable — included only for completeness

### Option E: Accept the blink, investigate further
1. The blink may be acceptable for the quota savings (25% reduction)
2. Use DevTools Performance panel to identify exactly what causes the flash
3. May turn out to be a single-frame paint issue fixable with CSS
