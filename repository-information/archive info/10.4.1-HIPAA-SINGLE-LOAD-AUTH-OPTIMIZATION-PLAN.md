## 1. Problem Statement

The HIPAA login flow currently requires **two separate `doGet()` calls** plus two `google.script.run` calls — a total of 4 GAS executions per sign-in:

| Load | URL | Purpose | GAS executions |
|------|-----|---------|----------------|
| Load 0 | `baseUrl` (no params) | Serve listener page → exchange OAuth token via postMessage | 1 `doGet()` + 1 `google.script.run` |
| Load 1 | `baseUrl?session=<token>` | Validate session → serve full app HTML → sign `gas-auth-ok` | 1 `doGet()` + 1 `google.script.run` |

**Total: 4 GAS executions per login** (2 `doGet()` + 2 `google.script.run`)

### Why this matters

- **Quota consumption:** All GAS executions (both `doGet()` and `google.script.run`) count against the per-account daily quota ([official quotas page](https://developers.google.com/apps-script/guides/services/quotas)). Two `doGet()` calls per login means double the quota burn for the most critical operation
- **Latency:** The second iframe load (`?session=`) requires a full HTTP round-trip to Google's servers (~1–3 seconds) during which the user sees nothing
- **Attack surface:** Each network round-trip is an interception opportunity — fewer round-trips = smaller attack surface (HIPAA principle: minimize ePHI exposure windows per [§164.502(b)](https://www.hhs.gov/hipaa/for-professionals/security/laws-regulations/index.html))
- **Token-in-URL exposure:** The `?session=<token>` reload puts the session token in a URL, which appears in browser history and server logs

### The goal

Eliminate Load 1 entirely. The listener page (Load 0) should be able to transform itself into the full app UI after the token exchange succeeds, without a second `doGet()`.

## 3. Why the Original Plan Dismissed This

The `10.4` plan concluded:

> *"The iframe can't rewrite itself with new HTML after `exchangeTokenForSession()` returns — `google.script.run` returns data, not HTML."*

This statement conflated two separate things:

1. **`google.script.run` returns data, not HTML** — TRUE. It returns JSON-serializable values (primitives, objects, arrays)
2. **You can't replace the page content with that data** — FALSE. The returned data can include an HTML string, and that string can be injected into the DOM via `innerHTML`

The original plan then proposed a workaround (pre-signing `gas-auth-ok` inside the exchange call) but abandoned it as "marginal and complex." It never considered the `innerHTML` SPA approach because it assumed the page content couldn't be replaced.

### What the original plan missed

`google.script.run` is a **JavaScript API object in the runtime scope** — it is NOT a DOM element. When you replace `document.body.innerHTML`, you destroy and rebuild DOM nodes, but you do NOT destroy JavaScript variables or API objects like `google.script.run`. The GAS-to-server communication bridge survives DOM manipulation.

This is a well-documented pattern in the GAS community:
- [Kanshi Tanaike's article on injecting HTML with JavaScript in GAS web apps](https://medium.com/google-cloud/inserting-html-including-javascript-on-web-apps-created-by-google-apps-script-7b051d04937b)
- [GAS community multi-page SPA discussion](https://groups.google.com/g/google-apps-script-community/c/3tuhzMBfAE4)
- [Google's official HTML Service communication docs](https://developers.google.com/apps-script/guides/html/communication)

The technique that DOES break `google.script.run` is `document.write()` / `document.open()` — those destroy the entire page including GAS-injected scripts. `innerHTML` does not.

## 4. The Breakthrough: innerHTML SPA Technique

### How it works

The GAS community has a well-established pattern for building single-page applications (SPAs) within a single `doGet()` load:

1. **Server-side function returns HTML as a string** — `google.script.run` can return any JSON-serializable value, including an HTML string
2. **Client-side injects the HTML via `innerHTML`** — replaces the page content without destroying the JavaScript runtime
3. **Scripts are manually executed** — `innerHTML` does not auto-execute `<script>` tags (browser security). They must be evaluated explicitly via `new Function()` or by creating `<script>` elements with `document.createElement('script')`
4. **`google.script.run` remains alive** — it's a JS runtime object, not a DOM node. The GAS-to-server bridge survives `innerHTML` replacement

### What breaks vs. what survives

| Operation | `google.script.run` | Event listeners | `<script>` execution |
|-----------|---------------------|-----------------|---------------------|
| `innerHTML` replacement | ✅ Survives | ❌ Lost (re-attach needed) | ❌ Must manually execute |
| `document.write()` | ❌ DESTROYED | ❌ Lost | ✅ Auto-executes |
| `document.createElement` + `appendChild` | ✅ Survives | ✅ Preserved | ✅ Auto-executes |

**We use `innerHTML` because it lets us replace the entire page body while keeping `google.script.run` alive.** The lost event listeners and non-executing scripts are handled by the manual script execution step — the app HTML's `<script>` block sets up all event listeners when it runs.

### Why `document.write()` is NOT an option

`document.write()` / `document.open()` destroys the entire page including the GAS-injected scripts that power `google.script.run`. This is confirmed by [Google's troubleshooting docs](https://developers.google.com/apps-script/guides/support/troubleshooting) and [MDN's documentation on document.write()](https://developer.mozilla.org/en-US/docs/Web/API/Document/write): calling `document.write()` after the page has loaded implicitly calls `document.open()`, which clears the current document.

### Confidence level

**High — each component is individually documented and confirmed:**
- `innerHTML` injection in GAS: [Kanshi Tanaike (Medium)](https://medium.com/google-cloud/inserting-html-including-javascript-on-web-apps-created-by-google-apps-script-7b051d04937b)
- `google.script.run` surviving DOM manipulation: [Google official docs](https://developers.google.com/apps-script/guides/html/communication) show `innerHTML` updates in `google.script.run` success handlers as the standard pattern
- Manual script execution after `innerHTML`: [Standard web technique](https://www.xjavascript.com/blog/can-scripts-be-inserted-with-innerhtml/)
- GAS SPA multi-page apps: [Community discussion](https://groups.google.com/g/google-apps-script-community/c/3tuhzMBfAE4)

**Untested combination:** The specific sequence (exchange token → return app HTML string → inject via `innerHTML` → execute scripts → `google.script.run.signAppMessage()` fires from injected code) has not been tested end-to-end in this exact auth context. A proof-of-concept is recommended before full implementation (see Verification Checklist, section 10).

## 5. Optimized Architecture (1 Load)

### HIPAA path — after optimization

```
User clicks "Sign In" → Google OAuth → handleTokenResponse()
    │
    ▼  HTML: exchangeViaPostMessage(googleAccessToken)
    │    ├─ Generate cryptographic nonce
    │    └─ Set iframe src to baseUrl (no params)
    │
    ▼
┌─ SINGLE LOAD: doGet() — no URL params ───────────────────────────────────┐
│  GAS returns listener page (modified):                                    │
│    ├─ Posts {type: 'gas-ready-for-token'} to parent                      │
│    ├─ Waits for {type: 'exchange-token', accessToken, nonce} message     │
│    └─ On receive:                                                         │
│        google.script.run.exchangeTokenAndBuildApp(token)                  │
│          ├─ exchangeTokenForSession(token) — all existing checks          │
│          ├─ IF success: validateSession(sessionToken)                     │
│          ├─ IF success: build app HTML string (reuse existing builder)    │
│          └─ Returns: {success, sessionToken, email, displayName,          │
│                       messageKey, absoluteTimeout, nonce,                  │
│                       appHtml: "<full app HTML body>"}                     │
│                                                                           │
│  Success handler (in listener page):                                      │
│    1. Posts gas-session-created to parent (with appLoaded: true)          │
│    2. Replaces document.body.innerHTML with result.appHtml                │
│    3. Manually executes <script> tags in the injected HTML               │
│       → google.script.run.signAppMessage() fires from injected code      │
│       → posts gas-auth-ok to parent                                       │
│                                                                           │
│  Failure handler: posts gas-session-created with error (unchanged)        │
└───────────────────────────────────────────────────────────────────────────┘
    │
    ▼  HTML receives gas-session-created (with appLoaded: true):
    │    ├─ Verify nonce
    │    ├─ _importHmacKey(messageKey)
    │    ├─ saveSession(token, email)
    │    ├─ _pendingSessionShow = {email, displayName}
    │    └─ SKIP iframe reload (appLoaded flag detected)
    │
    ▼  HTML receives gas-auth-ok (from same iframe — no reload):
    │    ├─ showApp(email, displayName)
    │    ├─ startCountdownTimers()
    │    ├─ startHeartbeat()
    │    └─ broadcastTabClaim()
    │
    ▼  ✅ APP VISIBLE (one load instead of two)
```

**Total: 1 `doGet()` + 2 `google.script.run` = 3 GAS executions (down from 4)**

### What changes vs. what stays the same

| Component | Before | After |
|-----------|--------|-------|
| Load 0 (`doGet()` no params) | Returns listener page | Returns listener page (modified success handler) |
| `exchangeTokenForSession()` | Called via `google.script.run` | Wrapped in new `exchangeTokenAndBuildApp()` |
| Load 1 (`doGet(?session=)`) | Returns full app HTML | **ELIMINATED** |
| `signAppMessage()` | Called from Load 1's inline script | Called from innerHTML-injected script |
| Session resume (page refresh) | Uses `?session=` path | **Unchanged** |
| Tab reclaim ("Use Here") | Uses `?session=` path | **Unchanged** |
| `gas-session-created` handler | Always reloads iframe | Skips reload when `appLoaded: true` |
| All security mechanisms | Active | **Unchanged** |

### Message ordering guarantee

The `gas-session-created` postMessage must arrive at the parent BEFORE `gas-auth-ok`. This is guaranteed because:

1. The listener page's success handler runs synchronously: it calls `window.top.postMessage(gas-session-created)` FIRST
2. Then it injects the app HTML and executes the scripts
3. The injected script calls `google.script.run.signAppMessage()` which is ASYNC (server round-trip)
4. Therefore `gas-session-created` is delivered and processed before `gas-auth-ok` even starts

The parent imports the HMAC key from `gas-session-created.messageKey`, then uses it to verify the signature on `gas-auth-ok` when it arrives — the same ordering as the current flow.

## 6. Implementation Steps

### Step 1: Create `exchangeTokenAndBuildApp()` server function (GAS)

**File:** `testauthgas1.gs` — new function

This wraps the existing `exchangeTokenForSession()` and adds app HTML generation on success:

```javascript
/**
 * Combined exchange + app HTML builder for HIPAA single-load optimization.
 * Called from the listener page via google.script.run.
 * Returns session data + app HTML string in one response.
 */
function exchangeTokenAndBuildApp(googleAccessToken) {
  // Step 1: Exchange token (existing function — unchanged)
  var result = exchangeTokenForSession(googleAccessToken);

  if (!result.success) {
    // Return error — no app HTML needed
    return {
      success: false,
      error: result.error || 'exchange_failed'
    };
  }

  // Step 2: Validate the just-created session
  var session = validateSession(result.sessionToken);
  if (session.status !== 'authorized') {
    return {
      success: false,
      error: 'session_validation_failed'
    };
  }

  // Step 3: Build app HTML as a string
  // Uses the same builder as the ?session= path, but returns a string
  // instead of an HtmlOutput object
  var appHtml = buildAppHtmlString(result.sessionToken, session, result.messageKey);

  return {
    success: true,
    sessionToken: result.sessionToken,
    email: result.email || '',
    displayName: result.displayName || '',
    absoluteTimeout: result.absoluteTimeout || 0,
    messageKey: result.messageKey || '',
    appHtml: appHtml
  };
}
```

**Risk:** Low — wraps existing functions, no new security logic.

### Step 2: Create `buildAppHtmlString()` helper (GAS)

**File:** `testauthgas1.gs` — new function

Extracts the app HTML body content (lines 1423–1604) as a plain string, WITHOUT the `HtmlService.createHtmlOutput()` wrapper. This is the same HTML that the `?session=` path serves, but returned as a string for `innerHTML` injection.

 * Used by exchangeTokenAndBuildApp() for innerHTML injection,
 * and by the ?session= path (wrapped in HtmlService.createHtmlOutput).

function buildAppHtmlString(sessionToken, session, appMsgKey) {
  var html = `
    <!-- App HTML body content — same as ?session= path -->
    <div style="...">
      <!-- Debug marker, save-note form, heartbeat input, version, email -->
      ...existing markup from lines 1448–1462...
    </div>
    <h2 id="version">${escapeHtml(VERSION)}</h2>

**Risk:** Low — pure refactor. The `?session=` path produces identical output.

### Step 3: Modify listener page success handler (GAS)

**File:** `testauthgas1.gs` — modify lines 1362–1400

var listenerHtml = '<!DOCTYPE html><html><head><meta charset="utf-8"></head><body><script>'

  + '        window.top.postMessage({'
  + '          type: "gas-session-created",'
  + '          success: result.success || false,'
  + '          error: result.error || "",'

  + '    })'
  + '    .withFailureHandler(function(err) {'
  + '      var code = "server_error";'
  + '      if (err && err.message && err.message.indexOf("HMAC_SECRET") !== -1) code = "hmac_secret_missing";'
  + '      window.top.postMessage({'
  + '        type: "gas-session-created",'
  + '        success: false,'
  + '        error: code,'
  + '        nonce: nonce'
  + '      }, ' + JSON.stringify(PARENT_ORIGIN) + ');'
  + '    })'
  + '    .exchangeTokenAndBuildApp(token);'
  + '});'

**Key detail on script execution:** We use `document.createElement('script')` + `appendChild` instead of `new Function()` because:
- `createElement('script')` preserves the normal script execution context (global scope, `this` binding)
- The injected scripts reference global `google.script.run` which must be in scope
- This technique is cleaner than `eval()` or `new Function()` and avoids CSP `unsafe-eval` issues

**Risk:** Medium — new control flow in the success path. Failure path is unchanged.

## 7. Security Invariants

Every security mechanism from Phases 1–9 must remain functional. This table maps each invariant to how the optimization affects it:

| # | Invariant | Status | Analysis |

| 7 | `showApp()` deferred until `gas-auth-ok` | ✅ Unchanged | The `_pendingSessionShow` pattern is identical — `gas-session-created` defers, `gas-auth-ok` triggers |

| 11 | Session resume still works | ✅ Unchanged | Resume uses `?session=` path which is not modified |
| 12 | "Use Here" tab reclaim still works | ✅ Unchanged | Tab reclaim uses `?session=` path which is not modified |
| 13 | Sign-out clears HMAC key | ✅ Unchanged | Sign-out logic is in the parent page, not in the GAS iframe |

### New security consideration: Script injection via `innerHTML`

The `innerHTML` injection introduces a theoretical attack vector: if an attacker could control the `result.appHtml` string returned by `exchangeTokenAndBuildApp()`, they could inject arbitrary scripts.

**Why this is NOT a risk:**
- `result.appHtml` is generated entirely server-side by `buildAppHtmlString()` — the function constructs the HTML from hardcoded template literals and server-validated session data
- The only dynamic values interpolated are `sessionToken` (server-generated 48-char random), `session.email` (from validated Google OAuth), `VERSION` (hardcoded constant), and `PARENT_ORIGIN` (hardcoded constant)
- All dynamic values are escaped via `escapeJs()` and `escapeHtml()` before interpolation
- The `google.script.run` call itself is authenticated — only the script owner's GAS functions can be called
- An attacker would need to compromise the GAS server-side code to modify `buildAppHtmlString()`, at which point they already have full control regardless of this optimization

Every distinct technique in this plan was individually evaluated against HIPAA requirements. Techniques that fail are rejected; techniques that pass are documented with their reasoning.

| # | Technique | HIPAA Verdict | Reasoning |

| 8 | **App HTML string (~10KB) via `google.script.run`** | ✅ PASS | Contains same data as current Load 1 (`_sessionToken` escaped, `session.email` escaped, constants). Same internal GAS communication channel. No new data exposure |

None. All techniques in this plan pass HIPAA evaluation.

### Overall HIPAA verdict

**This optimization is HIPAA-positive:**
- No new data exposure paths
- One fewer `doGet()` network round-trip = smaller attack surface
- Session token removed from URL during login = better token hygiene
- All existing security mechanisms (HMAC, nonce, origin validation, audit logging, escalating lockout) remain unchanged
- The `?session=` path continues to exist for session resume and tab reclaim — only the initial login flow is optimized
- Audit log entries remain separate per HIPAA §164.312(b) requirements

### Caveats requiring ongoing vigilance

1. **`buildAppHtmlString()` must never interpolate unescaped user input.** The current implementation only interpolates server-generated values (`sessionToken`, `session.email`, `VERSION`, `PARENT_ORIGIN`) through `escapeJs()` and `escapeHtml()`. If a future change adds user-supplied values, they MUST be escaped — otherwise the `innerHTML` injection becomes an XSS vector
2. **Phase 8 CSP must accommodate `createElement('script')`.** The dynamically created script elements need to be permitted by the Content Security Policy. This is noted in the Prerequisites (implement before Phase 8) — Phase 8's CSP design must account for this technique

## 10. Audit Log Consolidation — Evaluated and Rejected

### The question

Each login currently produces 2 audit log entries in the session audit spreadsheet:

| Row | Timestamp | Category | Email | Event | Details |
|-----|-----------|----------|-------|-------|---------|
| 590 | 2026-03-18T19:29:39.586Z | session_management | shadow@... | all_sessions_invalidated | {"count":1} |
| 591 | 2026-03-18T19:29:40.493Z | login_success | shadow@... | session_created | {"sessionId":"2589ccb3..."} |

Could these be consolidated into a single entry to reduce `SpreadsheetApp.appendRow()` calls?

### Code analysis

The two entries come from different functions at different points in the login flow:

1. **`all_sessions_invalidated`** — logged inside `invalidateAllSessions()` (line 836 in testauthgas1.gs), called from `exchangeTokenForSession()` at line 582. Only fires when there are existing sessions to kill (`tokens.length > 0`)
2. **`session_created`** — logged at line 618 in `exchangeTokenForSession()`, after the new session is stored in CacheService

### HIPAA analysis — why consolidation is rejected

Per [HIPAA audit trail requirements](https://compliancy-group.com/hipaa-audit-log-requirements/) and the [2026 HIPAA Security Rule](https://www.hipaavault.com/resources/2026-hipaa-changes/):

**1. These are distinct security events affecting different entities.**
- `all_sessions_invalidated` is a **forced logout** — it terminates existing sessions (potentially on other devices). HIPAA guidance states: *"Session management creates another layer of necessary documentation. You must track when sessions begin and end, document idle timeouts, and record any forced logouts."* Forced logouts require their own audit entries ([source](https://www.keragon.com/hipaa/hipaa-explained/hipaa-audit-log-requirements))
- `session_created` is a **login event** — a new session being established

Combining a forced logout and a login into one entry conflates two security-relevant actions. An investigator asking "when were the user's old sessions terminated?" would have to parse metadata inside a `session_created` entry instead of finding a dedicated `all_sessions_invalidated` entry.

**2. `invalidateAllSessions()` is called from multiple contexts.**
It's not only called during login — it can be called for security reasons (admin-triggered invalidation, suspicious activity). When called outside of login, the `all_sessions_invalidated` entry exists on its own. Consolidating it only during login creates an inconsistency: the same security event (`all_sessions_invalidated`) would sometimes be a standalone entry and sometimes be buried inside a `session_created` entry. This makes automated log analysis unreliable.

**3. Conditional firing creates format inconsistency.**
`all_sessions_invalidated` only fires when `tokens.length > 0`. On a first-ever login (no existing sessions), only `session_created` fires. Consolidating would mean the `session_created` entry sometimes contains invalidation data and sometimes doesn't — making the log harder to parse programmatically and harder for auditors to review.

**4. The performance cost is negligible.**
The two `SpreadsheetApp.appendRow()` calls add ~1 second to the login flow. They don't consume additional GAS execution quota (they run inside the same `exchangeTokenForSession()` execution). The optimization target of this plan is `doGet()` quota elimination, not spreadsheet write reduction.

### Verdict: REJECTED

**Keeping separate audit log entries is the HIPAA-correct approach.** The two entries represent distinct security events (forced logout vs. new login), are generated by different functions called in different contexts, and fire conditionally. Consolidating them would reduce audit trail clarity and create format inconsistencies — both of which work against HIPAA's audit control requirements under [§164.312(b)](https://docs.alertlogic.com/analyze/reports/compliance/HIPAA-164.312-audit-controls.htm).

The ~1 second of `SpreadsheetApp` overhead is an acceptable cost for audit trail integrity.

## 12. Rollback Plan

### Quick rollback (GAS-only)

1. **Revert `testauthgas1.gs`** — restore the original listener page (lines 1362–1400) that calls `exchangeTokenForSession()` instead of `exchangeTokenAndBuildApp()`
2. **Redeploy GAS**
3. **Done** — no HTML change needed

When the GAS side stops sending `appLoaded: true`, the HTML handler falls through to the `else` branch and reloads the iframe — exactly the current behavior.

### Why HTML doesn't need to change

The `appLoaded` check in the `gas-session-created` handler (Step 4) is a pure additive guard:
- `appLoaded: true` → skip reload (optimized path)
- `appLoaded` absent → reload iframe (current behavior)

If the GAS side reverts to the old listener page, `appLoaded` is never set, and the HTML behaves identically to before the optimization.

### Feature flag option

```javascript
// In AUTH_CONFIG presets:
SINGLE_LOAD_AUTH: true,  // true = optimized, false = legacy 2-load

// In listener page:
if (AUTH_CONFIG.SINGLE_LOAD_AUTH) {
  // Call exchangeTokenAndBuildApp (returns appHtml)
  google.script.run...exchangeTokenAndBuildApp(token);
} else {
  // Call exchangeTokenForSession (original behavior)
  google.script.run...exchangeTokenForSession(token);
}
```
