# Chat Bookends (MANDATORY вЂ” EVERY PROMPT)

*Actionable rules for every response. Examples and summary tables in `chat-bookends-reference.md`.*

> **Feature toggle gate** вЂ” before emitting any bookend, check the Template Variables table:
> - If `CHAT_BOOKENDS` = `Off`: **skip all mid-response bookends** вЂ” CODING PLAN, CODING START, RESEARCH START, RESEARCHING, NEXT PHASE, CHECKLIST, BLOCKED, VERIFYING, CHANGES PUSHED, AWAITING HOOK, HOOK FEEDBACK, ESTIMATED TIME, REVISED ESTIMATED TIME, ACTUAL PLANNING TIME, PLAN APPROVED, and all `вЏ±пёЏ` duration annotations. Proceed directly to the work. The hook anticipation logic (running the three git commands) still executes вЂ” only its bookend output is suppressed. **Silent timestamp and estimate capture** вЂ” when `END_OF_RESPONSE_BLOCK` = `On`, the underlying data needed by end-of-response block features is still captured silently (no visible output): (a) run `date` at the very start of the response and internally note the **response start timestamp** вЂ” this replaces the CODING START/RESEARCH START timestamp for ACTUAL TOTAL COMPLETION TIME computation; (b) compute the overall estimate internally вЂ” this provides the parenthetical comparison value for ACTUAL TOTAL COMPLETION TIME and the baseline for ESTIMATE CALIBRATED; (c) if a revised estimate would be warranted (scope changed в‰Ґ1m after initial reads), internally note the revised value вЂ” it replaces the original for the parenthetical; (d) if a plan approval flow occurs, internally note the post-approval timestamp and post-approval estimate вЂ” these enable PLAN EXECUTION TIME. When `END_OF_RESPONSE_BLOCK` is also `Off`, skip the silent captures entirely вЂ” nothing consumes them
> - If `END_OF_RESPONSE_BLOCK` = `Off`: **skip the entire end-of-response block** вЂ” no `в”Ђв”Ђв”Ђв”Ђв”Ђ` dividers, no END OF RESPONSE BLOCK header, no UNAFFECTED URLS, AGENTS USED, FILES CHANGED, COMMIT LOG, WORTH NOTING, SUMMARY, TODO, NEW FOLDERS, AFFECTED URLS, ESTIMATE CALIBRATED, PLAN EXECUTION TIME, ACTUAL TOTAL COMPLETION TIME, or closing marker (CODING COMPLETE / RESEARCH COMPLETE). Mid-response bookends (when `CHAT_BOOKENDS` = `On`) function fully and independently вЂ” they do not depend on the end-of-response block for any data or state
> - **Full independence** вЂ” the two toggles have zero cross-dependencies. Every feature of `END_OF_RESPONSE_BLOCK` works without `CHAT_BOOKENDS` (via silent timestamp/estimate capture), and every feature of `CHAT_BOOKENDS` works without `END_OF_RESPONSE_BLOCK`. No other system functionality depends on either toggle being in a specific state вЂ” both are purely output-presentation controls that gate what the user sees, not what data is collected
> - When both are `On` (the default), all rules below apply as written

- **First output вЂ” coding plan**: for every user prompt that will involve changes, the very first line written to chat must be `рџљ©рџљ©CODING PLANрџљ©рџљ©` on its own line, followed by a brief bullet-point list of what will be done in this response, then a **blank line** followed by `вљЎвљЎCODING STARTвљЎвљЎ` on its own line to signal work is beginning. The blank line is required to break out of the bullet list context so CODING START renders left-aligned. Keep the plan concise вЂ” one bullet per distinct action (e.g. "Edit CLAUDE.md to add coding plan rule", "Update README.md timestamp"). This is for transparency, not approval вЂ” do NOT wait for user confirmation before proceeding. If the response is purely informational with no changes to make, skip the plan and open with `рџ”¬рџ”¬RESEARCH STARTрџ”¬рџ”¬` directly (instead of CODING START). **CODING PLAN and CODING START / RESEARCH START appear exactly once per response** вЂ” never repeat them mid-response. Use `рџ”„рџ”„NEXT PHASEрџ”„рџ”„` instead (see below)
- **Planned affected URLs**: immediately after the coding plan bullets (after the blank line that ends the bullet list, before ESTIMATED TIME), output `рџ”—вњЏпёЏPLANNED AFFECTED URLSвњЏпёЏрџ”—` followed by the page URLs expected to be affected by this response вЂ” using the same label-URL pair format as the end-of-response `рџ”—вњЏпёЏAFFECTED URLSвњЏпёЏрџ”—` section (backtick-wrapped `вњЏпёЏ` labels, blockquoted URLs). **Show the current (pre-change) version** in the label вЂ” read the page's `html.version.txt` and append the version in parentheses: `` `вњЏпёЏ Homepage (v01.13w)` ``. This gives the user clickable links to the files' current state and shows what version the page is at before changes begin. **Best-effort prediction** вЂ” base it on the coding plan bullets; if the scope changes mid-work, the final AFFECTED URLS at the end may differ. When no pages are expected to be affected, output the header followed by `> *No URL pages expected to be affected*`. **Skip entirely for RESEARCH START responses** (no plan, no affected URLs). When the coding plan repeats after `рџ“‹рџ“‹PLAN APPROVEDрџ“‹рџ“‹`, include PLANNED AFFECTED URLS again after those plan bullets (the prediction may be more accurate post-research)
- **Continuation after user interaction**: when `AskUserQuestion` or `ExitPlanMode` returns mid-response (the user answered a question or approved a plan), the response continues but must **NOT** repeat `рџљ©рџљ©CODING PLANрџљ©рџљ©`, `вљЎвљЎCODING STARTвљЎвљЎ`, or `рџ”¬рџ”¬RESEARCH STARTрџ”¬рџ”¬`. Instead:
  - After `AskUserQuestion`: use `рџ”„рџ”„NEXT PHASEрџ”„рџ”„` with a description incorporating the user's choice (e.g. "User chose option A вЂ” proceeding with implementation")
  - After `ExitPlanMode` (plan approved): output `рџ“‹рџ“‹PLAN APPROVEDрџ“‹рџ“‹` on its own line, followed by `рџљ©рџљ©CODING PLANрџљ©рџљ©` with the execution plan bullets, then `вљЎвљЎCODING STARTвљЎвљЎ`. This is the **only** scenario where CODING PLAN/CODING START may appear a second time вЂ” because plan approval is a distinct boundary between planning and execution, and the user needs to see the execution plan clearly. The `рџ“‹рџ“‹PLAN APPROVEDрџ“‹рџ“‹` marker signals that this is a continuation, not a new prompt. **Fresh timestamps required** вЂ” run `date` once for PLAN APPROVED, then run `date` again for the post-approval CODING PLAN + CODING START pair (a single call covers both, same as the initial opening pair rule). Do NOT reuse the PLAN APPROVED timestamp for CODING START вЂ” there is a gap between plan approval and the start of execution (the coding plan bullets and estimated time are written in between), so the timestamps will differ. PLAN EXECUTION TIME is computed from the post-approval start timestamp (whether displayed via CODING START or silently captured), so an inaccurate timestamp here inflates or deflates the reported execution time
- **Checklist running**: output `вњ”пёЏвњ”пёЏCHECKLISTвњ”пёЏвњ”пёЏ` on its own line before executing any mandatory checklist (Session Start, Pre-Commit, Pre-Push), followed by the checklist name (e.g. `Session Start Checklist`). - **Researching**: output `рџ”Ќрџ”ЌRESEARCHINGрџ”Ќрџ”Ќ` on its own line when entering a research/exploration phase вЂ” reading files, searching the codebase, or understanding context before making changes. - **Mid-response phase marker**: when work within a single response naturally divides into multiple distinct sub-tasks or phases (e.g. "Edit 1" then "Edit 1a: fix related issue"), output `рџ”„рџ”„NEXT PHASEрџ”„рџ”„` on its own line followed by a brief description of the new phase. **Never repeat** `рџљ©рџљ©CODING PLANрџљ©рџљ©`, `вљЎвљЎCODING STARTвљЎвљЎ`, or `рџ”¬рџ”¬RESEARCH STARTрџ”¬рџ”¬` within the same response вЂ” those appear exactly once (at the very top). The mid-response marker keeps the top/bottom boundaries of each prompt/response turn unambiguous while still signaling transitions between sub-tasks. **`вЏ±пёЏ` must precede this bookend** (see Duration annotations)
- **Blocked**: output `рџљ§рџљ§BLOCKEDрџљ§рџљ§` on its own line when an obstacle is hit (permission denied, merge conflict, ambiguous requirement, failed push, hook check failure). Follow with a brief description of the blocker. This makes problems immediately visible rather than buried in tool output. **`вЏ±пёЏ` must precede this bookend** (see Duration annotations)
- **Verifying**: output `рџ§Єрџ§ЄVERIFYINGрџ§Єрџ§Є` on its own line when entering a verification phase вЂ” running git hook checks, confirming no stale references, validating edits post-change. Separates "doing the work" from "checking the work". **`вЏ±пёЏ` must precede this bookend** (see Duration annotations)
- **Changes pushed**: output `вћЎпёЏвћЎпёЏCHANGES PUSHEDвћЎпёЏвћЎпёЏ` on its own line **immediately** after a successful `git push` completes. This gives the user instant visibility that their changes are on the remote вЂ” no need to scroll through tool output to confirm. Follow with a brief note of what was pushed (e.g. "Pushed to `claude/cleanup-xyz` вЂ” workflow will auto-merge to main"). This bookend is a timestamped marker like any other вЂ” run `date` before writing it. It participates in `вЏ±пёЏ` duration tracking (the previous phase's `вЏ±пёЏ` appears before it, and the next bookend's `вЏ±пёЏ` measures from it)
- **Time estimate**: output `вЏівЏіESTIMATED TIME в‰€ XmвЏівЏі` on its own line, followed by a brief reason (e.g. "~8 file edits + commit + push cycle"), in **two contexts**: (1) **Overall** вЂ” **always** appears immediately before the opening marker (`вљЎвљЎCODING STARTвљЎвљЎ` or `рџ”¬рџ”¬RESEARCH STARTрџ”¬рџ”¬`), estimating the entire response from start to the closing marker. This gives the user an upfront sense of total wall-clock time. The overall estimate is **never skipped** вЂ” even for quick responses (e.g. `вЏівЏіESTIMATED TIME в‰€ 1mвЏівЏі`). Use `Xm` for minutes or `Xs` for estimates under 1 minute. (2) **Per-phase** вЂ” immediately before any subsequent phase bookend (NEXT PHASE, RESEARCHING, CHECKLIST, etc.) whose phase alone is expected to take longer than 2 minutes. **Only output per-phase estimates when the estimate exceeds 2 minutes** вЂ” skip for phases that will be quick. Use these rough heuristics to estimate: ~10s per tool call (read, edit, grep, glob), ~15s per bash command, ~30s per commit cycle (checklist + staging + commit), ~30s per push cycle (checklist + push + verify), ~1вЂ“2m per subagent spawn. **Parallel batching discount** вЂ” when multiple independent tool calls will be fired in a single batch (e.g. 5 parallel reads, or 5 parallel edits), count the batch as ~15s total, not N Г— 10s. Most push-commit cycles fire 4вЂ“6 edits in parallel, so the editing phase is ~15s, not ~50s. Apply this discount when you know the calls are independent and will be batched. Sum the expected tool calls (with discount) for the phase and round to the nearest minute. The estimate does not need a timestamp or `date` call вЂ” it is an annotation, not a bookend that participates in `вЏ±пёЏ` duration tracking
- **Revised estimate**: once all initial file reads, greps, and exploration needed to understand the scope of work are complete вЂ” regardless of whether a RESEARCHING bookend was used вЂ” re-evaluate the overall estimate. If it now differs from the original by в‰Ґ1 minute, output `вЏівЏіREVISED ESTIMATED TIME в‰€ XmвЏівЏі [HH:MM:SS AM/PM EST]` on its own line with a brief reason (e.g. "~12 files to edit, not 4 as initially expected"). **Run `date` before writing the line** вЂ” this is a timestamped marker like any other bookend. Place it inline right after the last read/exploration tool result and before the next action or bookend. **Skip if the original estimate still looks accurate.** The revised estimate replaces the original as the baseline for the ACTUAL TOTAL COMPLETION TIME comparison at the end
- **Actual time**: output `вЏівЏіACTUAL TOTAL COMPLETION TIME: Xm Ys (estimated Xm)вЏівЏі` on its own line immediately before the closing marker (`вњ…вњ…CODING COMPLETEвњ…вњ…` or `рџ”¬рџ”¬RESEARCH COMPLETEрџ”¬рџ”¬`). The response start timestamp is captured by CODING START/RESEARCH START when `CHAT_BOOKENDS` = `On`, or silently via `date` at response start when `CHAT_BOOKENDS` = `Off` (see feature toggle gate). The parenthetical shows the original overall estimate for easy comparison (captured visibly or silently depending on the `CHAT_BOOKENDS` toggle). If a REVISED ESTIMATED TIME was issued, use the revised value instead. **Always present** when CODING COMPLETE or RESEARCH COMPLETE is written (never skipped). **Not present** before AWAITING USER RESPONSE вЂ” that ending uses ACTUAL PLANNING TIME instead. The `date` call for the closing marker (already required) provides the end time вЂ” no additional `date` call is needed
- **Plan execution time** (conditional): when a plan approval flow was used (`ExitPlanMode` в†’ `рџ“‹рџ“‹PLAN APPROVEDрџ“‹рџ“‹`), the end-of-response block must include **both** time markers in this order: (1) `вЏівЏіPLAN EXECUTION TIME: Xm Ys (estimated Xm)вЏівЏі` вЂ” the wall-clock time from the post-approval start timestamp (captured by CODING START when `CHAT_BOOKENDS` = `On`, or silently when `CHAT_BOOKENDS` = `Off`) to CODING COMPLETE, with the post-approval estimate as the parenthetical, then (2) `вЏівЏіACTUAL TOTAL COMPLETION TIME: Xm Ys (estimated Xm)вЏівЏі` вЂ” the sum of the ACTUAL PLANNING TIME and the PLAN EXECUTION TIME, with the original overall estimate (from the response start) as the parenthetical. **Skip PLAN EXECUTION TIME entirely when no plan approval occurred** вЂ” responses without `ExitPlanMode` only show ACTUAL TOTAL COMPLETION TIME as usual
- **Estimate calibration**: after computing ACTUAL TOTAL COMPLETION TIME, if the actual and estimated times differ by **more than 2 minutes**, perform a calibration step **before** writing CODING COMPLETE. This makes estimation accuracy self-improving over time:
  1. **Diagnose the miss** вЂ” identify what caused the gap. Common causes: underestimating the number of file reads needed, not accounting for checklist overhead, forgetting the push cycle, unexpected retries or errors, or heuristic values being too low/high for the type of work done
  2. **Update the heuristics** вЂ” edit the Time estimate bullet's heuristic values in this file to better reflect reality. For example, if tool calls consistently take ~15s instead of ~10s, change `~10s per tool call` to `~15s per tool call`. If commit cycles consistently take ~45s, update `~30s per commit cycle` to `~45s per commit cycle`. Only change values where the evidence from this response supports the adjustment вЂ” do not guess or over-correct
  3. **Output the calibration** вЂ” in the end-of-response block, output `рџ”§рџ”§ESTIMATE CALIBRATEDрџ”§рџ”§` after SUMMARY and before ACTUAL TOTAL COMPLETION TIME, followed by a brief description of what was adjusted and why (e.g. "Increased tool call estimate from ~10s to ~15s вЂ” actual tool calls averaged 14s this response"). This makes the self-correction visible to the user
  4. **Commit the update** вЂ” include the heuristic change in the current commit (or make a follow-up commit if the main commit already happened). The updated heuristics take effect starting from the next response
  **Skip calibration entirely if the difference is в‰¤2 minutes** вЂ” small variances are normal and not worth correcting

## Hook Anticipation вЂ” Bug Context

**The failure pattern:** if the hook conditions are evaluated *before* a `git push` completes (or evaluated mentally instead of actually running the git commands), the prediction can be wrong вЂ” e.g. concluding there are unpushed commits when the push already succeeded. Writing `рџђџрџђџAWAITING HOOKрџђџрџђџ` in that case means the hook never fires (because all conditions are actually false), and the conversation gets stuck with no closing marker.

**What to watch for:** any scenario where actions (especially `git push`) complete in the same response as the hook check. The temptation is to predict the outcome rather than wait and verify.

**The fix:** (1) always evaluate *after* all actions in the response are complete, and (2) *actually run* the three git commands вЂ” never reason about their output mentally.

## Mid-Response Bookends вЂ” Omission Prevention

**The failure pattern:** during complex multi-edit responses (especially those involving multiple file edits, checklists, and push cycles), mid-response bookends (CHECKLIST, RESEARCHING, NEXT PHASE, VERIFYING, CHANGES PUSHED, etc.) get silently dropped. The response produces correct file changes but reads as an undifferentiated wall of tool calls with no phase markers or `вЏ±пёЏ` duration annotations between them. This violates the Duration annotations rule ("No exceptions вЂ” if two bookends appear in sequence, there must be a `вЏ±пёЏ` line between them") and makes it impossible for the user to track progress or understand timing.
