# Chat Bookends (MANDATORY тАФ EVERY PROMPT)

*Actionable rules for every response. Examples and summary tables in `chat-bookends-reference.md`.*

> **Feature toggle gate** тАФ before emitting any bookend, check the Template Variables table:
> - If `CHAT_BOOKENDS` = `Off`: **skip all mid-response bookends** тАФ CODING PLAN, CODING START, RESEARCH START, RESEARCHING, NEXT PHASE, CHECKLIST, BLOCKED, VERIFYING, CHANGES PUSHED, AWAITING HOOK, HOOK FEEDBACK, ESTIMATED TIME, REVISED ESTIMATED TIME, ACTUAL PLANNING TIME, PLAN APPROVED, and all `тП▒я╕П` duration annotations. Proceed directly to the work. The hook anticipation logic (running the three git commands) still executes тАФ only its bookend output is suppressed. **Silent timestamp and estimate capture** тАФ when `END_OF_RESPONSE_BLOCK` = `On`, the underlying data needed by end-of-response block features is still captured silently (no visible output): (a) run `date` at the very start of the response and internally note the **response start timestamp** тАФ this replaces the CODING START/RESEARCH START timestamp for ACTUAL TOTAL COMPLETION TIME computation; (b) compute the overall estimate internally тАФ this provides the parenthetical comparison value for ACTUAL TOTAL COMPLETION TIME and the baseline for ESTIMATE CALIBRATED; (c) if a revised estimate would be warranted (scope changed тЙе1m after initial reads), internally note the revised value тАФ it replaces the original for the parenthetical; (d) if a plan approval flow occurs, internally note the post-approval timestamp and post-approval estimate тАФ these enable PLAN EXECUTION TIME. When `END_OF_RESPONSE_BLOCK` is also `Off`, skip the silent captures entirely тАФ nothing consumes them
> - If `END_OF_RESPONSE_BLOCK` = `Off`: **skip the entire end-of-response block** тАФ no `тФАтФАтФАтФАтФА` dividers, no END OF RESPONSE BLOCK header, no UNAFFECTED URLS, AGENTS USED, FILES CHANGED, COMMIT LOG, WORTH NOTING, PROMPT, SUMMARY, TODO, NEW FOLDERS, AFFECTED URLS, ESTIMATE CALIBRATED, PLAN EXECUTION TIME, ACTUAL TOTAL COMPLETION TIME, or closing marker (CODING COMPLETE / RESEARCH COMPLETE). Mid-response bookends (when `CHAT_BOOKENDS` = `On`) function fully and independently тАФ they do not depend on the end-of-response block for any data or state
> - **Full independence** тАФ the two toggles have zero cross-dependencies. Every feature of `END_OF_RESPONSE_BLOCK` works without `CHAT_BOOKENDS` (via silent timestamp/estimate capture), and every feature of `CHAT_BOOKENDS` works without `END_OF_RESPONSE_BLOCK`. No other system functionality depends on either toggle being in a specific state тАФ both are purely output-presentation controls that gate what the user sees, not what data is collected
> - When both are `On` (the default), all rules below apply as written

- **First output тАФ coding plan**: for every user prompt that will involve changes, the very first line written to chat must be `ЁЯЪйЁЯЪйCODING PLANЁЯЪйЁЯЪй` on its own line, followed by a brief bullet-point list of what will be done in this response, then a **blank line** followed by `тЪбтЪбCODING STARTтЪбтЪб` on its own line to signal work is beginning. The blank line is required to break out of the bullet list context so CODING START renders left-aligned. **Prompt quote тАФ first bullet**: the very first bullet in the coding plan must be a blockquoted copy of the user's **full, unabridged** prompt, prefixed with `> **Prompt:**`. Format: `- > **Prompt:** "exact user prompt text here"`. **Never truncate, shorten, or abbreviate the prompt** тАФ do not use "..." or ellipsis to cut it short, do not paraphrase, do not summarize. The entire prompt must appear verbatim, no matter how long it is. This preserves the user's original request in the chat history for reference. **Incremental writing reminder тАФ second bullet**: the second bullet in the coding plan (immediately after the prompt quote) must always be: `- > **Reminder:** Write in small chunks тАФ use Edit to add subsequent sections one at a time. Do not attempt to write the entire document in a single Write call тАФ large writes can stall or fail silently. Build it up incrementally`. This reminder is mandatory on every coding plan тАФ it reinforces the Incremental Writing behavioral rule and keeps it visible at the start of every response. The remaining bullets describe what will be done. Keep the plan concise тАФ one bullet per distinct action (e.g. "Edit CLAUDE.md to add coding plan rule", "Update README.md timestamp"). This is for transparency, not approval тАФ do NOT wait for user confirmation before proceeding. If the response is purely informational with no changes to make, skip the plan and open with `ЁЯФмЁЯФмRESEARCH STARTЁЯФмЁЯФм` directly (instead of CODING START). **CODING PLAN and CODING START / RESEARCH START appear exactly once per response** тАФ never repeat them mid-response. Use `ЁЯФДЁЯФДNEXT PHASEЁЯФДЁЯФД` instead (see below)
- **Planned affected URLs**: immediately after the coding plan bullets (after the blank line that ends the bullet list, before ESTIMATED TIME), output `ЁЯФЧтЬПя╕ПPLANNED AFFECTED URLSтЬПя╕ПЁЯФЧ` followed by the page URLs expected to be affected by this response тАФ using the same label-URL pair format as the end-of-response `ЁЯФЧтЬПя╕ПAFFECTED URLSтЬПя╕ПЁЯФЧ` section (backtick-wrapped `тЬПя╕П` labels, blockquoted URLs). **Show the current (pre-change) version and status** in the label тАФ read the page's `html.version.txt`, determine the status emoji (ЁЯЯв/ЁЯЯб/ЁЯФ┤), and append the version in parentheses: `` `тЬПя╕ПЁЯМРЁЯЯв Homepage (v01.13w)` ``. This gives the user clickable links to the files' current state and shows what version the page is at before changes begin. **Best-effort prediction** тАФ base it on the coding plan bullets; if the scope changes mid-work, the final AFFECTED URLS at the end may differ. When no pages are expected to be affected, output the header followed by `> *No URL pages expected to be affected*`. **Skip entirely for RESEARCH START responses** (no plan, no affected URLs). When the coding plan repeats after `ЁЯУЛЁЯУЛPLAN APPROVEDЁЯУЛЁЯУЛ`, include PLANNED AFFECTED URLS again after those plan bullets (the prediction may be more accurate post-research)
- **Continuation after user interaction**: when `AskUserQuestion` or `ExitPlanMode` returns mid-response (the user answered a question or approved a plan), the response continues but must **NOT** repeat `ЁЯЪйЁЯЪйCODING PLANЁЯЪйЁЯЪй`, `тЪбтЪбCODING STARTтЪбтЪб`, or `ЁЯФмЁЯФмRESEARCH STARTЁЯФмЁЯФм`. Instead:
  - After `AskUserQuestion`: use `ЁЯФДЁЯФДNEXT PHASEЁЯФДЁЯФД` with a description incorporating the user's choice (e.g. "User chose option A тАФ proceeding with implementation")
  - After `ExitPlanMode` (plan approved): output `ЁЯУЛЁЯУЛPLAN APPROVEDЁЯУЛЁЯУЛ` on its own line, followed by `ЁЯЪйЁЯЪйCODING PLANЁЯЪйЁЯЪй` with the execution plan bullets, then `тЪбтЪбCODING STARTтЪбтЪб`. This is the **only** scenario where CODING PLAN/CODING START may appear a second time тАФ because plan approval is a distinct boundary between planning and execution, and the user needs to see the execution plan clearly. The `ЁЯУЛЁЯУЛPLAN APPROVEDЁЯУЛЁЯУЛ` marker signals that this is a continuation, not a new prompt. **Fresh timestamps required** тАФ run `date` once for PLAN APPROVED, then run `date` again for the post-approval CODING PLAN + CODING START pair (a single call covers both, same as the initial opening pair rule). Do NOT reuse the PLAN APPROVED timestamp for CODING START тАФ there is a gap between plan approval and the start of execution (the coding plan bullets and estimated time are written in between), so the timestamps will differ. PLAN EXECUTION TIME is computed from the post-approval start timestamp (whether displayed via CODING START or silently captured), so an inaccurate timestamp here inflates or deflates the reported execution time
- **Checklist running**: output `тЬФя╕ПтЬФя╕ПCHECKLISTтЬФя╕ПтЬФя╕П` on its own line before executing any mandatory checklist (Session Start, Pre-Commit, Pre-Push), followed by the checklist name (e.g. `Session Start Checklist`). - **Researching**: output `ЁЯФНЁЯФНRESEARCHINGЁЯФНЁЯФН` on its own line when entering a research/exploration phase тАФ reading files, searching the codebase, or understanding context before making changes. Skip if going straight to changes without research. **`тП▒я╕П` must precede this bookend** (see Duration annotations)
- **Mid-response phase marker**: when work within a single response naturally divides into multiple distinct sub-tasks or phases (e.g. "Edit 1" then "Edit 1a: fix related issue"), output `ЁЯФДЁЯФДNEXT PHASEЁЯФДЁЯФД` on its own line followed by a brief description of the new phase. **Never repeat** `ЁЯЪйЁЯЪйCODING PLANЁЯЪйЁЯЪй`, `тЪбтЪбCODING STARTтЪбтЪб`, or `ЁЯФмЁЯФмRESEARCH STARTЁЯФмЁЯФм` within the same response тАФ those appear exactly once (at the very top). The mid-response marker keeps the top/bottom boundaries of each prompt/response turn unambiguous while still signaling transitions between sub-tasks. **`тП▒я╕П` must precede this bookend** (see Duration annotations)
- **Blocked**: output `ЁЯЪзЁЯЪзBLOCKEDЁЯЪзЁЯЪз` on its own line when an obstacle is hit (permission denied, merge conflict, ambiguous requirement, failed push, hook check failure). Follow with a brief description of the blocker. This makes problems immediately visible rather than buried in tool output. **`тП▒я╕П` must precede this bookend** (see Duration annotations)
- **Verifying**: output `ЁЯзкЁЯзкVERIFYINGЁЯзкЁЯзк` on its own line when entering a verification phase тАФ running git hook checks, confirming no stale references, validating edits post-change. Separates "doing the work" from "checking the work". **`тП▒я╕П` must precede this bookend** (see Duration annotations)
- **Changes pushed**: output `тЮбя╕ПтЮбя╕ПCHANGES PUSHEDтЮбя╕ПтЮбя╕П` on its own line **immediately** after a successful `git push` completes. This gives the user instant visibility that their changes are on the remote тАФ no need to scroll through tool output to confirm. Follow with a brief note of what was pushed (e.g. "Pushed to `claude/cleanup-xyz` тАФ workflow will auto-merge to main"). This bookend is a timestamped marker like any other тАФ run `date` before writing it. It participates in `тП▒я╕П` duration tracking (the previous phase's `тП▒я╕П` appears before it, and the next bookend's `тП▒я╕П` measures from it)
- **Time estimate**: output `тП│тП│ESTIMATED TIME тЙИ XmтП│тП│` on its own line, followed by a brief reason (e.g. "~8 file edits + commit + push cycle"), in **two contexts**: (1) **Overall** тАФ **always** appears immediately before the opening marker (`тЪбтЪбCODING STARTтЪбтЪб` or `ЁЯФмЁЯФмRESEARCH STARTЁЯФмЁЯФм`), estimating the entire response from start to the closing marker. This gives the user an upfront sense of total wall-clock time. The overall estimate is **never skipped** тАФ even for quick responses (e.g. `тП│тП│ESTIMATED TIME тЙИ 1mтП│тП│`). Use `Xm` for minutes or `Xs` for estimates under 1 minute. **Plan-mode flows** тАФ when the response opens in plan mode (an `ExitPlanMode` call is anticipated), the original overall estimate **must include BOTH phases**: the research/planning phase AND the anticipated post-approval execution phase. Do NOT estimate only the planning work тАФ even though exact execution scope isn't known at the start, project execution time from the plan's expected shape (file reads, edits, commit cycle, push cycle, visual verification, calibration overhead) and include it in the original estimate. A plan-mode response is not "done" when `ExitPlanMode` is called тАФ it continues through approval and execution until `тЬЕтЬЕCODING COMPLETEтЬЕтЬЕ`, and the estimate must cover that full wall-clock span. (2) **Per-phase** тАФ immediately before any subsequent phase bookend (NEXT PHASE, RESEARCHING, CHECKLIST, etc.) whose phase alone is expected to take longer than 2 minutes. **Only output per-phase estimates when the estimate exceeds 2 minutes** тАФ skip for phases that will be quick. Use these rough heuristics to estimate: ~10s per tool call (read, edit, grep, glob) for payloads тЙд10 lines, **~15s per Edit call when the `new_string` payload exceeds 10 lines** (long replacement strings take longer to produce than short ones тАФ this was observed repeatedly on Mandatory Gate rewrites and bulk CHANGELOG section inserts), ~15s per bash command, ~30s per commit cycle (checklist + staging + commit), ~30s per push cycle (checklist + push + verify), ~1тАУ2m per Explore subagent spawn, ~3тАУ5m per Plan subagent spawn (Plan agents do deeper multi-section analysis than Explore), ~60тАУ90s for first-time Playwright install (`pip install playwright` + `playwright install chromium` ~110MB download), ~30тАУ60s per Playwright visual-verification run (launch + screenshots + read), but **expect 3тАУ5 iterations totaling 3тАУ6m for layout-sensitive changes** because the first run typically reveals overlay/parent-hidden state that needs additional JS injection, and subsequent iterations may surface CSS cascade or specificity bugs that require an edit-and-rerun cycle. Budget the higher end whenever the change touches flexbox, grid, aspect-ratio, or any property that could be overridden by an existing rule later in source order. **Parallel batching discount** тАФ when multiple independent tool calls will be fired in a single batch (e.g. 5 parallel reads, or 5 parallel edits), count the batch as ~15s total, not N ├Ч 10s. Most push-commit cycles fire 4тАУ6 edits in parallel, so the editing phase is ~15s, not ~50s. Apply this discount when you know the calls are independent and will be batched. **Template/consolidation discount** тАФ when the work is consolidating content against a canonical target that is already in place (e.g. collapsing a duplicate section into a pointer to an existing authoritative reference, replacing a near-duplicate block with a short cross-reference), budget **~50% of the from-scratch estimate**. The canonical target already carries the structure and wording; the consolidation is a trim-and-redirect, not a rewrite. This was observed repeatedly on template-separation and visual-verification consolidation work. Sum the expected tool calls (with discount) for the phase and round to the nearest minute. The estimate does not need a timestamp or `date` call тАФ it is an annotation, not a bookend that participates in `тП▒я╕П` duration tracking
- **Revised estimate**: once all initial file reads, greps, and exploration needed to understand the scope of work are complete тАФ regardless of whether a RESEARCHING bookend was used тАФ re-evaluate the overall estimate. Place it inline right after the last read/exploration tool result and before the next action or bookend. **Skip if the original estimate still looks accurate.** The revised estimate replaces the original as the baseline for the ACTUAL TOTAL COMPLETION TIME comparison at the end
- **Actual time**: output `тП│тП│ACTUAL TOTAL COMPLETION TIME: Xm Ys (estimated Xm)тП│тП│` on its own line immediately before the closing marker (`тЬЕтЬЕCODING COMPLETEтЬЕтЬЕ` or `ЁЯФмЁЯФмRESEARCH COMPLETEЁЯФмЁЯФм`). This is the real elapsed wall-clock time from the response start timestamp to the closing marker тАФ computed by subtracting the response start timestamp from the closing marker's timestamp. The response start timestamp is captured by CODING START/RESEARCH START when `CHAT_BOOKENDS` = `On`, or silently via `date` at response start when `CHAT_BOOKENDS` = `Off` (see feature toggle gate). The parenthetical shows the original overall estimate for easy comparison (captured visibly or silently depending on the `CHAT_BOOKENDS` toggle). If a REVISED ESTIMATED TIME was issued, use the revised value instead. **Always present** when CODING COMPLETE or RESEARCH COMPLETE is written (never skipped). **Not present** before AWAITING USER RESPONSE тАФ that ending uses ACTUAL PLANNING TIME instead. The `date` call for the closing marker (already required) provides the end time тАФ no additional `date` call is needed
- **Plan execution time** (conditional): when a plan approval flow was used (`ExitPlanMode` тЖТ `ЁЯУЛЁЯУЛPLAN APPROVEDЁЯУЛЁЯУЛ`), the end-of-response block must include **both** time markers in this order: (1) `тП│тП│PLAN EXECUTION TIME: Xm Ys (estimated Xm)тП│тП│` тАФ the wall-clock time from the post-approval start timestamp (captured by CODING START when `CHAT_BOOKENDS` = `On`, or silently when `CHAT_BOOKENDS` = `Off`) to CODING COMPLETE, with the post-approval estimate as the parenthetical, then (2) `тП│тП│ACTUAL TOTAL COMPLETION TIME: Xm Ys (estimated Xm)тП│тП│` тАФ the sum of the ACTUAL PLANNING TIME and the PLAN EXECUTION TIME, with the original overall estimate (from the response start) as the parenthetical. **Skip PLAN EXECUTION TIME entirely when no plan approval occurred** тАФ responses without `ExitPlanMode` only show ACTUAL TOTAL COMPLETION TIME as usual
- **Estimate calibration**: after computing ACTUAL TOTAL COMPLETION TIME, if the actual and estimated times differ by **more than 2 minutes**, perform a calibration step **before** writing CODING COMPLETE. This makes estimation accuracy self-improving over time:
  1. **Diagnose the miss** тАФ identify what caused the gap. Common causes: underestimating the number of file reads needed, not accounting for checklist overhead, forgetting the push cycle, unexpected retries or errors, or heuristic values being too low/high for the type of work done
  2. **Update the heuristics** тАФ edit the Time estimate bullet's heuristic values in this file to better reflect reality. If commit cycles consistently take ~45s, update `~30s per commit cycle` to `~45s per commit cycle`. Only change values where the evidence from this response supports the adjustment тАФ do not guess or over-correct
  3. **Output the calibration** тАФ in the end-of-response block, output `ЁЯФзЁЯФзESTIMATE CALIBRATEDЁЯФзЁЯФз` after SUMMARY and before ACTUAL TOTAL COMPLETION TIME, followed by a brief description of what was adjusted and why (e.g. "Increased tool call estimate from ~10s to ~15s тАФ actual tool calls averaged 14s this response"). This makes the self-correction visible to the user
  4. **Commit the update** тАФ include the heuristic change in the current commit (or make a follow-up commit if the main commit already happened). The updated heuristics take effect starting from the next response
  **Skip calibration entirely if the difference is тЙд2 minutes** тАФ small variances are normal and not worth correcting
- **Hook anticipation**: before writing the closing marker (`тЬЕтЬЕCODING COMPLETEтЬЕтЬЕ` or `ЁЯФмЁЯФмRESEARCH COMPLETEЁЯФмЁЯФм`), check whether the stop hook (`~/.claude/stop-hook-git-check.sh`) will fire. **This check must happen after all actions in the current response are complete** (including any `git push`) тАФ do not predict the pre-action state; check the actual post-action state. **Actually run** the three git commands (do not evaluate mentally): (a) uncommitted changes тАФ `git diff --quiet && git diff --cached --quiet`, (b) untracked files тАФ `git ls-files --others --exclude-standard`, (c) unpushed commits тАФ `git rev-list origin/<branch>..HEAD --count`. ## Hook Anticipation тАФ Bug Context

**The failure pattern:** if the hook conditions are evaluated *before* a `git push` completes (or evaluated mentally instead of actually running the git commands), the prediction can be wrong тАФ e.g. concluding there are unpushed commits when the push already succeeded. Writing `ЁЯРЯЁЯРЯAWAITING HOOKЁЯРЯЁЯРЯ` in that case means the hook never fires (because all conditions are actually false), and the conversation gets stuck with no closing marker.

**What to watch for:** any scenario where actions (especially `git push`) complete in the same response as the hook check. The temptation is to predict the outcome rather than wait and verify.

**The fix:** (1) always evaluate *after* all actions in the response are complete, and (2) *actually run* the three git commands тАФ never reason about their output mentally.
